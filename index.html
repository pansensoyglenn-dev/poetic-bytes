Python Key Concepts

Essential terms explained in plain language

variable

A named container that holds a value. Python infers the type automatically — no declaration needed.

function

A reusable block defined with def. First-class objects — you can pass them around like variables.

list

An ordered, mutable collection. Supports slicing, sorting, and the powerful list comprehension syntax.

dictionary

A key-value mapping ordered since 3.7+. Extremely fast O(1) lookups — backbone of most data-processing code.

class

A blueprint for creating objects that bundle data and behaviour. Foundation of OOP in Python.

decorator

A function that wraps another to extend its behaviour without modifying it. Written with @.

generator

Yields values one at a time instead of all at once. Memory-efficient for large datasets or infinite sequences.

exception

An error handled with try/except blocks. Robust exception handling separates production code from scripts.

🏛️ Object-Oriented Programming

🏗️ Defining Classes & __init__

A class bundles data (attributes) and behaviour (methods). __init__ runs on every new instance. self refers to the instance — not a keyword, just a strong convention.

# defining a class with __init__
class BankAccount:
    def __init__(self, owner: str, balance: float = 0.0):
        self.owner   = owner
        self.balance = balance
        self._history = []          # private by convention

    def deposit(self, amount: float):
        if amount <= 0:
            raise ValueError("Amount must be positive")
        self.balance += amount
        self._history.append(f"+{amount}")

acc1 = BankAccount("Alice", 500)
acc1.deposit(200)
print(acc1.balance)   # 700.0
💡 Every method receives self as its first argument. Python passes it automatically — you never supply it when calling acc1.deposit(200).

✨ Dunder Methods (Magic Methods)

Python calls dunder methods automatically in response to built-in operations. Implementing them makes your objects feel native — comparable with ==, printable, addable with +.

class Vector:
    def __init__(self, x, y):
        self.x, self.y = x, y

    def __repr__(self):
        return f"Vector({self.x}, {self.y})"

    def __add__(self, other):
        return Vector(self.x + other.x, self.y + other.y)

    def __eq__(self, other):
        return self.x == other.x and self.y == other.y

v1 = Vector(3, 4); v2 = Vector(1, 2)
print(v1 + v2)    # Vector(4, 6)
print(v1 == v2)   # False

🎯 @property, @classmethod, @staticmethod

@property turns a method into a computed attribute. @classmethod receives the class for alternative constructors. @staticmethod is a helper namespaced inside the class.

class Temperature:
    def __init__(self, celsius: float):
        self._celsius = celsius

    @property
    def celsius(self): return self._celsius

    @celsius.setter
    def celsius(self, value):
        if value < -273.15: raise ValueError("Below absolute zero!")
        self._celsius = value

    @property
    def fahrenheit(self): return self._celsius * 9/5 + 32

    @classmethod
    def from_fahrenheit(cls, f): return cls((f - 32) * 5/9)

t = Temperature.from_fahrenheit(212)
print(t.celsius)     # 100.0
print(t.fahrenheit)  # 212.0

📦 Dataclasses — Boilerplate-Free Classes

The @dataclass decorator (3.7+) auto-generates __init__, __repr__, and __eq__ from annotated fields. Dramatically reduces repetitive code.

from dataclasses import dataclass, field
from typing import List

@dataclass
class Student:
    name:   str
    grade:  int
    scores: List[float] = field(default_factory=list)

    @property
    def average(self) -> float:
        return sum(self.scores) / len(self.scores) if self.scores else 0.0

s = Student("Ana", 10)
s.scores.extend([88, 92, 95])
print(s)           # Student(name='Ana', grade=10, scores=[88, 92, 95])
print(s.average)   # 91.666...

⚠️ 10 Python Mistakes Every Beginner Makes

Learn from thousands of learners — avoid the errors that waste the most time

Every Python beginner makes the same set of mistakes. Not because they're not smart — but because Python has a handful of genuinely surprising behaviours that differ from plain logic or from other programming languages. Recognising these traps early saves hours of frustrated debugging. Each mistake below comes with a clear explanation of why it happens and exactly how to fix it.

❌ Mistake 1 — Using a mutable default argument

Default argument values are evaluated once when the function is defined — not each time it's called. If you use a mutable object like a list as a default, all calls that use the default share the same object.

# WRONG — the list is created once and shared
def add_item(item, items=[]):
    items.append(item)
    return items

print(add_item("apple"))   # ['apple']
print(add_item("banana"))  # ['apple', 'banana'] ← surprise!

# CORRECT — use None as sentinel, create fresh list each call
def add_item(item, items=None):
    if items is None:
        items = []
    items.append(item)
    return items
💡 This applies to dicts, sets, and any other mutable type. Always use None as the default and create the mutable object inside the function body.

❌ Mistake 2 — Confusing == with is

== checks whether two values are equal. is checks whether two names point to the exact same object in memory. For small integers and interned strings Python reuses objects, which can make is appear to work — until it silently doesn't.

# Appears to work (CPython caches small ints)
a = 256; b = 256
print(a is b)   # True — same cached object

a = 257; b = 257
print(a is b)   # False — separate objects beyond cache range

# CORRECT — use == for value comparison
print(a == b)   # True — always correct for value checks

# The only correct use of 'is' — checking against None
if result is None:
    print("No result yet")

❌ Mistake 3 — Modifying a list while iterating over it

Changing the size of a list during a for loop over it causes items to be skipped silently — one of the hardest bugs to spot because Python gives no error, it just produces wrong output.

# WRONG — items get skipped
nums = [1, 2, 3, 4, 5, 6]
for n in nums:
    if n % 2 == 0:
        nums.remove(n)
print(nums)  # [1, 3, 5] — looks right by accident, but fragile

# CORRECT — iterate over a copy, or use a list comprehension
nums = [1, 2, 3, 4, 5, 6]
nums = [n for n in nums if n % 2 != 0]
print(nums)  # [1, 3, 5] — always correct

❌ Mistake 4 — Catching too broad an exception

Writing except Exception: or bare except: swallows every error — including typos in variable names, import errors, and keyboard interrupts — making bugs nearly impossible to diagnose.

# WRONG — hides all errors silently
try:
    result = do_something()
except:
    print("Something went wrong")   # what? you'll never know

# CORRECT — catch only what you expect and handle it specifically
try:
    result = int(user_input)
except ValueError:
    print("Please enter a valid number")
except TypeError as e:
    print(f"Type error: {e}")
finally:
    cleanup()   # runs whether or not there was an error
💡 If you genuinely need to catch all exceptions (e.g. in a top-level error handler), at minimum log the full traceback with logging.exception(e) so the error isn't lost.

❌ Mistake 5 — Not understanding variable scope (LEGB rule)

Python resolves variable names in a specific order: Local → Enclosing → Global → Built-in. Forgetting this leads to confusing UnboundLocalError bugs where a variable seems to exist but can't be read.

count = 0

def increment():
    count += 1       # UnboundLocalError! Python sees assignment
                     # so treats count as local, but it's not defined locally yet

# Fix 1 — use global keyword (use sparingly)
def increment():
    global count
    count += 1

# Fix 2 — better design: pass and return values
def increment(count):
    return count + 1

count = increment(count)

❌ Mistake 6 — Using + to concatenate strings in a loop

Strings in Python are immutable. Every + creates a new string object and copies both strings into it. Inside a loop over thousands of items, this creates quadratic time complexity. The fix is instant and idiomatic.

words = ["Python", "is", "fast", "and", "readable"]

# WRONG — O(n²) — creates a new string on every iteration
result = ""
for word in words:
    result += word + " "

# CORRECT — O(n) — join builds the string in one pass
result = " ".join(words)
print(result)   # Python is fast and readable

# Same applies to building large texts — use a list then join
parts = []
for item in big_data:
    parts.append(process(item))
output = "\n".join(parts)

❌ Mistake 7 — Forgetting that integers and strings don't auto-convert

Python is strongly typed. Unlike JavaScript, it will never silently coerce a number to a string or vice versa. This causes TypeError that trips up beginners coming from more loosely typed languages.

age = 25

# WRONG — TypeError: can only concatenate str (not "int") to str
print("I am " + age + " years old")

# CORRECT — use an f-string (recommended)
print(f"I am {age} years old")

# ALSO CORRECT — explicit str() conversion
print("I am " + str(age) + " years old")

# f-strings also format numbers elegantly
pi = 3.14159265
print(f"Pi to 2 decimal places: {pi:.2f}")   # 3.14

❌ Mistake 8 — Shallow copy vs deep copy confusion

Assigning a list to a new variable does not copy it — both names point to the same object. A shallow copy duplicates the outer container but shares the inner objects. Only a deep copy is truly independent.

import copy

original = [[1, 2], [3, 4]]

# WRONG — just another name for the same list
alias = original
alias[0][0] = 99
print(original)   # [[99, 2], [3, 4]] — original changed!

# Shallow copy — outer list is new, inner lists still shared
shallow = original.copy()
shallow[0][0] = 42
print(original)   # [[42, 2], [3, 4]] — still changes original!

# CORRECT — deep copy is fully independent
deep = copy.deepcopy(original)
deep[0][0] = 0
print(original)   # [[42, 2], [3, 4]] — untouched ✓

❌ Mistake 9 — Ignoring return values from string methods

Strings are immutable. Methods like .upper(), .strip(), and .replace() do not change the string in place — they return a new string. Forgetting to capture the return value is one of the most common beginner bugs.

name = "  glenn  "

# WRONG — strip() returns a new string, name is unchanged
name.strip()
print(name)   # "  glenn  " — still has whitespace

# CORRECT — capture the returned value
name = name.strip()
print(name)   # "glenn"

# Same pattern for all string methods
text = "hello world"
text = text.upper().replace("WORLD", "PYTHON")
print(text)   # "HELLO PYTHON"

❌ Mistake 10 — Writing loops when a comprehension or built-in would do

Python has a rich set of built-in functions and comprehensions designed precisely to replace verbose loops. Writing an explicit loop where a one-liner exists is not just more verbose — it's usually slower and harder to read.

# WRONG — verbose loop to build a filtered list
squares = []
for n in range(10):
    if n % 2 == 0:
        squares.append(n ** 2)

# CORRECT — list comprehension (faster, more idiomatic)
squares = [n**2 for n in range(10) if n % 2 == 0]

# WRONG — manual sum loop
total = 0
for x in data: total += x

# CORRECT — use built-ins
total = sum(data)
maximum = max(data)
minimum = min(data)
print(total, maximum, minimum)
💡 Built-ins like sum(), max(), min(), any(), all(), zip(), enumerate(), and sorted() exist because these patterns appear in almost every Python program. Learn them early.

✅ Python Best Practices — Write Code Like a Pro

The habits that separate readable, maintainable Python from spaghetti code

Knowing the syntax is only half of Python. The other half is knowing the conventions, patterns, and habits that make your code easy to read, debug, and maintain — both for yourself six months from now and for anyone who works with your code. These practices are not opinions; they are the documented conventions of the Python community, codified in PEP 8 and reinforced by years of collective experience.

📏 Follow PEP 8 — Python's Style Guide

PEP 8 is the official style guide for Python code. It covers naming conventions, indentation, line length, blank lines, and import ordering. Most professional Python codebases enforce it automatically with tools like flake8, black, or ruff.

# PEP 8 naming conventions

# Variables and functions: snake_case
user_name = "Glenn"
def calculate_average(scores): ...

# Classes: PascalCase (CapWords)
class UserAccount: ...
class DataProcessor: ...

# Constants: UPPER_SNAKE_CASE
MAX_RETRIES = 3
BASE_URL = "https://api.example.com"

# Private by convention: leading underscore
_internal_helper = "not for external use"

# Imports: one per line, stdlib first, then third-party, then local
import os
import sys

import requests
import pandas as pd

from mymodule import helper

📝 Write Docstrings for Every Public Function and Class

Docstrings are not comments — they are first-class documentation that tools like Sphinx, IDE tooltips, and help() can read. A function without a docstring forces the reader to study the implementation to understand intent.

def calculate_bmi(weight_kg: float, height_m: float) -> float:
    """Calculate Body Mass Index (BMI).

    Args:
        weight_kg: Body weight in kilograms.
        height_m: Height in metres.

    Returns:
        BMI as a float, rounded to 2 decimal places.

    Raises:
        ValueError: If height_m is zero or negative.

    Example:
        >>> calculate_bmi(70, 1.75)
        22.86
    """
    if height_m <= 0:
        raise ValueError("Height must be positive")
    return round(weight_kg / height_m ** 2, 2)

🔒 Use Type Hints — They Are Not Optional in 2026

Python's type hints (introduced in PEP 484) don't change runtime behaviour, but they dramatically improve readability, enable IDE autocompletion, and allow static type checkers like mypy and pyright to catch bugs before your code runs.

from typing import Optional, Union
# Python 3.10+ can use X | Y syntax instead of Union[X, Y]

def find_user(user_id: int) -> Optional[dict]:
    """Return user dict or None if not found."""
    ...

def parse_value(raw: str) -> int | float:
    """Parse a string to int or float."""
    try:
        return int(raw)
    except ValueError:
        return float(raw)

# TypedDict for structured dicts
from typing import TypedDict

class Article(TypedDict):
    title: str
    author: str
    word_count: int
    published: bool

🧪 Write Tests — Even Simple Ones

Untested code is broken code waiting to be discovered. Python's built-in unittest and the popular third-party pytest make testing straightforward. Even a handful of tests catches regressions before they reach production and forces you to write more modular, testable code.

# test_bmi.py — using pytest (install with: pip install pytest)
import pytest
from bmi import calculate_bmi

def test_normal_bmi():
    assert calculate_bmi(70, 1.75) == 22.86

def test_obese_range():
    assert calculate_bmi(100, 1.70) > 30

def test_zero_height_raises():
    with pytest.raises(ValueError):
        calculate_bmi(70, 0)

# Run all tests with: pytest test_bmi.py -v
💡 The rule of thumb: if a bug is worth fixing, write a test that would have caught it. Over time, your test suite becomes a safety net that lets you refactor confidently.

🔧 Essential Python Built-in Functions You Must Know

Python ships with 70+ built-in functions — these are the ones that appear in virtually every real program

One of Python's greatest strengths is how much useful functionality is available without importing anything. These built-in functions are always available, highly optimised in C, and cover the most common programming needs. Knowing them means less code, fewer bugs, and faster programs.

🔢

enumerate() — Loop with index

Returns an iterator of (index, value) pairs. Eliminates the need for a manual counter variable in loops.

fruits = ["apple", "banana", "cherry"]
for i, fruit in enumerate(fruits, start=1):
    print(f"{i}. {fruit}")
# 1. apple  2. banana  3. cherry
🤐

zip() — Combine iterables

Pairs up items from multiple iterables. Perfect for looping over two related lists simultaneously.

names  = ["Ana", "Bob", "Cara"]
scores = [88, 92, 79]
for name, score in zip(names, scores):
    print(f"{name}: {score}")
🔀

sorted() & sort()

sorted() returns a new sorted list. .sort() sorts in place. Both accept a key function for custom ordering.

words = ["banana", "Apple", "cherry"]
print(sorted(words, key=str.lower))
# ['Apple', 'banana', 'cherry']

people = [{"name":"Bob","age":30},{"name":"Ana","age":25}]
people.sort(key=lambda p: p["age"])
🗺️

map() & filter()

Apply a function to every item (map), or keep only items matching a condition (filter). Both return lazy iterators.

nums = [1, 2, 3, 4, 5, 6]
doubled  = list(map(lambda x: x*2, nums))
evens    = list(filter(lambda x: x%2==0, nums))
print(doubled)  # [2,4,6,8,10,12]
print(evens)    # [2,4,6]
✔️

any() & all()

any() returns True if at least one item is truthy. all() returns True only if every item is truthy. Both short-circuit.

scores = [85, 92, 78, 95, 88]
print(all(s >= 70 for s in scores))  # True
print(any(s >= 90 for s in scores))  # True

# Great for validation
required = ["name", "email", "age"]
data = {"name":"Ana", "email":"a@b.com", "age":25}
print(all(k in data for k in required)) # True
📊

isinstance() & type()

isinstance() is the correct way to check an object's type — it handles inheritance. type() gives the exact type without inheritance awareness.

class Animal: pass
class Dog(Animal): pass

d = Dog()
print(isinstance(d, Dog))     # True
print(isinstance(d, Animal))  # True ← handles inheritance
print(type(d) is Animal)      # False ← exact match only

📂 File Handling & Error Handling in Python

Reading, writing, and processing files safely — skills every Python developer needs daily

Almost every real Python program interacts with files — reading configuration, processing CSVs, writing logs, parsing JSON. Doing this correctly means using context managers, handling exceptions gracefully, and choosing the right encoding. Here is everything you need to know to handle files like a professional.

📖 Reading and Writing Files Safely

Always use the with statement when opening files. It guarantees the file is closed even if an exception occurs — no resource leaks, no corrupted files.

# Reading a text file line by line (memory-efficient)
with open("data.txt", "r", encoding="utf-8") as f:
    for line in f:
        print(line.strip())

# Reading entire file into memory
with open("data.txt", encoding="utf-8") as f:
    content = f.read()         # entire file as one string
    lines   = f.readlines()   # list of lines

# Writing (overwrites existing file)
with open("output.txt", "w", encoding="utf-8") as f:
    f.write("Hello, file!\n")
    f.writelines(["line 1\n", "line 2\n"])

# Appending without overwriting
with open("log.txt", "a", encoding="utf-8") as f:
    f.write("New log entry\n")
💡 Always specify encoding="utf-8" explicitly. The default encoding varies by platform and can cause mysterious bugs when your code runs on a different machine.

📋 Working with CSV and JSON — The Most Common Data Formats

CSV and JSON are the two formats you will encounter most often in real-world Python work — spreadsheet exports, API responses, configuration files. Python's standard library handles both without any third-party dependencies.

import csv, json

# ─── CSV ───
# Reading CSV
with open("sales.csv", newline="", encoding="utf-8") as f:
    reader = csv.DictReader(f)   # each row is a dict
    for row in reader:
        print(row["product"], row["revenue"])

# Writing CSV
rows = [{"name":"Ana","score":95},{"name":"Bob","score":87}]
with open("results.csv", "w", newline="", encoding="utf-8") as f:
    writer = csv.DictWriter(f, fieldnames=["name","score"])
    writer.writeheader()
    writer.writerows(rows)

# ─── JSON ───
data = {"user": "Glenn", "level": "advanced", "scores": [95, 88, 92]}

# Write to JSON file
with open("user.json", "w") as f:
    json.dump(data, f, indent=2)

# Read from JSON file
with open("user.json") as f:
    loaded = json.load(f)
    print(loaded["user"])   # Glenn

🛡️ Robust Error Handling Patterns

Production Python code anticipates failure. Files might not exist. Networks might time out. Users might enter invalid input. A well-structured try/except/else/finally block handles each of these cases cleanly.

import logging

logging.basicConfig(level=logging.INFO)

def load_config(path: str) -> dict:
    """Load JSON config file with full error handling."""
    try:
        with open(path, encoding="utf-8") as f:
            config = json.load(f)
    except FileNotFoundError:
        logging.error(f"Config file not found: {path}")
        return {}
    except json.JSONDecodeError as e:
        logging.error(f"Invalid JSON in {path}: {e}")
        return {}
    except PermissionError:
        logging.error(f"No permission to read: {path}")
        raise                      # re-raise — caller should know
    else:
        logging.info(f"Config loaded: {len(config)} keys")
        return config
    finally:
        logging.debug("load_config() finished")  # always runs
💡 The else block runs only when no exception occurred — perfect for success-path logic. finally always runs — perfect for cleanup like closing database connections or releasing locks.

📦 Package Management — pip, venv, and poetry

Every Python project should live in a virtual environment — an isolated directory that contains its own Python interpreter and packages, completely separate from your system Python. This prevents version conflicts between projects and makes your code reproducible on any machine.

# Create and activate a virtual environment
python -m venv .venv
source .venv/bin/activate        # macOS / Linux
.venv\Scripts\activate           # Windows

# Install packages
pip install requests pandas flask

# Save exact dependency versions
pip freeze > requirements.txt

# Reproduce environment on another machine
pip install -r requirements.txt

# Poetry — modern all-in-one tool (recommended for new projects)
poetry new my-project            # create project with pyproject.toml
poetry add requests              # add dependency
poetry run python main.py        # run inside virtual environment

🎯 Python Interview Preparation — Questions & Answers

The most commonly asked Python interview questions at tech companies, with detailed answers

Python interviews at tech companies range from basic syntax questions to tricky language internals, data structure problems, and system design discussions. Preparation is everything. Below are the questions that appear most frequently across junior, mid-level, and senior Python developer interviews — each with a thorough explanation so you understand not just the answer but the reasoning behind it.

What is the difference between a list and a tuple in Python?

+

A list is mutable — you can add, remove, or change elements after creation. A tuple is immutable — once created, it cannot be changed. Tuples are faster to iterate over and can be used as dictionary keys (because they are hashable), whereas lists cannot. Use tuples for data that shouldn't change (coordinates, database records, config values) and lists for collections that will be modified. Immutability is a feature, not a limitation — it communicates intent to anyone reading your code.

What are Python generators and when should you use them?

+

A generator is a function that uses yield instead of return. When called, it returns a generator object — a lazy iterator that produces values one at a time on demand, rather than computing and storing all values in memory at once. Use generators when: (1) working with large datasets that don't fit in memory, (2) producing infinite sequences, (3) building data pipelines. A generator expression (x**2 for x in range(1000000)) uses negligible memory; the equivalent list comprehension would allocate memory for a million integers immediately.

Explain Python's GIL — what it is and why it matters

+

The Global Interpreter Lock (GIL) is a mutex in CPython (the reference implementation) that allows only one thread to execute Python bytecode at a time. This means that even on multi-core processors, Python threads cannot run in true parallel for CPU-bound tasks. However, the GIL is released during I/O operations, so threading works well for I/O-bound workloads (network calls, file reads). For true CPU parallelism in Python, use the multiprocessing module (separate processes, each with their own GIL), or concurrent.futures.ProcessPoolExecutor. Python 3.13 introduced experimental no-GIL builds.

What is the difference between @staticmethod and @classmethod?

+

A @staticmethod receives no implicit first argument — it is essentially a regular function that lives inside a class for organisational purposes. It has no access to the class or any instance. A @classmethod receives the class itself as its first argument (conventionally named cls). This makes it ideal for alternative constructors — factory methods that create instances in different ways (e.g. Date.from_string("2026-01-15")). Use @staticmethod for utilities that happen to belong logically to the class. Use @classmethod when the method needs to know about the class itself, especially for inheritance-aware construction.

How does Python's memory management and garbage collection work?

+

Python uses reference counting as its primary memory management strategy. Every object tracks how many names (references) point to it. When the count drops to zero, the memory is immediately freed. However, reference counting cannot handle circular references (where object A refers to B and B refers to A, even if nothing else refers to either). CPython's cyclic garbage collector runs periodically to detect and break these cycles. The gc module gives you control over this collector. For performance-critical code, minimise circular references and use __slots__ to reduce per-object memory overhead.

What is a Python decorator and how do you write one from scratch?

+

A decorator is a callable that takes a function as input and returns a new function that wraps the original — extending or modifying its behaviour without changing its source code. At its core, @my_decorator above a function is syntactic sugar for func = my_decorator(func). The pattern requires a wrapper function inside that calls the original and a return of that wrapper. Use @functools.wraps(func) to preserve the original function's metadata (name, docstring). Decorators power Flask routes (@app.route), Django views, pytest fixtures, caching (@lru_cache), and authentication everywhere in Python.

Explain list comprehensions vs generator expressions vs map()

+

A list comprehension [x**2 for x in range(10)] eagerly creates a full list in memory — use when you need to iterate multiple times or need random access. A generator expression (x**2 for x in range(10)) is lazy — it produces values on demand — use for large datasets or single-pass pipelines where memory matters. map() also returns a lazy iterator but requires a function object (often a lambda), making it slightly less readable than generator expressions for most use cases. In modern Python, list comprehensions and generator expressions are preferred for clarity; map() is mainly useful when you already have a named function to apply.

What are dunder methods and why do they matter?

+

Dunder (double underscore) methods — also called magic methods or special methods — are how Python implements its data model. When you write a + b, Python calls a.__add__(b). When you write len(obj), Python calls obj.__len__(). By implementing these methods in your own classes, your objects integrate seamlessly with Python's built-in syntax and functions. Key ones to know: __init__ (construction), __repr__ and __str__ (string representation), __eq__ and __hash__ (equality and hashing), __len__, __iter__ and __next__ (iteration), __enter__ and __exit__ (context managers), __getitem__ (indexing with []).
