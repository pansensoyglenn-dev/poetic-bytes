Python Key Concepts ‚Äî Quick Reference

Essential terms every Python learner should know, explained in plain language

variable

A named container that holds a value. Unlike many languages, Python variables don't need a type declaration ‚Äî Python infers the type automatically.

function

A reusable block of code that performs a specific task. Defined with def. Functions are first-class objects in Python ‚Äî you can pass them around like variables.

list

An ordered, mutable collection of items. Written as [1, 2, 3]. Lists can hold mixed types and support slicing, sorting, and list comprehensions.

dictionary

A key-value mapping. Written as {"key": value}. Python dicts are ordered (since 3.7+) and extremely fast for lookups ‚Äî the backbone of most data-processing code.

class

A blueprint for creating objects that bundle data and behavior. The foundation of Object-Oriented Programming (OOP) in Python.

decorator

A function that wraps another function to extend its behavior without modifying it. Written with @. Used extensively in Flask, Django, and testing frameworks.

generator

A function that yields values one at a time instead of returning them all at once. Memory-efficient for processing large datasets or infinite sequences.

exception

An error that disrupts normal program flow. Handled with try / except blocks. Writing robust exception handling is what separates production code from scripts.

üèóÔ∏è Defining Classes and __init__

A class is a blueprint that bundles data (attributes) and behavior (methods) into a single reusable unit. The __init__ method runs automatically whenever you create a new instance and is the place to set up initial state. self is a reference to the instance being created ‚Äî it is not a keyword, just a strong convention.

# defining a class with __init__
class BankAccount:
    def __init__(self, owner: str, balance: float = 0.0):
        self.owner   = owner
        self.balance = balance
        self._history = []          # private by convention

    def deposit(self, amount: float):
        if amount <= 0:
            raise ValueError("Amount must be positive")
        self.balance += amount
        self._history.append(f"+{amount}")

    def withdraw(self, amount: float):
        if amount > self.balance:
            raise ValueError("Insufficient funds")
        self.balance -= amount
        self._history.append(f"-{amount}")

# creating instances
acc1 = BankAccount("Alice", 500)
acc2 = BankAccount("Bob")      # balance defaults to 0.0
acc1.deposit(200)
print(acc1.balance)          # 700.0
üí° Every method receives self as its first argument. Python passes it automatically ‚Äî you never supply it when calling acc1.deposit(200).

üìä Class Variables vs Instance Variables

Instance variables live on each object and are unique per instance. Class variables are shared across all instances ‚Äî useful for counters, default settings, or constants that belong to the concept rather than any particular object. The distinction matters most when a class variable is mutable.

class Employee:
    # class variable ‚Äî shared
    company  = "Acme Corp"
    headcount = 0

    def __init__(self, name):
        # instance variables ‚Äî unique
        self.name = name
        Employee.headcount += 1

e1 = Employee("Diana")
e2 = Employee("Evan")
print(Employee.headcount)  # 2
print(e1.company)         # Acme Corp
# mutable class variable pitfall
class Buggy:
    items = []           # shared list!
    def add(self, x):
        self.items.append(x)

a = Buggy(); b = Buggy()
a.add(1); b.add(2)
print(a.items)  # [1, 2] ‚Üê surprise!

# fix: use __init__ for mutable defaults
class Fixed:
    def __init__(self):
        self.items = []  # per-instance

‚ö†Ô∏è Never use a mutable object (list, dict, set) as a class variable default. Always initialise it inside __init__.

‚ú® Dunder Methods (Magic Methods)

Python calls dunder ("double underscore") methods automatically in response to built-in operations. Implementing them makes your objects feel native ‚Äî they can be compared with ==, printed readably, added with +, iterated with for, or used as dictionary keys. This is how Python achieves operator overloading without special syntax.

class Vector:
    def __init__(self, x, y):
        self.x, self.y = x, y

    def __repr__(self):            # used by repr() and the REPL
        return f"Vector({self.x}, {self.y})"

    def __str__(self):             # used by print() and str()
        return f"({self.x}, {self.y})"

    def __add__(self, other):        # v1 + v2
        return Vector(self.x + other.x, self.y + other.y)

    def __eq__(self, other):         # v1 == v2
        return self.x == other.x and self.y == other.y

    def __len__(self):             # len(v)
        import math
        return int(math.sqrt(self.x**2 + self.y**2))

v1 = Vector(3, 4)
v2 = Vector(1, 2)
print(v1 + v2)   # (4, 6)
print(len(v1))   # 5
print(v1 == v2)  # False

üéØ Decorators: @property, @classmethod, @staticmethod

Python provides three built-in decorators that change how methods behave. @property turns a method into a computed attribute with optional validation on set. @classmethod receives the class itself as the first argument and is ideal for alternative constructors. @staticmethod is a plain function namespaced inside the class ‚Äî it receives neither instance nor class.

class Temperature:
    def __init__(self, celsius: float):
        self._celsius = celsius

    
    @property
    def celsius(self):
        return self._celsius

    @celsius.setter
    def celsius(self, value):
        if value < -273.15:
            raise ValueError("Below absolute zero!")
        self._celsius = value

    @property
    def fahrenheit(self):           # read-only computed property
        return self._celsius * 9/5 + 32

    
    @classmethod
    def from_fahrenheit(cls, f: float):
        return cls((f - 32) * 5/9)

    
    @staticmethod
    def is_valid(celsius: float) -> bool:
        return celsius >= -273.15

t = Temperature.from_fahrenheit(212)
print(t.celsius)            # 100.0
print(t.fahrenheit)         # 212.0
print(Temperature.is_valid(-300))  # False
üí° Use @property when you want attribute-style access but need control (validation, lazy computation). Use @classmethod to provide multiple ways to construct an object. Use @staticmethod for helpers logically related to the class but not tied to any instance.

üîó Composition ‚Äî Building Objects from Objects

Inheritance models an "is-a" relationship. Composition models a "has-a" relationship and is usually the better choice. A Car is not a special kind of Engine ‚Äî it has an Engine. Composition keeps classes small, focused, and independently testable. You can swap components without restructuring a class hierarchy.

class Engine:
    def __init__(self, horsepower: int):
        self.horsepower = horsepower
        self.running = False

    def start(self):
        self.running = True
        return "Engine started"

class GPS:
    def route(self, destination):
        return f"Routing to {destination}..."

class Car:
    def __init__(self, model: str, hp: int):
        self.model  = model
        self.engine = Engine(hp)    # has-a Engine
        self.gps    = GPS()          # has-a GPS

    def drive(self, destination):
        msg = self.engine.start()
        nav = self.gps.route(destination)
        return f"{self.model}: {msg} | {nav}"

tesla = Car("Tesla Model 3", 450)
print(tesla.drive("Airport"))
# Tesla Model 3: Engine started | Routing to Airport...
üí° Prefer composition when the relationship between two classes changes over time or you need to swap implementations. Prefer inheritance when a genuine "is-a" relationship exists and you want to share a consistent interface.

üì¶ Dataclasses ‚Äî Boilerplate-Free Classes

The @dataclass decorator (Python 3.7+) auto-generates __init__, __repr__, and __eq__ from annotated fields. It dramatically reduces repetitive code for classes whose primary job is holding data. You can mark fields as mutable or immutable (frozen=True), add default factories, and still add your own methods.

from dataclasses import dataclass, field
from typing import List

@dataclass
class Student:
    name:   str
    grade:  int
    scores: List[float] = field(default_factory=list)  # safe mutable default

    @property
    def average(self) -> float:
        return sum(self.scores) / len(self.scores) if self.scores else 0.0

    def add_score(self, s: float):
        self.scores.append(s)

s = Student("Ana", 10)
s.add_score(88); s.add_score(92); s.add_score(95)
print(s)             # Student(name='Ana', grade=10, scores=[88, 92, 95])
print(s.average)     # 91.666...

# frozen=True makes it immutable and hashable
@dataclass(frozen=True)
class Point:
    x: float
    y: float

p = Point(3.0, 4.0)
print(p)  # Point(x=3.0, y=4.0)

üîë Context Managers ‚Äî The with Statement

A context manager guarantees that set-up and tear-down code runs reliably ‚Äî even if an exception is raised. Implement __enter__ and __exit__ to make any class usable with with. This pattern is the backbone of file I/O, database connections, locks, and any resource that must be released after use.

class ManagedFile:
    def __init__(self, path: str, mode: str = "r"):
        self.path = path
        self.mode = mode
        self.file = None

    def __enter__(self):               # runs at 'with' entry
        self.file = open(self.path, self.mode)
        return self.file

    def __exit__(self, exc_type, exc_val, exc_tb):
        if self.file:
            self.file.close()         # always runs, even on error
        return False                  # don't suppress exceptions

# usage
with ManagedFile("data.txt", "w") as f:
    f.write("Python context managers are clean!")
# file is closed automatically here

# shorter with contextlib
from contextlib import contextmanager

@contextmanager
def timer():
    import time
    start = time.perf_counter()
    yield
    elapsed = time.perf_counter() - start
    print(f"Elapsed: {elapsed:.4f}s")
üí° Python's built-in open() already implements the context manager protocol. Whenever you write resource-management code, ask: "should this be a context manager?"

‚ö° __slots__ ‚Äî Memory-Efficient Classes

By default Python stores instance attributes in a dictionary (__dict__), which is flexible but uses extra memory. Declaring __slots__ replaces that dictionary with a fixed-size C-level array, reducing memory by 40‚Äì60% per instance ‚Äî critical when you create millions of small objects (e.g., in simulations, parsers, or data pipelines).

# without __slots__ (default)
class PointNormal:
    def __init__(self, x, y):
        self.x = x
        self.y = y

p = PointNormal(1, 2)
p.z = 99  # allowed, adds to __dict__
print(p.__dict__)
# {'x':1, 'y':2, 'z':99}
# with __slots__ (memory optimised)
class PointSlotted:
    __slots__ = ("x", "y")
    def __init__(self, x, y):
        self.x = x
        self.y = y

p2 = PointSlotted(1, 2)
# p2.z = 99  ‚Üê AttributeError
# no __dict__, ~50% less memory
