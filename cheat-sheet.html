✅ Python Best Practices

Write clean, maintainable, and professional Python code by following these proven guidelines.

📏 Follow PEP 8 — Python's Style Guide

PEP 8 is the official style guide for Python. It covers naming conventions, indentation, line length, blank lines, and import ordering. Professional Python codebases enforce it automatically with tools like black or ruff.

# PEP 8 naming conventions
user_name = "Glenn"
def calculate_average(scores): ...
class UserAccount: ...
class DataProcessor: ...
MAX_RETRIES = 3
BASE_URL = "https://api.example.com"
_internal_helper = "not for external use"

Key PEP 8 rules: Use 4 spaces per indentation level. Limit lines to 79 characters. Use blank lines to separate functions and classes. Put imports at the top of the file, grouped by standard library, third-party, and local modules.

📝 Write Docstrings for Every Public Function

Docstrings are first-class documentation that tools like Sphinx, IDE tooltips, and help() can read. A function without a docstring forces the reader to study the implementation to understand intent.

def calculate_bmi(weight_kg: float, height_m: float) -> float:
    """Calculate Body Mass Index (BMI).
    Args:
        weight_kg: Body weight in kilograms.
        height_m: Height in metres.
    Returns:
        BMI as a float, rounded to 2 decimal places.
    Raises:
        ValueError: If height_m is zero or negative.
    Example:
        >>> calculate_bmi(70, 1.75)
        22.86
    """
    if height_m <= 0:
        raise ValueError("Height must be positive")
    return round(weight_kg / height_m ** 2, 2)

🔒 Use Type Hints — They Are Not Optional in 2026

Type hints dramatically improve readability, enable IDE autocompletion, and allow static type checkers like mypy to catch bugs before your code runs. Python 3.10+ supports the modern | union syntax.

from typing import Optional, TypedDict
def find_user(user_id: int) -> Optional[dict]:
    ...
def parse_value(raw: str) -> int | float:
    try: return int(raw)
    except ValueError: return float(raw)
class Article(TypedDict):
    title: str
    author: str
    word_count: int
    published: bool

🧪 Write Tests — Even Simple Ones

Untested code is broken code waiting to be discovered. Python's built-in unittest and pytest make testing straightforward. A good test suite catches regressions before they reach production.

import pytest
def test_normal_bmi():
    assert calculate_bmi(70, 1.75) == 22.86
def test_zero_height_raises():
    with pytest.raises(ValueError):
        calculate_bmi(70, 0)

⚠️ 15 Common Python Mistakes Every Developer Makes

Learn from the most frequent errors — save hours of debugging by recognising these traps.

❌ Mistake 1 — Using a mutable default argument

Default argument values are evaluated once when the function is defined, not each time it is called. If you use a mutable object like a list as a default, all calls that use the default share the same object.

def add_item(item, items=[]):   # WRONG — shared list
    items.append(item)
    return items
print(add_item("apple"))   # ['apple']
print(add_item("banana"))  # ['apple', 'banana'] ← surprise!
def add_item_correct(item, items=None):  # CORRECT
    if items is None: items = []
    items.append(item)
    return items

❌ Mistake 2 — Confusing == with is

== checks whether two values are equal. is checks whether two names point to the exact same object in memory. For small integers and interned strings Python reuses objects, which can make is appear to work — until it silently does not.

a = 256; b = 256
print(a is b)   # True (CPython caches small ints)
a = 257; b = 257
print(a is b)   # False — separate objects
print(a == b)    # True — always correct for value checks
# The only correct use of 'is' — checking against None
if result is None: print("No result yet")

❌ Mistake 3 — Modifying a list while iterating over it

Changing the size of a list during a for loop causes items to be skipped silently — one of the hardest bugs to spot.

nums = [1, 2, 3, 4, 5, 6]
for n in nums:          # WRONG — items get skipped
    if n % 2 == 0: nums.remove(n)
print(nums)  # [1, 3, 5] — fragile
# CORRECT — iterate over a copy
nums = [1, 2, 3, 4, 5, 6]
nums = [n for n in nums if n % 2 != 0]
print(nums)  # [1, 3, 5] — always correct

❌ Mistake 4 — Catching too broad an exception

Writing bare except: swallows every error — including typos, import errors, and keyboard interrupts — making bugs nearly impossible to diagnose.

try: result = int(user_input)
except ValueError:             # CORRECT — catch specific
    print("Please enter a valid number")
except TypeError as e:          # CORRECT — handle multiple
    print(f"Type error: {e}")
finally: cleanup()

❌ Mistake 5 — Not understanding variable scope (LEGB)

Python resolves variable names in a specific order: Local → Enclosing → Global → Built-in. Forgetting this leads to confusing UnboundLocalError bugs.

count = 0
def increment():
    global count          # Use sparingly
    count += 1
# Better: pass and return values
def increment(count): return count + 1
count = increment(count)

❌ Mistake 6 — Using + to concatenate strings in a loop

Strings are immutable. Every + creates a new string object and copies both into it. Inside a loop over thousands of items, this creates quadratic time complexity.

words = ["Python", "is", "fast", "and", "readable"]
result = ""
for word in words: result += word + " "  # O(n²) — WRONG
result = " ".join(words)                  # O(n) — CORRECT

❌ Mistake 7 — Forgetting integers and strings do not auto-convert

Python is strongly typed. Unlike JavaScript, it never silently coerces a number to a string or vice versa.

age = 25
print(f"I am {age} years old")   # CORRECT — f-string
print("I am " + str(age) + " years old")  # Also correct

❌ Mistake 8 — Shallow copy vs deep copy confusion

Assigning a list to a new variable does not copy it — both names point to the same object. Only a deep copy is truly independent.

import copy
original = [[1, 2], [3, 4]]
alias = original          # WRONG — just another name
alias[0][0] = 99
print(original)   # [[99, 2], [3, 4]] — original changed!
deep = copy.deepcopy(original)   # CORRECT — fully independent

❌ Mistake 9 — Ignoring return values from string methods

Strings are immutable. Methods like .upper(), .strip(), and .replace() return a new string; they do not modify in place.

name = "  glenn  "
name.strip()          # WRONG — returns new string, name unchanged
name = name.strip()   # CORRECT — capture the returned value

❌ Mistake 10 — Writing loops when a built-in would do

Python ships with rich built-ins. Writing an explicit loop where a one-liner exists is slower and harder to read.

# WRONG — verbose loop
squares = []
for n in range(10):
    if n % 2 == 0: squares.append(n ** 2)
# CORRECT — list comprehension
squares = [n**2 for n in range(10) if n % 2 == 0]
# Use built-ins: sum(), max(), min(), any(), all()

❌ Mistake 11 — Forgetting to use with for file handling

Always use with when opening files. It guarantees the file is closed even if an exception occurs — no resource leaks.

with open("file.txt", "r", encoding="utf-8") as f:
    content = f.read()
# File is automatically closed when the block exits

❌ Mistake 12 — Modifying a dict while iterating over it

Like lists, dictionaries cannot be resized during iteration. Use list(d.items()) to iterate over a copy.

d = {"a": 1, "b": 2, "c": 3}
for k, v in list(d.items()):   # CORRECT — iterate over copy
    if v > 1: del d[k]

❌ Mistake 13 — Using range(len(lst)) instead of enumerate

enumerate is cleaner, faster, and more Pythonic than manually indexing with range(len()).

fruits = ["apple", "banana", "cherry"]
for i, fruit in enumerate(fruits):   # CORRECT
    print(i, fruit)

❌ Mistake 14 — Not using __slots__ for many instances

When creating thousands of objects, __slots__ drastically reduces memory usage by preventing the creation of a __dict__ for each instance.

class Point:
    __slots__ = ("x", "y")
    def __init__(self, x, y): self.x = x; self.y = y

❌ Mistake 15 — Misusing try/except for flow control

Exceptions are for exceptional conditions, not for regular flow control. Use if/else for expected cases.

# WRONG — using try/except for flow control
try: value = d["key"]
except KeyError: value = "default"
# CORRECT — use .get()
value = d.get("key", "default")

🎯 Python Interview Q&A

Most commonly asked Python questions at tech companies — with detailed answers.

Q: What is the difference between a list and a tuple?

A: A list is mutable — you can add, remove, or change elements after creation. A tuple is immutable — once created it cannot be changed. Tuples are faster to iterate and can be used as dictionary keys (they are hashable), whereas lists cannot. Use tuples for data that should not change (coordinates, database records) and lists for collections that will be modified.

Q: What are Python generators and when should you use them?

A: A generator is a function that uses yield instead of return. It returns a generator object — a lazy iterator that produces values one at a time on demand, rather than computing and storing all values at once. Use generators when working with large datasets that do not fit in memory, producing infinite sequences, or building data pipelines. A generator expression uses negligible memory; the equivalent list comprehension would allocate memory for all items immediately.

Q: Explain Python's GIL — what it is and why it matters

A: The Global Interpreter Lock (GIL) is a mutex in CPython that allows only one thread to execute Python bytecode at a time. Even on multi-core processors, Python threads cannot run truly in parallel for CPU-bound tasks. However, the GIL is released during I/O operations, so threading works well for I/O-bound workloads. For true CPU parallelism use multiprocessing or concurrent.futures.ProcessPoolExecutor. Python 3.13 introduced experimental no-GIL builds.

Q: What is the difference between @staticmethod and @classmethod?

A: A @staticmethod receives no implicit first argument — it is essentially a regular function that lives inside a class for organisational purposes. A @classmethod receives the class itself as its first argument (cls). This makes it ideal for alternative constructors — factory methods that create instances in different ways. Use @staticmethod for utilities logically belonging to the class. Use @classmethod when the method needs to know about the class itself, especially for inheritance-aware construction.

Q: How does Python's memory management and garbage collection work?

A: Python uses reference counting as its primary memory management strategy. Every object tracks how many names point to it. When the count drops to zero, memory is immediately freed. However, reference counting cannot handle circular references. CPython's cyclic garbage collector runs periodically to detect and break these cycles. For performance-critical code, minimise circular references and use __slots__ to reduce per-object memory overhead.

Q: What is a Python decorator and how do you write one from scratch?

A: A decorator is a callable that takes a function as input and returns a new function that wraps the original — extending or modifying its behaviour without changing its source code. @my_decorator above a function is syntactic sugar for func = my_decorator(func). Always use @functools.wraps(func) inside your wrapper to preserve the original function's metadata. Decorators power Flask routes, pytest fixtures, @lru_cache, and authentication everywhere in Python.

Q: Explain list comprehensions vs generator expressions vs map()

A: A list comprehension eagerly creates a full list in memory — use when you need to iterate multiple times or need random access. A generator expression is lazy — it produces values on demand — use for large datasets or single-pass pipelines. map() also returns a lazy iterator but requires a function object, making it slightly less readable. In modern Python, list comprehensions and generator expressions are preferred for clarity.

Q: What are dunder methods and why do they matter?

A: Dunder (double underscore) methods — also called magic methods — are how Python implements its data model. When you write a + b, Python calls a.__add__(b). When you write len(obj), Python calls obj.__len__(). By implementing these methods in your own classes, your objects integrate seamlessly with Python's built-in syntax. Key ones: __init__, __repr__, __str__, __eq__, __hash__, __len__, __iter__, __next__, __enter__, __exit__.

Q: What is asyncio and when should you use async Python?

A: asyncio is Python's built-in library for writing concurrent code using the async/await syntax. It is event-loop based — instead of blocking while waiting for I/O, a coroutine yields control back to the event loop, which can run other tasks. Use async Python when your bottleneck is I/O: network requests, database queries, file reads. Do not use it for CPU-bound work — for that, use multiprocessing. The key rule: never call time.sleep() inside an async function — use await asyncio.sleep() instead.

Q: What are Python context managers and how do you create one?

A: A context manager is any object that implements __enter__ and __exit__. The with statement calls __enter__ on entry and __exit__ on exit — even if an exception is raised. This pattern guarantees cleanup: closing files, releasing locks, committing or rolling back database transactions. You can also create context managers with @contextlib.contextmanager — a generator function where code before yield is the setup and code after is the teardown.

✨ Python 3.12+ New Features

What's new in modern Python — features you should be using today.

📝 Improved F-strings (Python 3.12)

You can now use quotes inside f-strings and embed expressions without parentheses in many cases. F-strings are more flexible than ever.

name = "Alice"
print(f"Hello, {name.upper()}!")
# You can now reuse quotes inside f-strings
print(f"Hello {name}")

📦 Type Parameter Syntax (PEP 695)

Python 3.12 introduces a cleaner syntax for type parameters, making generic classes and functions more concise and readable.

def first[T](items: list[T]) -> T:
    return items[0]
class Box[T]:
    def __init__(self, value: T): self.value = value
    def get(self) -> T: return self.value

⚡ Performance Improvements

Python 3.12 brings significant performance improvements, with overall execution speedup of approximately 5-10% for most workloads. The interpreter is faster, and the garbage collector has been optimised.

🔧 Exception Groups (PEP 654)

Exception groups allow you to raise and handle multiple exceptions simultaneously. This is particularly useful for concurrent and asynchronous code.

try:
    raise ExceptionGroup("Multiple errors", [ValueError("invalid"), TypeError("bad type")])
except *ValueError:
    print("Handling ValueError")
except *TypeError:
    print("Handling TypeError")

🔬 Experimental Free-Threading (Python 3.13)

Python 3.13 introduces an experimental build with the GIL disabled, allowing true parallel execution of threads on multi-core systems. This is a major step toward making Python competitive with languages like Go and Rust for CPU-bound workloads.

⚡ Python Performance Tips

Make your Python code faster and more efficient with these proven techniques.

🚀 Use Built-in Functions and Libraries

Built-in functions like sum(), max(), min(), and sorted() are implemented in C and are significantly faster than Python loops. Similarly, libraries like NumPy and Pandas are optimised for speed.

🚀 Avoid Attribute Lookup in Loops

Inside a tight loop, repeatedly looking up an attribute (like method()) can be expensive. Bind it to a local variable before the loop.

# SLOW — attribute lookup every iteration
for x in data: result.append(x**2)
# FAST — bind append to local
append = result.append
for x in data: append(x**2)

🚀 Use List Comprehensions Instead of Loops

List comprehensions are faster than explicit for-loops because they are executed in C and avoid the overhead of repeated Python bytecode execution.

# SLOW — explicit loop
squares = []
for x in range(1000): squares.append(x**2)
# FAST — comprehension
squares = [x**2 for x in range(1000)]

🚀 Use functools.lru_cache for Memoization

For expensive pure functions that are called repeatedly with the same arguments, @lru_cache caches results and dramatically improves performance.

from functools import lru_cache
@lru_cache(maxsize=1024)
def expensive(n):
    # Some expensive computation
    return n ** 3 + n ** 2 + 1

🚀 Use __slots__ to Reduce Memory

When creating many instances of a class, __slots__ reduces memory consumption by preventing the creation of a per-instance __dict__.

class Point:
    __slots__ = ("x", "y")
    def __init__(self, x, y): self.x = x; self.y = y

🔍 Python Debugging Techniques

Effective strategies to find and fix bugs in your Python code.

🐛 Use the Built-in Debugger (pdb)

pdb is Python's built-in interactive debugger. You can set breakpoints, step through code, inspect variables, and evaluate expressions.

import pdb
def buggy_function(x, y):
    pdb.set_trace()  # Debugger will start here
    result = x / y
    return result

🐛 Use print() with repr()

print(repr(variable)) shows you the exact representation of a variable, including escape characters and special characters.

s = "hello\nworld"
print(repr(s))  # 'hello\\nworld'

🐛 Use logging Instead of print

For production code, use logging with appropriate levels (DEBUG, INFO, WARNING, ERROR). This gives you more control and allows you to filter messages by severity.

import logging
logging.basicConfig(level=logging.DEBUG)
logging.debug("Variable x is: %s", x)

🐛 Use breakpoint() (Python 3.7+)

breakpoint() is a simpler way to start the debugger. It's equivalent to pdb.set_trace() but can be configured via the PYTHONBREAKPOINT environment variable.

def process_data(data):
    breakpoint()  # Debugger starts here
    return data * 2

🧪 Python Testing Strategies

Write robust, maintainable tests for your Python applications.

📋 Unit Testing with unittest

Python's built-in unittest framework provides a robust testing environment. Create test classes that inherit from TestCase and write methods starting with test_.

import unittest
def add(a, b): return a + b
class TestMath(unittest.TestCase):
    def test_add(self):
        self.assertEqual(add(2, 3), 5)
        self.assertNotEqual(add(2, 3), 6)
# Run with: python -m unittest test_file.py

🔬 Pytest — The Modern Testing Framework

Pytest is simpler, more powerful, and more widely used than unittest. It supports fixtures, parameterised tests, and rich plugin ecosystem.

import pytest
def test_addition():
    assert 2 + 3 == 5
def test_division_by_zero():
    with pytest.raises(ZeroDivisionError):
        10 / 0
# Run with: pytest test_file.py -v

🎭 Mocking External Dependencies

Use unittest.mock to replace external dependencies (like API calls or database queries) during testing.

from unittest.mock import Mock, patch
def fetch_data(url): return requests.get(url).json()
def test_fetch_data():
    with patch("requests.get") as mock_get:
        mock_get.return_value.json.return_value = {"key": "value"}
        result = fetch_data("https://api.example.com")
        assert result == {"key": "value"}

📦 Python Packaging and Deployment

Package your Python code for distribution and deploy it to production.

📁 Virtual Environments (venv)

Always use a virtual environment for every project. It keeps dependencies isolated and reproducible.

# Create and activate a virtual environment
python -m venv .venv
source .venv/bin/activate        # Linux/macOS
.venv\Scripts\activate           # Windows
# Install dependencies
pip install requests pandas
pip freeze > requirements.txt
# Install from requirements
pip install -r requirements.txt

📦 PyProject.toml (PEP 518/621)

The modern standard for Python project metadata. Use pyproject.toml instead of setup.py for new projects.

[build-system]
requires = ["setuptools>=61.0", "wheel"]
build-backend = "setuptools.build_meta"
[project]
name = "my-package"
version = "0.1.0"
dependencies = ["requests", "pandas"]

🐳 Docker for Python Applications

Containerise your Python application for consistent deployment across environments.

# Dockerfile example
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
CMD ["python", "app.py"]
