PYTHON FUNCTIONS: A COMPLETE GUIDE (2026)

1.0 THE THEORETICAL AND PRACTICAL FOUNDATIONS OF FUNCTIONS

Functions are the fundamental unit of abstraction in Python. They allow you to name a computation, isolate it from surrounding code, parameterise its inputs, and reuse it freely. Without functions, every program would be a single undifferentiated sequence of statements — impossible to test, understand, or extend. Functions are not merely convenient; they are the primary mechanism by which software engineers manage complexity.

Python treats functions as first-class objects. This is not just a technical detail — it is the philosophical cornerstone of the language's expressiveness. A function can be stored in a variable, passed as an argument to another function, returned from a function, placed in a list or dictionary, and created dynamically at runtime. This makes Python a genuinely multi-paradigm language: procedural, object-oriented, and functional all at once, with functions as the thread connecting all three.

1.1 Why This Guide Exists — Our Educational Philosophy

This resource has been designed as a comprehensive educational tool for computer science students, self-directed learners, and professional developers seeking to deepen their understanding of Python functions. Every explanation has been crafted to meet university-level instructional standards: we emphasise not just the "how" but the "why" behind each concept, connecting theory to practical software design.

1.2 Prerequisites and Learning Path

This guide assumes familiarity with Python variables, basic data types (integers, strings, lists, dicts), and simple control flow (if, for, while). No prior knowledge of classes or object-oriented programming is required. If you have not yet studied Python's control flow, we recommend reviewing the POETIC BYTES Python Control Flow Guide first.

1.3 Educational Objectives

  • Define functions using def and understand the anatomy of a Python function: parameters, body, docstring, and return value.
  • Master all parameter kinds — positional, keyword, default, *args, **kwargs, and positional-only / keyword-only separators.
  • Understand scope and the LEGB rule, and correctly use global and nonlocal where appropriate.
  • Build closures and understand how they capture free variables from enclosing scopes.
  • Write and apply decorators with and without arguments, and use functools.wraps to preserve function metadata.
  • Use lambda expressions and higher-order functions like map, filter, sorted, and functools.reduce.
  • Design recursive functions, identify base cases, and understand stack depth limits and tail-call alternatives.
  • Apply type hints to function signatures and understand how they improve IDE support, readability, and static analysis.
  • Implement advanced patterns including partial application, memoisation, and dispatch tables.
  • Recognise and avoid common function pitfalls including mutable default arguments and late-binding closures.

2.0 FUNCTION BASICS — ANATOMY OF A PYTHON FUNCTION

2.1 Defining and Calling a Function

A function is defined with the def keyword, followed by the function name (by convention in snake_case), a parameter list in parentheses, and a colon. The indented body contains the function's logic. A function is called by writing its name followed by parentheses containing any required arguments.

When Python encounters a def statement, it creates a function object and binds it to the name in the current scope — it does not execute the body. The body executes only when the function is called. This means a function can be defined once and called many times, with different arguments each time.

2.2 The return Statement and Return Values

The return statement ends the function's execution and optionally passes a value back to the caller. A function that falls off the end without a return, or that uses a bare return, implicitly returns None. Python functions can return multiple values by returning a tuple — Python's unpacking syntax makes this feel like returning multiple variables simultaneously.

2.3 Docstrings

A docstring is a string literal placed as the first statement of a function body. It documents the function's purpose, parameters, return value, and any exceptions it might raise. Accessible via function.__doc__ and displayed by help(), docstrings are the standard mechanism for in-source documentation in Python. Well-written docstrings are not optional in professional code — they are a contractual specification of the function's behaviour.

💡 Naming Convention: Function names should be snake_case and should describe what the function does, not how it does it. Prefer verb phrases: calculate_tax(), parse_date(), find_user(). Avoid vague names like do_stuff() or process(). A good name makes the call site self-documenting.

EXAMPLE 2.1 — Function Definition, Docstrings, return, and Multiple Returns 
def celsius_to_fahrenheit(celsius: float) -> float:
    """Convert a temperature from Celsius to Fahrenheit.

    Args:
        celsius: Temperature in degrees Celsius.

    Returns:
        The equivalent temperature in degrees Fahrenheit.
    """
    return celsius * 9 / 5 + 32

# Basic call
print(celsius_to_fahrenheit(100))    # 212.0
print(celsius_to_fahrenheit(0))      # 32.0
print(celsius_to_fahrenheit(-40))   # -40.0  (same in both scales)

# Functions are first-class objects — they can be stored in variables
convert = celsius_to_fahrenheit
print(convert(37))                   # 98.6 — body temperature

# Returning multiple values (implicitly a tuple)
def min_max(numbers: list[float]) -> tuple[float, float]:
    """Return the minimum and maximum of a non-empty list."""
    return min(numbers), max(numbers)

lo, hi = min_max([4, 1, 7, 2, 9, 3])
print(f"min={lo}, max={hi}")          # min=1, max=9

# Implicit None return
def greet(name: str) -> None:
    """Print a greeting. Returns None implicitly."""
    print(f"Hello, {name}!")

result = greet("Alice")
print(f"greet() returned: {result!r}")  # None

# Accessing the docstring
print(celsius_to_fahrenheit.__doc__.strip().splitlines()[0])
Output

3.0 PARAMETERS AND ARGUMENTS — THE COMPLETE TAXONOMY

3.1 Positional and Keyword Arguments

Every parameter in a Python function can be passed as either a positional argument (matched by position) or a keyword argument (matched by name). Keyword arguments improve readability at call sites, eliminate order-dependency errors, and make optional parameters self-documenting. When mixing, all positional arguments must come before keyword arguments.

3.2 Default Parameter Values

Parameters can have default values, making them optional at call sites. Defaults are evaluated once at function definition time, not at call time. This is the source of one of Python's most notorious pitfalls: using a mutable object (like a list or dict) as a default value. The same object is reused across all calls, causing shared-state bugs. Use None as the default and initialise inside the function body.

3.3 *args and **kwargs — Variadic Parameters

*args collects all extra positional arguments into a tuple. **kwargs collects all extra keyword arguments into a dictionary. These names are conventional but not mandatory — what matters is the * and ** prefixes. They enable flexible APIs that accept an arbitrary number of inputs without requiring the caller to package them manually.

3.4 Positional-Only and Keyword-Only Parameters

The / separator in a parameter list marks everything before it as positional-only — these parameters cannot be passed by name. The * separator marks everything after it as keyword-only — these parameters must be passed by name. These constraints appear in many built-in functions (len(obj) is positional-only) and are invaluable for API design: they allow renaming implementation-internal parameters without breaking callers.

💡 Parameter Order Rule: Python enforces a strict order in function signatures: positional-only parameters, then regular parameters, then *args, then keyword-only parameters, then **kwargs. Violating this order is a SyntaxError. The mnemonic is: most constrained first, least constrained last.

EXAMPLE 3.1 — All Parameter Kinds: Defaults, *args, **kwargs, /, * 
# ── 3.1 Positional and keyword arguments ────────────────────────────
def describe_pet(animal: str, name: str, age: int = 1) -> str:
    """Return a description of a pet."""
    return f"A {age}-year-old {animal} named {name}."

print(describe_pet("cat", "Whiskers"))           # uses default age
print(describe_pet("dog", "Rex", age=3))         # keyword for age
print(describe_pet(name="Nemo", animal="fish"))  # all keyword

# ── 3.2 The mutable default trap — and the fix ───────────────────────
def append_to(item, lst=None):
    """Safely append item to a list (None default avoids shared state)."""
    if lst is None:
        lst = []
    lst.append(item)
    return lst

print(append_to(1))   # [1]
print(append_to(2))   # [2]  — NOT [1, 2]

# ── 3.3 *args — variadic positional ─────────────────────────────────
def summarise(*values: float) -> dict:
    """Compute summary statistics for any number of numeric inputs."""
    if not values:
        return {}
    return {
        "count": len(values),
        "sum":   sum(values),
        "mean":  sum(values) / len(values),
        "min":   min(values),
        "max":   max(values),
    }

stats = summarise(10, 20, 30, 40, 50)
print(stats)

# ── 3.4 **kwargs — variadic keyword ─────────────────────────────────
def create_profile(name: str, **attributes) -> dict:
    """Build a user profile from a name and any extra keyword attributes."""
    return {"name": name, **attributes}

profile = create_profile("Alice", age=30, role="engineer", city="London")
print(profile)

# ── 3.5 Positional-only (/) and keyword-only (*) ────────────────────
def format_number(value: float, /, *, decimals: int = 2, sep: str = ",") -> str:
    """Format a number.
    value is positional-only (/).
    decimals and sep are keyword-only (*)."""
    formatted = f"{value:,.{decimals}f}"
    if sep != ",":
        formatted = formatted.replace(",", sep)
    return formatted

print(format_number(1234567.891))                        # 1,234,567.89
print(format_number(1234567.891, decimals=0))            # 1,234,568
print(format_number(1234567.891, decimals=3, sep="."))   # 1.234.567.891

4.0 SCOPE AND NAMESPACES — THE LEGB RULE

4.1 What Is a Namespace?

A namespace is a mapping from names to objects. Python maintains separate namespaces for different scopes — the global (module-level) namespace, each function's local namespace, and the built-in namespace. When you write x = 5 inside a function, you create a new binding in that function's local namespace, leaving the global namespace unchanged. When the function returns, its local namespace is destroyed and all local names go out of scope.

4.2 The LEGB Rule

When Python encounters a name, it searches four scopes in order: Local (the current function), Enclosing (any outer functions, from innermost outward), Global (the module level), and Built-in (Python's built-in names like len, print, None). The first match wins. This is the LEGB rule, and understanding it eliminates most "why is my variable the wrong value?" bugs.

4.3 global and nonlocal

By default, assignment inside a function creates a new local variable, even if a global variable with that name exists. To modify a module-level variable from inside a function, declare it with global name. To modify a variable in an enclosing (but not global) scope — typically inside a closure — use nonlocal name. Both statements are rarely needed in well-designed code; frequent use of global is a design smell suggesting the state should be encapsulated in a class.

⚠️ The UnboundLocalError Trap: If you assign to a name anywhere in a function body, Python treats that name as local throughout the entire function — even before the assignment line. Referencing a local name before its assignment raises UnboundLocalError. The fix is either to rename the local variable, or to use global / nonlocal intentionally.

EXAMPLE 4.1 — LEGB Rule, global, and nonlocal 
# ── 4.1 LEGB lookup chain ────────────────────────────────────────────
x = "global"

def outer():
    x = "enclosing"

    def inner():
        x = "local"
        print(f"inner sees:     {x}")   # Local

    inner()
    print(f"outer sees:     {x}")       # Enclosing

outer()
print(f"module sees:    {x}")           # Global
print(f"built-in len:   {len}")         # Built-in

# ── 4.2 global statement ────────────────────────────────────────────
call_count = 0

def increment_counter():
    global call_count
    call_count += 1

increment_counter()
increment_counter()
increment_counter()
print(f"\nGlobal counter: {call_count}")   # 3

# ── 4.3 nonlocal statement ──────────────────────────────────────────
def make_counter(start: int = 0):
    """Return a counter function that increments a private nonlocal state."""
    count = start

    def counter() -> int:
        nonlocal count
        count += 1
        return count

    return counter

c1 = make_counter()
c2 = make_counter(10)

print(f"\nc1: {c1()}, {c1()}, {c1()}")   # 1, 2, 3
print(f"c2: {c2()}, {c2()}")             # 11, 12  — independent state
Output

5.0 CLOSURES AND FREE VARIABLES

5.1 What Is a Closure?

A closure is a nested function that remembers the variables from its enclosing scope, even after that scope has finished executing. More precisely, a closure object bundles together the inner function and a snapshot of the bindings for any free variables — names the function uses but does not define locally. You can inspect a closure's captured variables via function.__closure__ and the values via cell.cell_contents.

5.2 Why Closures Matter

Closures are the mechanism behind many powerful Python patterns. They enable factory functions that produce customised callables, they are the foundation on which decorators are built, and they provide a lightweight alternative to single-method classes. A closure is essentially a function with private, persistent state — without any class machinery. Understanding closures deeply is the prerequisite for understanding decorators.

5.3 The Late-Binding Pitfall

Python closures are late-binding: when the inner function executes, it looks up free variables in the enclosing scope at that moment, not at the time the closure was created. In a loop, all closures created share the same variable reference, so they all see the final value of the loop variable. The fix is to bind the value early using a default argument: lambda i=i: ....

💡 Closures vs Classes: A closure with one "method" (the inner function itself) can replace a class with a single __call__ method. Closures are lighter and often clearer. Prefer a class when the object needs multiple methods, inheritance, or to be identified by type. Prefer a closure for short-lived, single-purpose stateful callables.

EXAMPLE 5.1 — Closures, Factory Functions, and the Late-Binding Trap 
# ── 5.1 Closure factory ──────────────────────────────────────────────
def make_multiplier(factor: float):
    """Return a function that multiplies its argument by factor."""
    def multiply(x: float) -> float:
        return x * factor        # factor is a free variable
    return multiply

double = make_multiplier(2)
triple = make_multiplier(3)

print(f"double(7)  = {double(7)}")   # 14
print(f"triple(7)  = {triple(7)}")   # 21

# Inspect the closure cell
print(f"double captures: {double.__closure__[0].cell_contents}")  # 2

# ── 5.2 Closure with mutable state via nonlocal ──────────────────────
def make_accumulator():
    """Return a function that accumulates a running total."""
    total = 0.0
    def add(amount: float) -> float:
        nonlocal total
        total += amount
        return total
    return add

acc = make_accumulator()
print(f"\nacc(100) = {acc(100)}")   # 100.0
print(f"acc(50)  = {acc(50)}")    # 150.0
print(f"acc(25)  = {acc(25)}")    # 175.0

# ── 5.3 Late-binding trap and fix ────────────────────────────────────
# WRONG: all lambdas capture the same 'i' reference
bad_funcs = [lambda: i for i in range(5)]
print(f"\nbad_funcs[0]() = {bad_funcs[0]()}")   # 4  (not 0!)

# CORRECT: bind i at creation time via a default argument
good_funcs = [lambda i=i: i for i in range(5)]
print(f"good_funcs[0]() = {good_funcs[0]()}")  # 0 ✓
print(f"good_funcs[3]() = {good_funcs[3]()}")  # 3 ✓
Output

6.0 DECORATORS — TRANSFORMING FUNCTIONS WITH @SYNTAX

6.1 What Is a Decorator?

A decorator is a callable that takes a function as its only argument and returns a new callable. The @decorator syntax is syntactic sugar: @my_decorator above a def statement is exactly equivalent to writing func = my_decorator(func) after the definition. Decorators allow you to extend or modify a function's behaviour without changing its source code — a clean application of the Open/Closed Principle.

6.2 Preserving Function Identity with functools.wraps

A naive decorator replaces the original function with its wrapper, losing the original's __name__, __doc__, and other attributes. This breaks introspection, documentation tools, and anything that inspects function metadata. Always apply @functools.wraps(func) to the inner wrapper function — it copies all metadata from the original to the wrapper, making the decorator transparent to callers.

6.3 Decorators with Arguments

To create a decorator that accepts its own arguments, you need one additional layer of nesting: a factory function that takes the arguments and returns the actual decorator. The call stack is then: factory_call(args) → decorator(func) → wrapper(*args, **kwargs). This three-level pattern is the standard form for parameterised decorators.

💡 Stacking Decorators: Multiple decorators can be applied to a single function by stacking them. They apply from bottom to top — the decorator closest to the def is applied first, and the outermost decorator wraps the result. @A @B def f is equivalent to f = A(B(f)).

EXAMPLE 6.1 — Decorators, functools.wraps, and Parameterised Decorators 
import functools
import time

# ── 6.1 Simple decorator — execution timer ───────────────────────────
def timer(func):
    """Measure and print the execution time of any function."""
    @functools.wraps(func)            # preserve __name__, __doc__, etc.
    def wrapper(*args, **kwargs):
        start  = time.perf_counter()
        result = func(*args, **kwargs)
        elapsed = time.perf_counter() - start
        print(f"[timer] {func.__name__!r} took {elapsed:.6f}s")
        return result
    return wrapper

@timer
def slow_sum(n: int) -> int:
    """Sum integers 0..n-1 the slow way."""
    return sum(range(n))

result = slow_sum(1_000_000)
print(f"Result: {result:,}")
print(f"Name preserved: {slow_sum.__name__}")   # 'slow_sum', not 'wrapper'

# ── 6.2 Decorator with arguments — retry on exception ────────────────
def retry(max_attempts: int = 3, exceptions: tuple = (Exception,)):
    """Retry a function up to max_attempts times on specified exceptions."""
    def decorator(func):
        @functools.wraps(func)
        def wrapper(*args, **kwargs):
            last_exc = None
            for attempt in range(1, max_attempts + 1):
                try:
                    return func(*args, **kwargs)
                except exceptions as e:
                    last_exc = e
                    print(f"  Attempt {attempt}/{max_attempts} failed: {e}")
            raise last_exc
        return wrapper
    return decorator

attempt_num = 0

@retry(max_attempts=3, exceptions=(ValueError,))
def flaky_parse(data: str) -> int:
    """Simulates a flaky operation that fails twice then succeeds."""
    global attempt_num
    attempt_num += 1
    if attempt_num < 3:
        raise ValueError(f"Simulated failure (attempt {attempt_num})")
    return int(data)

print()
parsed = flaky_parse("42")
print(f"  Final result: {parsed}")

# ── 6.3 Stacking decorators ──────────────────────────────────────────
def log_call(func):
    """Log function calls."""
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        print(f"\n[log] calling {func.__name__}()")
        return func(*args, **kwargs)
    return wrapper

@log_call
@timer
def compute(n: int) -> int:
    return n * n

compute(7)   # log_call wraps timer-wrapped compute
Output

7.0 LAMBDA EXPRESSIONS AND FUNCTIONAL PROGRAMMING TOOLS

7.1 Lambda Expressions

A lambda expression creates an anonymous function inline: lambda parameters: expression. It is limited to a single expression — no statements, no assignments, no return keyword. The expression's value is automatically returned. Lambdas are not a replacement for def — they are a shorthand for throwaway functions used as arguments to other functions, especially in sorting, filtering, and mapping operations.

7.2 map() and filter()

map(function, iterable) applies a function to every element of an iterable, returning a lazy iterator of results. filter(function, iterable) returns a lazy iterator of elements for which the function returns a truthy value. Both are memory-efficient and composable, but in modern Python they are often superseded by list comprehensions and generator expressions, which most find more readable.

7.3 functools.reduce() and Aggregation

functools.reduce(function, iterable, initializer) applies a binary function cumulatively to elements of an iterable, reducing it to a single value. It is the functional equivalent of an accumulator loop. While reduce is powerful for expressing certain algorithms declaratively (e.g., computing factorials, building nested data structures), Python's built-ins like sum(), max(), any(), and all() cover most common aggregation cases more clearly.

💡 Lambda Readability Rule: If your lambda is hard to read, it should be a named function with a def. The purpose of a lambda is not brevity at all costs — it is to express a simple, self-evident transformation inline. A well-named one-liner def is almost always clearer than a complex lambda.

EXAMPLE 7.1 — Lambda, sorted, map, filter, reduce 
from functools import reduce

# ── 7.1 Lambda with sorted ───────────────────────────────────────────
students = [
    {"name": "Alice",  "gpa": 3.9, "year": 3},
    {"name": "Bob",    "gpa": 3.5, "year": 2},
    {"name": "Carol",  "gpa": 3.9, "year": 1},
    {"name": "Dave",   "gpa": 3.7, "year": 4},
]

# Sort by GPA descending, then name ascending
ranked = sorted(students, key=lambda s: (-s["gpa"], s["name"]))
print("By GPA (desc), name (asc):")
for s in ranked:
    print(f"  {s['name']:<8s} GPA={s['gpa']}")

# ── 7.2 map and filter ───────────────────────────────────────────────
temps_c = [0, 20, 37, 100, -10]
temps_f = list(map(lambda c: c * 9/5 + 32, temps_c))
print(f"\nCelsius:    {temps_c}")
print(f"Fahrenheit: {temps_f}")

above_freezing = list(filter(lambda c: c > 0, temps_c))
print(f"Above 0°C:  {above_freezing}")

# ── 7.3 reduce — build factorial and product ─────────────────────────
factorial_5 = reduce(lambda acc, x: acc * x, range(1, 6))
print(f"\n5! via reduce = {factorial_5}")   # 120

# ── 7.4 Immediately-invoked lambda (IIFE) ────────────────────────────
square_7 = (lambda x: x ** 2)(7)
print(f"7² = {square_7}")

# ── 7.5 any() / all() — functional aggregation ───────────────────────
numbers = [2, 4, 6, 7, 8]
print(f"\nAll even:  {all(x % 2 == 0 for x in numbers)}")   # False
print(f"Any even:  {any(x % 2 == 0 for x in numbers)}")   # True
Output

8.0 RECURSION AND THE CALL STACK

8.1 The Mechanics of Recursion

A recursive function is one that calls itself. Every recursive solution must have two components: a base case that terminates the recursion without a further call, and a recursive case that reduces the problem toward the base case. Without a base case, or if the recursive case never reduces the problem, the function will recurse infinitely until Python raises a RecursionError (default stack limit: 1000 frames).

8.2 When to Use Recursion

Recursion is the natural fit for problems defined recursively — traversing tree structures (file systems, ASTs, JSON), implementing divide-and-conquer algorithms (merge sort, quicksort, binary search), and computing mathematical sequences defined in terms of previous values. For problems that are equally well-expressed iteratively, prefer the iterative form: it is faster, uses O(1) stack space, and does not risk hitting the recursion limit.

8.3 Memoisation with functools.lru_cache

Many naive recursive algorithms recompute the same subproblems exponentially. Memoisation caches the result of each unique input, turning exponential time into linear. Python provides @functools.lru_cache(maxsize=None) (aliased as @functools.cache in Python 3.9+) as a decorator that automatically memoises a pure function — one whose output depends only on its inputs.

⚠️ Recursion Limit: Python's default recursion limit is sys.getrecursionlimit() = 1000. For large inputs, deep recursion will raise RecursionError. Avoid recursion for problems where the depth could exceed a few hundred frames — use an explicit stack or iterative approach instead. Increasing the limit with sys.setrecursionlimit() is possible but rarely the correct solution.

EXAMPLE 8.1 — Recursion, Base Cases, and lru_cache Memoisation 
import functools
import time

# ── 8.1 Classic recursion — factorial ────────────────────────────────
def factorial(n: int) -> int:
    """Return n! recursively. Base case: 0! = 1."""
    if n < 0:
        raise ValueError("Factorial undefined for negative integers.")
    if n == 0:
        return 1                  # base case
    return n * factorial(n - 1)  # recursive case

for n in [0, 1, 5, 10]:
    print(f"{n}! = {factorial(n)}")

# ── 8.2 Naive Fibonacci (exponential time) vs memoised ───────────────
def fib_slow(n: int) -> int:
    """Naive Fibonacci — O(2^n) time due to repeated subproblems."""
    if n <= 1:
        return n
    return fib_slow(n - 1) + fib_slow(n - 2)

@functools.cache
def fib_fast(n: int) -> int:
    """Memoised Fibonacci — O(n) time, each subproblem computed once."""
    if n <= 1:
        return n
    return fib_fast(n - 1) + fib_fast(n - 2)

print()
t0 = time.perf_counter(); fib_slow(32); t_slow = time.perf_counter() - t0
t0 = time.perf_counter(); fib_fast(32); t_fast = time.perf_counter() - t0
print(f"fib_slow(32) = {fib_slow(30):>10}  time: {t_slow:.6f}s")
print(f"fib_fast(32) = {fib_fast(30):>10}  time: {t_fast:.6f}s")
print(f"Speedup: ~{t_slow/max(t_fast,1e-9):.0f}x")

# ── 8.3 Recursive tree traversal ─────────────────────────────────────
def flatten(nested) -> list:
    """Recursively flatten an arbitrarily nested list."""
    result = []
    for item in nested:
        if isinstance(item, list):
            result.extend(flatten(item))   # recurse
        else:
            result.append(item)            # base case
    return result

deeply_nested = [1, [2, [3, [4]], 5], [6, 7], 8]
print(f"\nflatten({deeply_nested})")
print(f"        = {flatten(deeply_nested)}")
Output

9.0 TYPE HINTS AND ANNOTATIONS

9.1 The Purpose of Type Hints

Type hints, introduced in PEP 484 (Python 3.5) and significantly enhanced in subsequent versions, are optional annotations that declare the expected types of function parameters and return values. Python's runtime does not enforce type hints — they are purely informational. Their value comes from: enabling static analysis tools (mypy, pyright) to catch type errors before runtime, improving IDE autocompletion and error highlighting, making function signatures self-documenting, and facilitating code review.

9.2 Modern Syntax with Python 3.10+

Python 3.10 introduced the X | Y union syntax as a cleaner alternative to Union[X, Y]. Use X | None instead of Optional[X]. The from __future__ import annotations import makes all annotations strings by default (postponed evaluation), which is especially useful for forward references and improves startup performance. Python 3.12 introduced type statements for cleaner type aliases.

9.3 Callable, Protocol, and TypeVar

For functions that accept other functions as arguments, use Callable[[ArgTypes], ReturnType] or, more expressively, define a Protocol describing the interface. TypeVar enables generic functions — functions that work correctly with multiple types while preserving the relationship between input and output types. These tools bring the expressiveness of typed functional programming to Python.

💡 Gradual Typing: You do not have to annotate everything at once. Python's type system is gradual — you can annotate the most important public interfaces first, leaving internal helpers unannotated. Start by annotating function signatures; add body annotations only where a static checker asks for clarification. The goal is communication, not exhaustiveness.

EXAMPLE 9.1 — Type Hints, Callable, TypeVar, and Protocol 
from __future__ import annotations
from collections.abc import Callable, Sequence
from typing import TypeVar, ParamSpec

# ── 9.1 Basic parameter and return annotations ───────────────────────
def clamp(value: float, lo: float, hi: float) -> float:
    """Clamp value into [lo, hi]."""
    return max(lo, min(value, hi))

print(clamp(15.0, 0.0, 10.0))   # 10.0
print(clamp(-3.0, 0.0, 10.0))  # 0.0
print(clamp(5.5,  0.0, 10.0))  # 5.5

# ── 9.2 Union types with | (Python 3.10+) ────────────────────────────
def parse_int(value: str | int | None) -> int | None:
    """Parse a value to int, returning None on failure."""
    if value is None:
        return None
    try:
        return int(value)
    except (ValueError, TypeError):
        return None

for v in ["42", "abc", None, 99]:
    print(f"  parse_int({v!r:>6}) = {parse_int(v)!r}")

# ── 9.3 TypeVar for generic functions ────────────────────────────────
T = TypeVar("T")

def first(seq: Sequence[T]) -> T | None:
    """Return the first element of a sequence, or None if empty."""
    return seq[0] if seq else None

print(first([10, 20, 30]))     # 10  (int)
print(first(["a", "b"]))       # 'a' (str)
print(first([]))               # None

# ── 9.4 Callable annotation ──────────────────────────────────────────
def apply_twice(func: Callable[[float], float], value: float) -> float:
    """Apply a float→float function twice to a value."""
    return func(func(value))

import math
print(apply_twice(math.sqrt, 256.0))   # sqrt(sqrt(256)) = 4.0
print(apply_twice(lambda x: x * 2, 3.0))  # 12.0

10.0 ADVANCED PATTERNS — PARTIAL APPLICATION, DISPATCH, AND MORE

10.1 Partial Application with functools.partial

functools.partial(func, *args, **kwargs) returns a new callable that behaves like func but with some arguments pre-filled. This is partial application — a way to specialise a general function into a more specific one without writing a new def. It is particularly useful when adapting a multi-argument function to an interface that expects a single-argument callable, such as a callback or a key function for sorted().

10.2 Dispatch Tables

A dispatch table is a dictionary mapping keys (strings, types, enums) to callable handlers. Instead of a long chain of if/elif statements, you look up the handler for an input in O(1) and call it. Dispatch tables are more extensible, more testable, and more data-driven than conditional chains — adding a new case requires only a new dictionary entry.

10.3 Function Introspection

Python exposes rich introspection through the inspect module. inspect.signature(func) returns a Signature object with the complete parameter list and their defaults. inspect.getsource(func) returns the source code. These tools are the foundation of frameworks like FastAPI (which inspects function signatures to generate API schemas) and pytest (which uses parameter names to inject fixtures).

💡 Single-Dispatch Polymorphism: @functools.singledispatch transforms a regular function into a generic function that dispatches to different implementations based on the type of its first argument. This is Python's functional analogue of method overloading and is ideal for visitor patterns and type-aware processing pipelines.

EXAMPLE 10.1 — functools.partial, Dispatch Tables, and singledispatch 
import functools
import math

# ── 10.1 functools.partial ───────────────────────────────────────────
def power(base: float, exponent: float) -> float:
    return base ** exponent

square = functools.partial(power, exponent=2)
cube   = functools.partial(power, exponent=3)

print("Squares:", [square(x) for x in range(1, 6)])
print("Cubes:  ", [cube(x)   for x in range(1, 6)])

# partial with sorted — sort a list of strings by length then alpha
words = ["banana", "fig", "apple", "kiwi", "date"]
sort_key = functools.partial(lambda word, _: (len(word), word), _=None)
print(sorted(words, key=lambda w: (len(w), w)))

# ── 10.2 Dispatch table ──────────────────────────────────────────────
def op_add(a, b):  return a + b
def op_sub(a, b):  return a - b
def op_mul(a, b):  return a * b
def op_div(a, b):
    if b == 0:
        raise ZeroDivisionError("Cannot divide by zero.")
    return a / b

OPERATIONS: dict[str, callable] = {
    "+": op_add,
    "-": op_sub,
    "*": op_mul,
    "/": op_div,
}

def calculate(a: float, op: str, b: float) -> float:
    """Evaluate a simple arithmetic expression using a dispatch table."""
    if op not in OPERATIONS:
        raise ValueError(f"Unknown operator: {op!r}")
    return OPERATIONS[op](a, b)

print()
for a, op, b in [(10, "+", 5), (10, "-", 3), (6, "*", 7), (22, "/", 7)]:
    print(f"  {a} {op} {b} = {calculate(a, op, b):.4f}")

# ── 10.3 functools.singledispatch ────────────────────────────────────
@functools.singledispatch
def describe(obj) -> str:
    return f"Unknown type: {type(obj).__name__}"

@describe.register(int)
def _(obj: int) -> str:
    return f"Integer: {obj} ({'even' if obj % 2 == 0 else 'odd'})"

@describe.register(str)
def _(obj: str) -> str:
    return f"String: {obj!r} ({len(obj)} chars)"

@describe.register(list)
def _(obj: list) -> str:
    return f"List of {len(obj)} elements, first={obj[0]!r}" if obj else "Empty list"

print()
for val in [42, "hello", [1, 2, 3], 3.14]:
    print(f"  {describe(val)}")

SAMPLE PROGRAM 1 — Pipeline Builder with Decorators

A composable data-processing pipeline built entirely with functions. Demonstrates closures, higher-order functions, functools.reduce, the decorator pattern, and *args parameter packing. Functions are treated as first-class values that can be stored in lists, passed as arguments, and composed dynamically at runtime.

Concepts demonstrated: First-class functions, closures, *args, functools.reduce, function composition, decorator factories, and the pipe operator pattern.

SAMPLE 1 — Composable Data Pipeline 
import functools
from collections.abc import Callable

# ── Pipeline composition ──────────────────────────────────────────────
def compose(*funcs: Callable) -> Callable:
    """Compose functions right-to-left: compose(f, g, h)(x) == f(g(h(x)))."""
    def composed(x):
        return functools.reduce(lambda v, f: f(v), reversed(funcs), x)
    return composed

def pipe(*funcs: Callable) -> Callable:
    """Compose functions left-to-right: pipe(f, g, h)(x) == h(g(f(x)))."""
    def piped(x):
        return functools.reduce(lambda v, f: f(v), funcs, x)
    return piped

# ── Individual transformation functions ──────────────────────────────
def strip_whitespace(text: str) -> str:
    return text.strip()

def to_lower(text: str) -> str:
    return text.lower()

def remove_punctuation(text: str) -> str:
    import string
    return text.translate(str.maketrans("", "", string.punctuation))

def split_words(text: str) -> list[str]:
    return text.split()

def filter_stopwords(words: list[str]) -> list[str]:
    stopwords = {"the", "a", "an", "is", "it", "in", "on", "at", "to", "of", "and", "or"}
    return [w for w in words if w not in stopwords]

def count_frequencies(words: list[str]) -> dict[str, int]:
    freq: dict[str, int] = {}
    for word in words:
        freq[word] = freq.get(word, 0) + 1
    return freq

def top_n(n: int) -> Callable[[dict], list[tuple[str, int]]]:
    """Return a closure that extracts the top-n most frequent words."""
    def _top(freq: dict[str, int]) -> list[tuple[str, int]]:
        return sorted(freq.items(), key=lambda kv: kv[1], reverse=True)[:n]
    return _top

# ── Build the pipeline ────────────────────────────────────────────────
text_pipeline = pipe(
    strip_whitespace,
    to_lower,
    remove_punctuation,
    split_words,
    filter_stopwords,
    count_frequencies,
    top_n(8),
)

text = """
    Python is a high-level programming language that emphasises code readability.
    Python's design philosophy emphasises code readability and the use of significant
    indentation. Python is dynamically typed and garbage-collected. It supports multiple
    programming paradigms including structured, object-oriented and functional programming.
"""

results = text_pipeline(text)
print("TOP 8 WORDS (after stop-word filtering):")
print(f"{'Rank':<6} {'Word':<20} {'Count':>5}")
print("-" * 35)
for rank, (word, count) in enumerate(results, start=1):
    bar = "█" * count
    print(f"  {rank:<4} {word:<20} {count:>3}  {bar}")

# ── Demonstrate compose (right-to-left) ──────────────────────────────
text_normalise = compose(remove_punctuation, to_lower, strip_whitespace)
print(f"\nNormalised: {text_normalise('  Hello, World!  ')!r}")

SAMPLE PROGRAM 2 — Memoised Recursive Expression Evaluator

A recursive descent parser and evaluator for simple arithmetic expressions, combining recursion, closures, memoisation, and type hints. The evaluator tokenises an input string and recursively builds and evaluates an expression tree without constructing an explicit AST, demonstrating how recursion elegantly handles hierarchical structures.

Concepts demonstrated: Recursion with base and recursive cases, closures for shared state between recursive calls, functools.lru_cache memoisation, type hints, and structured exception handling within a recursive call chain.

SAMPLE 2 — Recursive Arithmetic Expression Evaluator 
"""
Recursive descent arithmetic evaluator.
Grammar:
    expr   := term   (('+' | '-') term)*
    term   := factor (('*' | '/') factor)*
    factor := NUMBER | '(' expr ')'
"""
from __future__ import annotations

class ParseError(ValueError):
    pass

def evaluate(expression: str) -> float:
    """Parse and evaluate a simple arithmetic expression string."""
    tokens: list[str] = tokenise(expression)
    pos    = [0]          # mutable position shared by all recursive helpers

    def peek() -> str | None:
        return tokens[pos[0]] if pos[0] < len(tokens) else None

    def consume(expected: str | None = None) -> str:
        token = peek()
        if token is None:
            raise ParseError("Unexpected end of expression.")
        if expected and token != expected:
            raise ParseError(f"Expected {expected!r}, got {token!r}.")
        pos[0] += 1
        return token

    def parse_number() -> float:
        token = consume()
        try:
            return float(token)
        except ValueError:
            raise ParseError(f"Expected a number, got {token!r}.")

    def parse_factor() -> float:
        if peek() == "(":
            consume("(")
            value = parse_expr()
            consume(")")
            return value
        elif peek() in ("+", "-") and (pos[0] == 0 or tokens[pos[0]-1] in ("(", "+", "-", "*", "/")):
            sign = consume()
            return -parse_factor() if sign == "-" else parse_factor()
        else:
            return parse_number()

    def parse_term() -> float:
        left = parse_factor()
        while peek() in ("*", "/"):
            op = consume()
            right = parse_factor()
            if op == "*":
                left *= right
            else:
                if right == 0:
                    raise ZeroDivisionError("Division by zero.")
                left /= right
        return left

    def parse_expr() -> float:
        left = parse_term()
        while peek() in ("+", "-"):
            op = consume()
            right = parse_term()
            left = left + right if op == "+" else left - right
        return left

    result = parse_expr()
    if peek() is not None:
        raise ParseError(f"Unexpected token: {peek()!r}")
    return result


def tokenise(expr: str) -> list[str]:
    """Split an expression string into a list of tokens."""
    import re
    pattern = r'\d+\.?\d*|[+\-*/()]'
    return re.findall(pattern, expr.replace(" ", ""))


# ── Test the evaluator ────────────────────────────────────────────────
test_cases = [
    ("2 + 3",              5.0),
    ("10 - 4 * 2",         2.0),
    ("(10 - 4) * 2",      12.0),
    ("100 / 4 / 5",        5.0),
    ("2 * (3 + 4) * 5",   70.0),
    ("((2 + 3) * (4 - 1)) / 5", 3.0),
]

print(f"{'Expression':<35} {'Expected':>10} {'Got':>10} {'OK':>4}")
print("-" * 65)
for expr, expected in test_cases:
    got = evaluate(expr)
    ok  = "✓" if abs(got - expected) < 1e-9 else "✗"
    print(f"  {expr:<33} {expected:>10.2f} {got:>10.2f} {ok:>4}")

SAMPLE PROGRAM 3 — Decorator Stack: Validation, Logging, and Timing

A realistic API handler decorated with three layered decorators: one for input validation, one for timing, and one for structured logging. Demonstrates the full decorator pattern with arguments, functools.wraps, stacking order, and how real frameworks like Flask and FastAPI use this approach to separate cross-cutting concerns from business logic.

Concepts demonstrated: Decorator factories with arguments, stacking order, functools.wraps, *args/**kwargs pass-through, exception handling inside wrappers, and the separation of cross-cutting concerns.

SAMPLE 3 — Layered Decorator Stack 
import functools
import time
from collections.abc import Callable

# ── Decorator 1: Input type validator ────────────────────────────────
def validate_types(**expected_types):
    """Decorator factory. Validates keyword arguments against expected types."""
    def decorator(func: Callable) -> Callable:
        @functools.wraps(func)
        def wrapper(*args, **kwargs):
            for param, expected in expected_types.items():
                if param in kwargs and not isinstance(kwargs[param], expected):
                    actual = type(kwargs[param]).__name__
                    raise TypeError(
                        f"Parameter '{param}' must be {expected.__name__}, got {actual}."
                    )
            return func(*args, **kwargs)
        return wrapper
    return decorator

# ── Decorator 2: Execution timer ─────────────────────────────────────
def timed(func: Callable) -> Callable:
    """Record execution time and attach it to the return value wrapper."""
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        t0     = time.perf_counter()
        result = func(*args, **kwargs)
        elapsed = time.perf_counter() - t0
        return {"result": result, "elapsed_ms": round(elapsed * 1000, 3)}
    return wrapper

# ── Decorator 3: Structured logger ───────────────────────────────────
def logged(level: str = "INFO"):
    """Decorator factory. Logs call metadata at the specified level."""
    def decorator(func: Callable) -> Callable:
        @functools.wraps(func)
        def wrapper(*args, **kwargs):
            arg_repr = ", ".join(
                [repr(a) for a in args] +
                [f"{k}={v!r}" for k, v in kwargs.items()]
            )
            print(f"[{level}] → {func.__name__}({arg_repr})")
            try:
                result = func(*args, **kwargs)
                print(f"[{level}] ✓ {func.__name__} returned in "
                      f"{result.get('elapsed_ms', '?')}ms")
                return result
            except Exception as e:
                print(f"[ERROR] ✗ {func.__name__} raised {type(e).__name__}: {e}")
                raise
        return wrapper
    return decorator

# ── Business logic — stack the decorators ────────────────────────────
@logged(level="INFO")          # applied 3rd (outermost)
@timed                         # applied 2nd
@validate_types(amount=float, currency=str)  # applied 1st (innermost)
def process_payment(user_id: int, *, amount: float, currency: str) -> dict:
    """Process a payment and return a confirmation."""
    if amount <= 0:
        raise ValueError("Payment amount must be positive.")
    return {
        "user_id":    user_id,
        "amount":     amount,
        "currency":   currency.upper(),
        "status":     "APPROVED",
        "reference":  f"PAY-{user_id:04d}-{int(amount):06d}",
    }

# ── Call with valid data ──────────────────────────────────────────────
print("=== Valid payment ===")
response = process_payment(42, amount=149.99, currency="usd")
print(f"    Confirmation: {response['result']['reference']}")

# ── Call with type error ──────────────────────────────────────────────
print("\n=== Type violation ===")
try:
    process_payment(42, amount="not-a-float", currency="gbp")
except TypeError as e:
    print(f"    Caught: {e}")

# ── Call with business rule violation ────────────────────────────────
print("\n=== Business rule violation ===")
try:
    process_payment(99, amount=-10.0, currency="eur")
except ValueError as e:
    print(f"    Caught: {e}")

# ── Verify decorator transparency ────────────────────────────────────
print(f"\n  __name__ preserved: {process_payment.__name__!r}")
print(f"  __doc__  preserved: {process_payment.__doc__!r}")

11.0 COMMON PITFALLS AND HOW TO AVOID THEM

MistakeProblem DescriptionCorrect Approach
Mutable Default Argument Using def f(lst=[]) — the list is created once at definition time and shared across all calls. Mutations in one call persist to the next. Use None as the default and initialise inside the body: if lst is None: lst = [].
Late-Binding Closures in Loops All closures created in a loop share the same variable reference. When called later, they all see the final value of the loop variable, not the value at creation. Capture the current value immediately using a default argument: lambda i=i: ... or a factory function.
Forgetting functools.wraps A decorator without @functools.wraps(func) replaces the wrapped function's __name__, __doc__, and other attributes with the wrapper's, breaking introspection and documentation tools. Always apply @functools.wraps(func) to the inner wrapper function in every decorator you write.
UnboundLocalError Surprise Referencing a variable before assigning to it in the same function body — Python treats it as local throughout the function. x = x + 1 after using x from the enclosing scope causes this. Use global x or nonlocal x explicitly, or rename the local variable to avoid shadowing the outer one.
Calling vs Passing a Function Writing sorted(lst, key=my_func()) instead of sorted(lst, key=my_func) — the first calls my_func once immediately and passes the return value, not the function itself. Pass the function object without parentheses when providing a callback: key=my_func, callback=handler.
Unbounded Recursion Missing or incorrect base case, or a recursive case that does not reduce the problem, leads to infinite recursion and RecursionError. Always verify your base case handles all terminating inputs and that every recursive call reduces the problem. Add input validation to reject illegal values early.

12.0 KNOWLEDGE CHECK: PYTHON FUNCTIONS QUIZ

1. What is the difference between a parameter and an argument?
Click to reveal answer
A parameter is the variable in the function definition — it is the name used inside the function body. An argument is the actual value passed to the function at the call site. When you write def greet(name):, name is the parameter. When you write greet("Alice"), "Alice" is the argument. The parameter receives the argument's value when the function is called.
2. Why is using a mutable default argument dangerous, and how do you fix it?
Click to reveal answer
Default argument values are evaluated once at function definition time, not on each call. If that default is a mutable object (list, dict, set), every call that mutates it modifies the same shared object, so side effects accumulate across calls. Fix: use None as the default and initialise a new mutable object inside the function body — if items is None: items = []. This creates a fresh object for each call.
3. What does the LEGB rule describe, and in what order does Python search for a name?
Click to reveal answer
LEGB describes Python's variable lookup order: Local (current function's namespace), Enclosing (outer functions' namespaces, from innermost outward), Global (the module's namespace), Built-in (Python's built-in names like len and print). Python searches in this order and stops at the first match. Assignment always creates a binding in the local scope unless global or nonlocal is declared.
4. What is a closure, and what condition is required for a closure to exist?
Click to reveal answer
A closure is a nested function that retains access to variables from its enclosing scope even after that scope has finished executing. Three conditions are required: (1) there must be a nested function, (2) the nested function must reference at least one variable from the enclosing scope (a free variable), and (3) the enclosing function must return (or otherwise pass out) the nested function. The closure object stores the function plus a reference to the captured cells.
5. Why must you apply @functools.wraps to a decorator's wrapper function, and what does it do?
Click to reveal answer
Without @functools.wraps(func), a decorator replaces the original function with its wrapper, causing the wrapped function to impersonate the wrapper: its __name__ becomes "wrapper", its __doc__ is lost, and tools that introspect functions (help(), debuggers, testing frameworks, type checkers) see the wrapper instead of the original. @functools.wraps(func) copies __name__, __doc__, __annotations__, __qualname__, __module__, and __wrapped__ from the original to the wrapper, making the decorator transparent.
