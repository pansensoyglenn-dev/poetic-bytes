PYTHON FUNCTIONS: A COMPLETE GUIDE (2026)

1.0 THE THEORETICAL AND PRACTICAL FOUNDATIONS OF FUNCTIONS

In the landscape of computer science, functions represent one of the most profound abstractions ever devised. At their core, functions are mechanisms for encapsulating behavior they take inputs, perform computations, and produce outputs. But this simple description belies their true power. Functions are the building blocks of modular reasoning, the units of reuse, and the foundation upon which scalable software systems are constructed. When we teach functions, we are not merely teaching syntax; we are teaching a way of thinking about problems that scales from the smallest script to the largest distributed systems.

Python's implementation of functions draws from decades of programming language theory while remaining accessible to beginners. Unlike purely functional languages, Python embraces multiple paradigms: you can write procedural code, object-oriented code, and functional code often within the same function definition. This flexibility makes Python an ideal teaching language because it allows us to introduce concepts incrementally while always having a solid foundation to build upon.

1.1 Why This Guide Exists Our Educational Philosophy

This resource has been designed as a comprehensive educational tool for computer science students, self-directed learners, and professional developers seeking to deepen their understanding. Every explanation has been crafted to meet university-level instructional standards: we emphasize not just the "how" but the "why" behind each concept.

1.2 Prerequisites and Learning Path

This guide assumes basic familiarity with Python syntax variables, data types, conditionals, and loops. If you're entirely new to programming, we recommend starting with a "Python Fundamentals" course before diving into functions.

2.0 FOUNDATIONS: THE ANATOMY OF PYTHON FUNCTIONS

2.1 Function Definition Syntax A Detailed Examination

Every Python function begins with the def keyword, followed by the function name, parentheses containing parameters, and a colon. The indented block that follows constitutes the function body. Functions in Python are first-class objects they can be assigned to variables, passed as arguments, returned from other functions, and stored in data structures.

The function name follows the same rules as variable names: it must start with a letter or underscore, contain only letters, numbers, or underscores, and should be descriptive. Python's style guide (PEP 8) recommends lowercase_with_underscores for function names.

Pro Tip: A well-named function reads like a sentence. calculate_total_price(items) is far more readable than ctp(i). Invest time in naming it pays dividends in maintainability.

2.2 Parameter Types A Comprehensive Taxonomy

Positional Parameters: Values are assigned to parameters based on their position in the function call. The number of arguments must match exactly unless defaults are provided.

Default Parameters: Parameters can have default values, making them optional. Critical nuance: default values are evaluated only once at function definition time, not each call.

Keyword Arguments: Callers can specify which parameter receives which value using the parameter name, improving readability.

Variable Positional Arguments (*args): The *args parameter collects any number of positional arguments into a tuple, enabling functions to accept arbitrary numbers of inputs.

Variable Keyword Arguments (**kwargs): The **kwargs parameter collects any number of keyword arguments into a dictionary, essential for functions that need to accept arbitrary configuration options.

EXAMPLE 1.1 Basic Function Definition and Introspection 
# A simple function with documentation
def calculate_area(length, width):
    """
    Calculate the area of a rectangle.
    
    Parameters:
        length (float): The length of the rectangle.
        width  (float): The width of the rectangle.
    Returns:
        float: The computed area (length * width).
    
    This demonstrates basic function structure:
    - Parameters: length and width (both numbers)
    - Return value: product of length and width
    - Docstring: provides documentation
    """
    area = length * width
    return area

# Calling the function with different arguments
result1 = calculate_area(5, 3)
print(f"Area of 53 rectangle: {result1}")   # Output: 15

# Functions are objects  they have attributes
print(f"Function name: {calculate_area.__name__}")
print(f"Documentation: {calculate_area.__doc__.strip()}")
print(f"Function type: {type(calculate_area)}")

# Assign function to a variable (functions are first-class)
area_calculator = calculate_area
print(f"Area of 72 rectangle: {area_calculator(7, 2)}")  # 14

3.0 VARIABLE SCOPE AND CLOSURES UNDERSTANDING NAMESPACE RESOLUTION

3.1 The LEGB Rule How Python Resolves Names

When Python encounters a variable name in a function, it searches for the definition in a specific order: Local Enclosing Global Built-in. This is the LEGB rule. Understanding this search order is essential for predicting how code will behave and avoiding subtle bugs.

Local: Names defined within the current function. Created when the function is called and destroyed when it returns.

Enclosing: Names in the local scope of enclosing functions (used in nested functions). This is the scope closures capture.

Global: Names defined at the top level of the module, accessible throughout the file.

Built-in: Names predefined in Python's built-in namespace (print, len, range, type, etc.).

3.2 The global and nonlocal Keywords

By default, assignment to a name inside a function creates a new local variable. To modify a global variable inside a function, you must declare it with the global keyword. To modify a variable in an enclosing (but non-global) scope from a nested function, use nonlocal. Overuse of global is a code smell prefer passing values as parameters and returning results.

3.3 Closures Functions That Capture Their Environment

A closure is a nested function that references variables from its enclosing scope even after that enclosing function has finished executing. The inner function "closes over" those variables, creating a persistent, private environment. Closures are the foundation of decorators, memoization, and many functional programming patterns. They provide a lightweight alternative to classes for maintaining state.

Key Insight: Closures give you object-like state without defining a class. They are perfect for factory functions functions that create and return other customized functions.

EXAMPLE 2.1 LEGB Rule and Closures 
# Global scope
global_var = "I am global"

def outer_function(x):
    # Enclosing scope (captured by inner_function)
    enclosing_var = "I am enclosing"
    
    def inner_function(y):
        # Local scope
        local_var = "I am local"
        print(f"Local: {local_var}")
        print(f"Enclosing: {enclosing_var}")   # from outer_function
        print(f"Global: {global_var}")
        return x + y    # x is from enclosing scope
    
    return inner_function   # Return the closure

# Create and call a closure
closure = outer_function(10)   # outer_function returns inner_function
result = closure(5)           # calls inner_function(5), x=10 is remembered
print(f"Result (10+5): {result}")

# Counter factory  closure maintaining state via nonlocal
def counter_factory(start=0):
    """Returns a counter function that remembers its state."""
    count = start
    def increment(step=1):
        nonlocal count       # allow modifying enclosing variable
        count += step
        return count
    return increment

counter_a = counter_factory(0)
counter_b = counter_factory(100)  # independent state

print(f"\ncounter_a: {counter_a()}, {counter_a()}, {counter_a(5)}")  # 1, 2, 7
print(f"counter_b: {counter_b()}, {counter_b()}")              # 101, 102

4.0 LAMBDA FUNCTIONS ANONYMOUS OPERATIONS

Lambda functions are small, anonymous functions defined with the lambda keyword. They can take any number of arguments but must consist of a single expression, which is implicitly returned. Lambdas are syntactic sugar for simple functions, making code more concise when the function logic is trivial. They are particularly valuable when passed as arguments to higher-order functions like map(), filter(), sorted(), and reduce().

4.1 When to Use Lambdas

Use lambdas when the function is simple enough to be expressed in one line and when it is only needed in one place. Common use cases include: sorting with custom keys, filtering sequences with filter(), transforming sequences with map(), and callbacks in asynchronous or event-driven code.

When NOT to Use Lambdas: If the logic is complex, needs a docstring, or is used in multiple places, a named def function is always preferable. Lambdas should never be assigned to variables just to avoid using def that defeats the purpose and PEP 8 discourages it.

4.2 Lambda vs def A Direct Comparison

Syntactically, lambda x: x**2 is equivalent to def f(x): return x**2. The difference is that lambdas are anonymous expressions, while def creates named function objects with full support for docstrings and multiple statements.

EXAMPLE 3.1 Lambda Functions in Action 
# Basic lambda  single argument
square = lambda x: x ** 2
print(f"Square of 7: {square(7)}")   # 49

# Multi-argument lambda
add = lambda a, b: a + b
print(f"5 + 3 = {add(5, 3)}")       # 8

# Sorting  sort list of dicts by a specific key
students = [
    {"name": "Alice", "grade": 88},
    {"name": "Bob", "grade": 95},
    {"name": "Charlie", "grade": 72},
]
sorted_students = sorted(students, key=lambda s: s["grade"], reverse=True)
print("Ranked by grade:", [(s["name"], s["grade"]) for s in sorted_students])

# map()  transform every element
numbers = [1, 2, 3, 4, 5]
cubes = list(map(lambda x: x ** 3, numbers))
print(f"Cubes: {cubes}")           # [1, 8, 27, 64, 125]

# filter()  keep only matching elements
even = list(filter(lambda x: x % 2 == 0, numbers))
print(f"Even numbers: {even}")     # [2, 4]

# reduce()  cumulative folding operation
from functools import reduce
product = reduce(lambda acc, x: acc * x, numbers)
print(f"Product: {product}")       # 120

# Conditional lambda (ternary inside lambda)
abs_val = lambda x: x if x >= 0 else -x
print(f"|7| = {abs_val(-7)}")     # 7

5.0 DECORATORS FUNCTION TRANSFORMATION AND AUGMENTATION

5.1 What Are Decorators?

Decorators are one of Python's most powerful features. A decorator is a function that takes another function as an argument, extends its behavior, and returns a new function (the "wrapper"). This allows you to add functionality to existing functions without modifying their source code this is the Open/Closed Principle from software design: open for extension, closed for modification.

The @decorator_name syntax above a function definition is pure syntactic sugar for func = decorator_name(func). Understanding this equivalence demystifies decorators entirely.

5.2 Common Use Cases

  • Timing: Measure function execution time for performance profiling
  • Logging: Log function calls, arguments, and return values automatically
  • Authentication: Check permissions or tokens before allowing execution
  • Caching / Memoization: Store results of expensive calls for reuse
  • Validation: Validate or sanitize inputs before the function body runs
  • Retry Logic: Automatically retry failed operations with backoff

5.3 Preserving Metadata with @wraps

When you wrap a function, the wrapper obscures the original function's name, docstring, and annotations. Always use @functools.wraps(func) inside your decorator to copy these attributes from the original to the wrapper. This ensures introspection tools, debuggers, and documentation generators work correctly on decorated functions.

5.4 Decorators with Arguments (Decorator Factories)

To create a decorator that accepts its own arguments (e.g., @retry(max_attempts=3)), you need an additional level of nesting: a factory function that accepts the arguments and returns the actual decorator, which in turn returns the wrapper. This is a decorator factory pattern.

EXAMPLE 4.1 Decorator Patterns 
import time
from functools import wraps

#  Basic decorator (no arguments) 
def timer(func):
    """Decorator: measures and prints execution time."""
    @wraps(func)
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = func(*args, **kwargs)
        elapsed = time.perf_counter() - start
        print(f" {func.__name__} took {elapsed:.6f}s")
        return result
    return wrapper

#  Decorator with logging 
def logger(func):
    """Decorator: logs arguments and return value."""
    wraps(func)
    def wrapper(*args, **kwargs):
        sig = ", ".join([repr(a) for a in args])
        print(f" Calling {func.__name__}({sig})")
        result = func(*args, **kwargs)
        print(f" {func.__name__} returned: {result!r}")
        return result
    return wrapper

#  Decorator factory (accepts arguments) 
def retry(max_attempts=3, delay=0.5):
    """Decorator factory: retries function on exception."""
    def decorator(func):
        wraps(func)
        def wrapper(*args, **kwargs):
            for attempt in range(1, max_attempts + 1):
                try:
                    return func(*args, **kwargs)
                except Exception as e:
                    print(f" Attempt {attempt} failed: {e}")
                    if attempt < max_attempts:
                        time.sleep(delay)
            raise RuntimeError(f"All {max_attempts} attempts failed")
        return wrapper
    return decorator

# Stack multiple decorators (applied bottom-up, executed top-down)
timer
logger
def compute_sum(n):
    """Return sum of 0..n-1."""
    time.sleep(0.05)
    return sum(range(n))

result = compute_sum(1000)
print(f"Metadata preserved: {compute_sum.__name__}")  # 'compute_sum', not 'wrapper'

6.0 GENERATORS LAZY EVALUATION WITH YIELD

6.1 Understanding Generators

Generators are functions that can pause and resume their execution, producing a sequence of results over time rather than computing them all at once. They are defined like normal functions but use the yield statement instead of return. When called, a generator function returns a generator object without starting execution. Each call to next() on the generator (or each iteration) advances it to the next yield statement, at which point the function's local state is preserved until the next call.

6.2 Why Generators Matter

Generators provide two major benefits: memory efficiency and the ability to work with infinite sequences. For large datasets, generators avoid loading everything into memory at once a list of 10 million integers consumes hundreds of megabytes, while a generator producing them uses constant memory. For streams, pipelines, or real-time data, generators process values as they arrive rather than waiting for all data to be available.

6.3 Generator Expressions

Just as list comprehensions create lists, generator expressions (using parentheses) create generators lazily. (x**2 for x in range(1000)) creates a generator that computes squares one at a time, consuming almost no memory regardless of the range size. Use them anywhere you'd use a list comprehension but only need to iterate once.

yield vs return: return exits the function permanently. yield pauses it, saving all local variables and the execution position, then resumes from exactly that point on the next call. A function can yield multiple times but can only return once.

EXAMPLE 5.1 Generator Fundamentals 
# Simple countdown generator
def countdown(start):
    """Generator: counts down from start to 1, then signals done."""
    print(f"Starting countdown from {start}")
    while start > 0:
        yield start       # pause here, return value, resume next call
        start -= 1
    print("Countdown finished!")   # runs after last yield

for num in countdown(5):
    print(f"  Tick: {num}")

# Infinite generator (never exhausts)
def integers_from(n):
    """Yield an infinite sequence starting at n."""
    while True:
        yield n
        n += 1

# Safe to use with take-style patterns
gen = integers_from(10)
first_five = [next(gen) for _ in range(5)]
print(f"First five integers from 10: {first_five}")  # [10,11,12,13,14]

# Memory comparison: list vs generator
import sys
list_sq = [x**2 for x in range(1000)]
gen_sq  = (x**2 for x in range(1000))
print(f"List size: {sys.getsizeof(list_sq)} bytes")   # ~8856 bytes
print(f"Generator size: {sys.getsizeof(gen_sq)} bytes") # ~104 bytes

# Generator pipeline  compose generators for data processing
def read_lines(lines):
    for line in lines: yield line.strip()

def filter_empty(lines):
    for line in lines:
        if line: yield line

def uppercase(lines):
    for line in lines: yield line.upper()

raw = ["  hello  ", "", "  world  ", "  python  "]
pipeline = uppercase(filter_empty(read_lines(raw)))
print(list(pipeline))   # ['HELLO', 'WORLD', 'PYTHON']

8.0 TYPE HINTS WRITING SELF-DOCUMENTING, SAFER CODE

8.1 Introduction to Type Hints

Type hints, introduced in Python 3.5 (PEP 484), allow you to annotate the expected types of function parameters and return values. Python remains dynamically typed at runtime type hints are not enforced by the interpreter. However, they enable static type checkers like mypy and pyright to catch type-related errors before runtime, and they serve as machine-checkable documentation that can never go out of date with the code itself.

8.2 Modern Type Hint Syntax (Python 3.10+)

Python 3.9+ allows using built-in types directly as generics: list[int] instead of List[int]. Python 3.10+ introduced the union operator X | Y instead of Union[X, Y]. Python 3.12+ introduced type aliases for cleaner type definitions. These improvements make type hints more readable and reduce the need to import from typing.

8.3 Protocol and Structural Typing

The Protocol class from typing enables structural subtyping (duck typing with type safety): instead of requiring a specific class inheritance, you specify which methods and attributes an object must have. This is Python's way of expressing "I need something that supports these operations" without tying to a concrete type hierarchy.

EXAMPLE 7.1 Type Hints in Practice 
from typing import List, Tuple, Dict, Optional, Union, Callable, Any
from typing import Protocol

# Basic type hints  parameters and return value
def greet(name: str, age: int = 0) -> str:
    return f"Hello {name}, age {age}"

# Collection types
def process_numbers(numbers: List[float]) -> Tuple[float, float, float]:
    """Returns (min, max, average) of the list."""
    return min(numbers), max(numbers), sum(numbers) / len(numbers)

# Optional  value may be None (common for DB lookups)
def find_user(user_id: int) -> Optional[Dict[str, Any]]:
    users = {1: {"name": "Alice"}, 2: {"name": "Bob"}}
    return users.get(user_id)   # returns dict or None

# Union type  accepts multiple types (Python 3.10+: int | str)
def stringify(value: Union[int, float, str]) -> str:
    return str(value)

# Callable type  function as a parameter
def apply_twice(func: Callable[[int], int], x: int) -> int:
    """Apply func to x twice: func(func(x))."""
    return func(func(x))

double = lambda x: x * 2
print(apply_twice(double, 3))   # 12 (3612)

# Protocol  structural subtyping (duck typing + safety)
class Drawable(Protocol):
    def draw(self) -> None: ...

def render(item: Drawable) -> None:
    """Works with any object that has a draw() method."""
    item.draw()

# Access type hints at runtime
print(f"\nType hints: {process_numbers.__annotations__}")

9.0 FUNCTOOLS PROFESSIONAL FUNCTION MANIPULATION

9.1 partial Fixing Arguments

functools.partial allows you to "freeze" some arguments of a function, creating a new function with fewer parameters. This is a form of partial application a functional programming concept. It is particularly useful for adapting functions to interfaces that expect a specific signature (e.g., as callbacks) or for creating specialized versions of generic utilities.

9.2 lru_cache Memoization Made Easy

The @lru_cache decorator automatically caches function return values based on arguments. When the same arguments are passed again, the cached result is returned immediately without re-executing the function. This can transform an exponential-time recursive algorithm into a linear one. LRU stands for "Least Recently Used" when the cache reaches maxsize, the least recently accessed entry is evicted. Using maxsize=None creates an unlimited cache (also available as @cache in Python 3.9+).

9.3 wraps Preserving Function Metadata

When writing decorators, @wraps copies the original function's name, docstring, and annotations to the wrapper. Without it, all decorated functions would appear to have the name "wrapper" and no docstring, breaking introspection, help systems, and tools like Sphinx documentation generators.

9.4 reduce Cumulative Folding

functools.reduce(func, iterable) applies a two-argument function cumulatively to the items of an iterable, reducing it to a single value. It is the functional programming equivalent of a fold operation, powerful for computing running products, building strings character by character, or combining nested structures.

EXAMPLE 8.1 Functools Deep Dive 
from functools import partial, lru_cache, wraps, reduce
import time

#  partial: freeze arguments 
def power(base, exponent):
    """Raise base to exponent."""
    return base ** exponent

square  = partial(power, exponent=2)  # locks exponent=2
cube    = partial(power, exponent=3)  # locks exponent=3
tenth   = partial(power, exponent=10) # locks exponent=10

print(f"square(7): {square(7)}")   # 49
print(f"cube(5): {cube(5)}")       # 125
print(f"tenth(2): {tenth(2)}")     # 1024

#  lru_cache: transform exponential to linear 
# Without cache: O(2^n)  extremely slow for n > 40
lru_cache(maxsize=None)    # unlimited cache
def fib(n):
    """nth Fibonacci number. Memoized for O(n) time."""
    if n < 2:
        return n
    return fib(n-1) + fib(n-2)

print(f"\nfib(50) = {fib(50)}")
print(f"Cache info: {fib.cache_info()}")  # hits, misses, size
fib.cache_clear()   # reset if needed

#  reduce: folding over a sequence 
numbers = [1, 2, 3, 4, 5]
product = reduce(lambda acc, x: acc * x, numbers, 1)
print(f"\nProduct via reduce: {product}")   # 120

# Flatten nested list with reduce
nested = [[1,2], [3,4], [5,6]]
flat = reduce(lambda acc, lst: acc + lst, nested, [])
print(f"Flattened: {flat}")               # [1, 2, 3, 4, 5, 6]

10.0 HIGHER-ORDER FUNCTIONS FUNCTIONS AS FIRST-CLASS CITIZENS

10.1 What Are Higher-Order Functions?

A higher-order function is any function that either takes one or more functions as arguments, returns a function as its result, or both. In Python, functions are first-class objects, which means they can be stored in variables, added to lists, passed to functions, and returned like any other value. This property enables powerful compositional programming patterns.

10.2 Function Composition

Function composition is the act of chaining multiple functions so that the output of one becomes the input of the next: h(x) = f(g(x)). This is a fundamental concept in mathematics and functional programming. In Python, you can implement a generic compose() utility using functools.reduce, enabling you to build complex data transformations from simple, testable pieces.

10.3 Map, Filter, Reduce The Functional Triad

map(func, iterable) applies func to every element, returning a lazy iterator. filter(pred, iterable) keeps only elements where pred returns True. reduce(func, iterable) combines all elements into a single value using a binary function. Together, these three operations can express a vast range of data transformations without explicit loops, and they compose cleanly with lambda and partial.

10.4 Currying

Currying is the technique of transforming a function that takes multiple arguments into a chain of functions each taking a single argument: add(3)(4) instead of add(3, 4). While Python doesn't do this automatically (unlike Haskell), you can implement curried functions manually or with functools.partial. Currying enables elegant point-free style programming and powerful function factories.

EXAMPLE 9.1 Higher-Order Functions and Composition 
from functools import reduce

#  Function Composition 
def compose(*funcs):
    """
    Compose multiple functions right-to-left.
    compose(f, g, h)(x) == f(g(h(x)))
    """
    return lambda x: reduce(lambda v, f: f(v), reversed(funcs), x)

double    = lambda x: x * 2
increment = lambda x: x + 1
square    = lambda x: x ** 2

# Right-to-left: first square, then increment, then double
transform = compose(double, increment, square)
print(f"compose(double, increment, square)(3): {transform(3)}")
# square(3)=9, increment(9)=10, double(10)=20

#  Currying 
def curry(func):
    """Convert a two-argument function into a curried form."""
    return lambda a: lambda b: func(a, b)

add     = curry(lambda a, b: a + b)
multiply= curry(lambda a, b: a * b)

add5    = add(5)        # partially applied
triple  = multiply(3)   # partially applied

print(f"add5(10): {add5(10)}")       # 15
print(f"triple(7): {triple(7)}")   # 21

#  Higher-order pipeline 
def pipeline(*steps):
    """Apply a series of (filter/map/sort) steps to data."""
    def process(data):
        result = data
        for step in steps:
            result = step(result)
        return result
    return process

numbers = range(1, 21)
process = pipeline(
    lambda d: filter(lambda x: x % 2 == 0, d),  # keep evens
    lambda d: map(lambda x: x ** 2, d),          # square them
    lambda d: filter(lambda x: x > 50, d),       # keep > 50
    list                                               # materialize
)
print(f"\nPipeline result: {process(numbers)}")
# Evens: [2,4,6,...,20]  squared  [4,16,36,64,100,144,...]  >50  [64,100,...]

11.0 ASYNC FUNCTIONS CONCURRENT PROGRAMMING WITH ASYNCIO

11.1 What Are Async Functions?

Async functions, defined with async def, are coroutines functions that can be paused and resumed cooperatively. When an async function encounters an await expression, it suspends its execution and yields control back to the event loop, allowing other coroutines to run. This is cooperative multitasking unlike threads, only one coroutine runs at a time, but they interleave at I/O boundaries, making async ideal for I/O-bound workloads like web requests, file operations, and database queries.

11.2 When to Use Async

Async is most beneficial when your program spends significant time waiting for external resources: HTTP requests, database queries, file reads, socket communication. For CPU-bound work (numerical computation, image processing), async provides no benefit use multiprocessing instead. The golden rule: if your bottleneck involves waiting, async; if it involves computing, processes.

11.3 async def vs def Key Differences

Calling a regular function executes it immediately. Calling an async function returns a coroutine object execution doesn't begin until it is awaited or passed to an event loop. This is why you must await async functions, and why async functions can only call other async functions with await (synchronous code can be run in an executor if needed).

Event Loop: The event loop is the runtime that manages and runs coroutines. asyncio.run(main()) creates an event loop, runs the given coroutine until completion, then closes the loop. In production, the loop often stays alive for the entire program lifetime.

EXAMPLE 10.1 Async Functions with asyncio 
import asyncio
import time

#  Basic async function 
async def fetch_data(url: str, delay: float) -> str:
    """Simulate a slow network request without blocking."""
    print(f"   Fetching {url}...")
    await asyncio.sleep(delay)   # yields control; other coros can run
    return f"Data from {url}"

# Sequential (slow): total time = sum of all delays
async def sequential_fetch():
    start = time.perf_counter()
    r1 = await fetch_data("api/users", 1.0)
    r2 = await fetch_data("api/posts", 0.8)
    r3 = await fetch_data("api/comments", 0.6)
    elapsed = time.perf_counter() - start
    print(f"Sequential total: {elapsed:.2f}s")  #  2.4s
    return [r1, r2, r3]

# Concurrent (fast): total time  max single delay
async def concurrent_fetch():
    start = time.perf_counter()
    # gather launches all coroutines simultaneously
    results = await asyncio.gather(
        fetch_data("api/users", 1.0),
        fetch_data("api/posts", 0.8),
        fetch_data("api/comments", 0.6),
    )
    elapsed = time.perf_counter() - start
    print(f"Concurrent total: {elapsed:.2f}s")  #  1.0s
    return results

#  Async context manager 
async def process_with_timeout():
    try:
        # asyncio.timeout cancels if task exceeds duration
        async with asyncio.timeout(0.5):
            result = await fetch_data("api/slow", 2.0)
    except asyncio.TimeoutError:
        print("Request timed out  handled gracefully")

# Run with asyncio.run()  entry point for async programs
async def main():
    print("=== Sequential ===")
    await sequential_fetch()
    print("\n=== Concurrent ===")
    await concurrent_fetch()
    print("\n=== Timeout handling ===")
    await process_with_timeout()

asyncio.run(main())

13.0 COMMON PITFALLS AND HOW TO AVOID THEM

MistakeProblem DescriptionCorrect Approach
Mutable Default ArgumentsUsing a mutable object (list, dict) as a default parameter causes it to be shared across all calls, accumulating data unexpectedly.Use None as sentinel and create the object inside the body:
def func(x, lst=None):
  lst = lst or []
Late Binding in ClosuresA lambda inside a loop captures the variable by reference, not by value all lambdas end up using the final loop value.Capture by value using a default argument:
lambda x, i=i: i * x
Forgetting to ReturnWhen a return statement is missing or a branch is skipped, Python implicitly returns None, causing unexpected results downstream.Always include an explicit return, even when returning None, to make intent clear.
Unintended Variable ShadowingAssigning to a name inside a nested function creates a new local variable, hiding the enclosing one instead of modifying it.Declare nonlocal varname at the top of the inner function to modify the enclosing variable.
Too Many ParametersFunctions with 5+ parameters are hard to call, test, and maintain a sign the function is doing too much.Group related params into a dataclass or TypedDict; split the function into smaller, focused units.
Missing @wraps in DecoratorsWithout @wraps, all decorated functions report the name "wrapper" and have no docstring, breaking debugging and documentation tools.Always add @functools.wraps(func) inside every decorator's wrapper function.

14.0 KNOWLEDGE CHECK: PYTHON FUNCTIONS QUIZ

1. What will be the output of this code?
def func(x, lst=[]):
  lst.append(x)
  return lst
print(func(1))
print(func(2))
Click to reveal answer
This demonstrates the mutable default argument pitfall. The list is created once at function definition, shared across all calls. Output: [1] then [1, 2]. Fix: def func(x, lst=None): lst = lst or []
2. What does the LEGB rule stand for?
Click to reveal answer
Local, Enclosing, Global, Built-in the order Python searches for variable names when resolving a name reference inside a function.
3. What's the difference between return and yield?
Click to reveal answer
return exits the function entirely, destroying all local state. yield pauses the function, saving all local state (variables, execution position), and can be resumed later. A function with yield is a generator.
4. What does @functools.wraps(func) do and why is it important?
Click to reveal answer
It copies the original function's __name__, __doc__, __annotations__, and other metadata to the wrapper function. Without it, all decorated functions appear as "wrapper" with no docstring, breaking introspection, help(), and documentation generators like Sphinx.
5. When should you use async def instead of def?
Click to reveal answer
Use async def for I/O-bound operations network requests, file reads, database queries where the function spends time waiting for external resources. For CPU-bound operations, use regular functions with multiprocessing instead, as async provides no concurrency benefit there.
