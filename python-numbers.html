1.0 UNDERSTANDING NUMERIC TYPES IN PYTHON

Numbers are the foundation of almost every Python program ‚Äî from simple counters to machine learning models, from currency calculations to physics simulations. Python's float type follows the IEEE 754 double-precision standard, which means it shares the same representation and rounding behaviour as floats in virtually every other modern language. Understanding these design decisions ‚Äî and their consequences ‚Äî is what separates a Python programmer who truly understands numbers from one who only uses them.

2.0 THE INTEGER TYPE ‚Äî ARBITRARY-PRECISION ARITHMETIC

2.1 Python Integers Are Unbounded

Unlike every fixed-width integer type in C, Java, or Rust, Python's int has no maximum or minimum value. It grows as large as memory allows. CPython implements integers using a variable-length array of 30-bit "digits" (platform-dependent), so arithmetic on large integers is slightly slower than on machine-word integers, but the correctness guarantee is absolute ‚Äî there is no overflow, no silent wraparound, and no undefined behaviour. This makes Python integers ideal for cryptography, combinatorics, and exact big-number arithmetic.

2.2 Integer Literals and Number Separators

Python supports integer literals in four bases: decimal (no prefix), binary (0b or 0B), octal (0o or 0O), and hexadecimal (0x or 0X). Since Python 3.6, underscores can be used as visual separators in numeric literals: 1_000_000 is identical to 1000000. This greatly improves readability for large constants and binary patterns.

2.3 Boolean as a Subclass of int

bool is a subclass of int in Python. True has the value 1 and False has the value 0. This means booleans can be used directly in arithmetic: True + True == 2, sum([True, False, True, True]) == 3. This design is intentional and enables elegant patterns like counting truthy values in a list with sum(x > 0 for x in data).

üí° Practical Rule: Use underscores in large integer literals for readability: POPULATION = 8_200_000_000. Use hex literals for bit masks and memory addresses: MASK = 0xFF_FF_FF_FF. Use binary literals when working with flags: FLAGS = 0b1010_1100.

3.0 THE FLOAT TYPE ‚Äî IEEE 754 DOUBLE PRECISION

3.1 What IEEE 754 Means in Practice

Python's float is a C double ‚Äî a 64-bit IEEE 754 double-precision floating-point number. It has a sign bit, an 11-bit exponent, and a 52-bit mantissa (significand), giving approximately 15‚Äì17 significant decimal digits of precision and a range from roughly ¬±5√ó10‚Åª¬≥¬≤‚Å¥ to ¬±1.8√ó10¬≥‚Å∞‚Å∏. The critical consequence of binary representation is that most decimal fractions cannot be stored exactly. 0.1 in binary is a repeating fraction, just as 1/3 is a repeating fraction in decimal. The stored value is the closest 64-bit binary approximation.

3.2 Special Float Values

IEEE 754 defines several special values that Python supports: positive and negative infinity (float('inf') and float('-inf')), and Not a Number (float('nan')). Infinity is returned by operations that overflow the float range, such as 1e308 * 10. NaN is the result of undefined operations like float('inf') - float('inf') or 0.0 / 0.0 (which raises ZeroDivisionError in Python, but NaN can still be obtained via math.nan). NaN has a critical property: it is not equal to anything, including itself ‚Äî float('nan') == float('nan') is False.

3.3 Float Comparison ‚Äî Never Use == Directly

Comparing floats with == is almost always a bug. Because of rounding errors, two mathematically equal expressions may produce slightly different float values. The correct approach is to test whether two floats are within an acceptable tolerance: use math.isclose(a, b) for relative comparison (the default relative tolerance is 1e-9), or compare abs(a - b) < epsilon for absolute tolerance. For exact equality checks, use integers or the Decimal type instead.

‚ö†Ô∏è Never Compare Floats with ==: 0.1 + 0.2 == 0.3 evaluates to False in Python. Use math.isclose(0.1 + 0.2, 0.3) which returns True. This is not a Python bug ‚Äî it is a fundamental property of IEEE 754 binary floating-point, shared by every language that uses the same representation.

4.0 COMPLEX NUMBERS ‚Äî BUILT-IN SUPPORT FOR IMAGINARY ARITHMETIC

4.1 Complex Number Literals and Construction

Python has first-class support for complex numbers ‚Äî they are a built-in type, not a library. A complex literal is written as a + bj where j (not i) is the imaginary unit. The real and imaginary parts are both stored as IEEE 754 doubles. Complex numbers can be constructed with the complex(real, imag) constructor or by using the j suffix on a numeric literal: 3j is the imaginary number 0 + 3i. Complex numbers support all arithmetic operations: addition, subtraction, multiplication, division, and power. They do not support comparison with < or > ‚Äî complex numbers have no natural ordering.

4.2 The cmath Module for Complex Mathematics

The standard math module operates only on real numbers ‚Äî passing a complex number raises a TypeError. For complex mathematics, Python provides the cmath module, which mirrors math but operates on complex values. cmath.sqrt(-1) returns 1j rather than raising an error. Key functions include cmath.phase() for the argument (angle) of a complex number, cmath.polar() for polar form, and cmath.rect() to convert from polar back to rectangular form.

4.3 Real-World Uses of Complex Numbers

Complex numbers appear in signal processing (Fourier transforms), electrical engineering (impedance calculations), quantum mechanics, control systems, and 2D geometry (where the complex plane represents a 2D coordinate system with rotation via multiplication). Python's native complex type makes these domains feel natural ‚Äî no third-party library is required for basic complex arithmetic.

5.0 ARITHMETIC OPERATORS ‚Äî COMPLETE REFERENCE

5.1 The Seven Arithmetic Operators

Python provides seven arithmetic operators: addition (+), subtraction (-), multiplication (*), true division (/), floor division (//), modulo (%), and exponentiation (**). The critical one to understand is / vs //: in Python 3, / always returns a float (true division), even when both operands are integers. The // operator performs floor division, rounding the result towards negative infinity, and returns an int when both operands are int.

5.2 Modulo and Its Relationship with Floor Division

Python's modulo operator % always returns a result with the same sign as the divisor, not the dividend. This is consistent with floor division: a == (a // b) * b + (a % b) always holds. This differs from C/Java where the sign of the remainder matches the dividend. Python's behaviour is mathematically cleaner for calendar arithmetic, cycle detection, and modular arithmetic. For example, -7 % 3 is 2 in Python (not -1 as in C), because floor division of -7 by 3 is -3 (rounds toward negative infinity), and -3 * 3 + 2 == -7.

5.3 Augmented Assignment Operators

All arithmetic operators have augmented assignment forms: +=, -=, *=, /=, //=, %=, and **=. For immutable types like int and float, these create a new object and rebind the name ‚Äî they do not mutate in place. For mutable numeric containers (like numpy arrays), the augmented forms often operate in-place and are significantly more efficient.

üí° divmod() Built-in: When you need both the quotient and remainder of a division, use the divmod(a, b) built-in function. It returns (a // b, a % b) as a tuple and is typically faster than computing each separately, since the CPU computes both at once in many architectures.

6.0 BITWISE OPERATORS ‚Äî BIT-LEVEL INTEGER MANIPULATION

6.1 The Six Bitwise Operators

Python supports six bitwise operators that operate on the binary representations of integers: AND (&), OR (|), XOR (^), NOT (~), left shift (<<), and right shift (>>). These operators treat integers as sequences of bits and apply logical operations to each bit position independently. They work on Python's arbitrary-precision integers, so bit operations on values larger than 64 bits are fully supported ‚Äî unlike in C where the behavior depends on the word size.

6.2 Bitwise NOT ‚Äî The Two's Complement Surprise

The ~n operator implements bitwise NOT, which inverts all bits. For Python integers, the result is always ~n == -(n + 1). This follows from two's complement representation: ~0 == -1, ~1 == -2, ~255 == -256. This identity is useful for creating mask complements and toggle patterns. Unlike in fixed-width languages, there is no risk of sign extension issues since Python integers can grow arbitrarily.

6.3 Practical Applications

Bitwise operators are essential in systems programming, permissions modeling, binary protocol parsing, and performance-critical flag checking. Setting a bit uses OR: flags | MASK. Clearing a bit uses AND with NOT: flags & ~MASK. Toggling a bit uses XOR: flags ^ MASK. Testing a bit uses AND: bool(flags & MASK). Left shift by n is equivalent to multiplication by 2‚Åø, and right shift by n is equivalent to floor division by 2‚Åø ‚Äî both are generally faster than multiplication on hardware.

EXAMPLE 6.1 ‚Äî Bitwise Operators and Flag Patterns

7.0 DECIMAL ‚Äî EXACT DECIMAL ARITHMETIC FOR FINANCIAL CODE

7.1 Why Decimal Exists

The decimal.Decimal type implements the IBM General Decimal Arithmetic specification, providing exact decimal floating-point arithmetic. Unlike float, which is binary, Decimal stores numbers in base 10, so Decimal('0.1') + Decimal('0.2') == Decimal('0.3') is True. This makes it indispensable for financial, accounting, and tax calculations where binary rounding errors are unacceptable. The price for this accuracy is speed ‚Äî Decimal arithmetic is significantly slower than float arithmetic and is implemented in Python (though a C accelerator exists in CPython).

7.2 Context and Precision

Every Decimal operation is governed by a context object that controls the number of significant digits (default: 28), rounding mode (default: ROUND_HALF_EVEN, also called banker's rounding), and which exceptional conditions raise errors vs. set flags. You can modify the global context with decimal.getcontext().prec = 50 or create a local context with decimal.localcontext(). The rounding mode ROUND_HALF_EVEN rounds 0.5 to the nearest even digit to eliminate systematic bias ‚Äî this is the IEEE 754 default and the statistically correct choice for accumulated calculations.

7.3 Always Construct from Strings

A critical rule: always construct Decimal values from strings, not from floats. Decimal(0.1) creates a Decimal from the float 0.1, which is already imprecise, so you get Decimal('0.1000000000000000055511151231257827021181583404541015625'). Decimal('0.1') creates an exact Decimal representation of the number one-tenth. The difference is critical for correctness.

‚ö†Ô∏è Always Use Strings with Decimal: Decimal(0.1) does NOT give you exact one-tenth. Use Decimal('0.1') always. This is the single most important rule when working with the Decimal type ‚Äî a common mistake in financial code that produces subtle, hard-to-debug errors.

EXAMPLE 7.1 ‚Äî Decimal for Exact Financial Arithmetic

# ‚îÄ‚îÄ Exact arithmetic ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ

# ‚îÄ‚îÄ Currency calculation example ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ

# ‚îÄ‚îÄ Custom precision with localcontext ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ

# ‚îÄ‚îÄ Converting from float safely ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ

8.0 FRACTION ‚Äî EXACT RATIONAL ARITHMETIC

8.1 Exact Representation of Rational Numbers

The fractions.Fraction type represents rational numbers as exact numerator/denominator pairs. It auto-reduces fractions to lowest terms: Fraction(2, 4) becomes Fraction(1, 2). Unlike float or Decimal, Fraction never rounds ‚Äî arithmetic results are exact, at the cost of potentially very large numerators and denominators. Fraction(1, 3) + Fraction(1, 6) == Fraction(1, 2) is True because both sides reduce to exactly one-half.

8.2 Construction and Conversion

Fraction can be constructed from two integers, from a string (Fraction('1/3')), from a decimal string (Fraction('0.5')), or from a float (Fraction(0.1) ‚Äî which, like Decimal, gives the exact rational representation of the float's stored value, not the intended decimal fraction). The limit_denominator(max_denominator) method finds the closest rational approximation with a denominator no larger than the given limit ‚Äî useful for recognizing rational patterns in float data.

8.3 When to Use Fraction

Fraction is ideal for educational tools, symbolic mathematics, recipe scaling, unit conversion systems, and any domain where fractions must remain exact through arbitrary arithmetic. It is significantly slower than both float and Decimal, so it is not appropriate for performance-critical numerical computation. Use it when exactness of rational relationships is the priority and speed is not.

EXAMPLE 8.1 ‚Äî Fraction for Exact Rational Arithmetic

10.0 NUMBER FORMATTING ‚Äî F-STRINGS AND FORMAT SPECIFICATIONS

10.1 The Format Specification Mini-Language

Python's f-string and format() function support a powerful format specification mini-language for controlling how numbers are displayed. The general form is {value:[[fill]align][sign][z][#][0][width][grouping_option][.precision][type]}. The most important components are width (minimum field width), .precision (decimal places for floats, significant digits for general format), and type (d for integer, f for fixed-point float, e for scientific, g for general, % for percentage, b/o/x for integer bases, n for locale-aware formatting).

10.2 Thousands Separators and Locale Formatting

The , grouping option in format specs adds comma thousands separators: f"{1234567:,}" gives "1,234,567". The _ grouping option uses underscores: f"{1234567:_}" gives "1_234_567". For locale-aware formatting (e.g. European thousands separators using dots), use the n type with locale.setlocale(), or use the babel library for internationalised number formatting in production code.

10.3 Scientific and Percentage Notation

Scientific notation uses the e or E type: f"{0.000123:.2e}" gives "1.23e-04". The g type automatically switches between fixed and scientific notation based on magnitude. Percentage formatting multiplies by 100 and appends %: f"{0.857:.1%}" gives "85.7%". These formats are essential for scientific output, financial reporting, and data visualisation.

EXAMPLE 10.1 ‚Äî Number Formatting with f-strings and Format Specs

11.0 TYPE CONVERSION AND NUMERIC COERCION

11.1 Explicit Conversion Functions

Python provides explicit conversion functions for all numeric types: int(x), float(x), complex(x). int() truncates floats toward zero (same as math.trunc()) ‚Äî it does not round. int('42') parses a decimal string. int('0x1a', 16) parses a hexadecimal string. float('inf') and float('nan') create special float values. Conversion from Decimal to float may lose precision if the Decimal value has more significant digits than a float can represent.

11.2 Implicit Numeric Coercion

Python applies a numeric tower for implicit type coercion in mixed-type arithmetic. When you combine an int and a float, the result is a float. When you combine a float and a complex, the result is complex. When you divide two ints with /, the result is always float. This tower is: int ‚Üí float ‚Üí complex. bool sits below int in the tower since it is a subclass. Decimal and Fraction do not participate in this implicit coercion ‚Äî mixing them with float raises a TypeError to prevent silent precision loss.

11.3 Safe Numeric Conversion Patterns

In real-world code, numeric input often comes as strings and may be malformed. Always wrap numeric conversions in try/except ValueError blocks, or use a helper function that returns a default value on failure. For scientific data with potential NaN or Inf values, use math.isfinite() to validate before storing. When converting user-facing decimal strings to Decimal, always strip whitespace and validate the format first.

EXAMPLE 11.1 ‚Äî Type Conversion, Coercion, and Safe Parsing 
import math
from decimal import Decimal

# ‚îÄ‚îÄ Explicit conversion ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
print(int(3.9))           # 3    (truncates, does NOT round)
print(int(-3.9))          # -3   (toward zero)
print(float(7))           # 7.0
print(complex(5))         # (5+0j)
print(complex(3, 4))      # (3+4j)

# ‚îÄ‚îÄ Implicit coercion: numeric tower ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
print(type(1 + 2))          # <class 'int'>
print(type(1 + 2.0))        # <class 'float'>
print(type(1.0 + 2j))       # <class 'complex'>
print(type(7 / 2))          # <class 'float'>  (always)
print(type(True + 1))        # <class 'int'>  (bool is int subclass)

# ‚îÄ‚îÄ Decimal does NOT mix with float implicitly ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
try:
    result = Decimal('1.5') + 0.5
except TypeError as e:
    print(f"TypeError: {e}")  # unsupported operand type(s) for +: 'Decimal' and 'float'

result = Decimal('1.5') + Decimal('0.5')  # correct: keep types consistent

# ‚îÄ‚îÄ Safe string-to-number parsing ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
def safe_int(s: str, default: int = 0) -> int:
    try:
        return int(s.strip())
    except (ValueError, AttributeError):
        return default

def safe_float(s: str, default: float = float('nan')) -> float:
    try:
        result = float(s.strip())
        return result if math.isfinite(result) else default
    except (ValueError, AttributeError):
        return default

print(safe_int("  42  "))   # 42
print(safe_int("abc"))     # 0  (default)
print(safe_float("3.14"))  # 3.14
print(safe_float("inf"))   # nan (default ‚Äî inf rejected as non-finite)

# ‚îÄ‚îÄ Number theory: check if float is whole number ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
print(4.0.is_integer())    # True
print(4.5.is_integer())    # False

12.0 REAL-WORLD NUMERIC PATTERNS IN PRODUCTION CODE

12.1 Financial Calculations with Decimal

Every financial application that processes money must use Decimal, not float. The correct pattern is: receive monetary values as strings (from user input, database TEXT/VARCHAR, or JSON), construct Decimal objects immediately, perform all arithmetic in Decimal, round only at display time using quantize(), and store the result as a string or as an integer number of cents. Never store money as a float in a database ‚Äî the rounding errors accumulate and reconciliation becomes impossible.

12.2 Statistical Calculations

Python 3.8+ includes the statistics module with functions like statistics.mean(), statistics.median(), statistics.stdev(), and statistics.variance(). These are numerically stable implementations safe for general use. For scientific computing with large arrays, NumPy is the standard ‚Äî its float64 arrays are backed by C doubles and are orders of magnitude faster than Python lists of floats. The statistics module also accepts Decimal and Fraction inputs for exact computation.

12.3 Bit Manipulation in Networking and Protocols

Binary protocols, network packet parsing, and hardware interface code often require precise bit manipulation. Python's arbitrary-precision integers make this safe and readable: IP address calculations, CRC computation, bitmask operations for hardware registers, and protocol field extraction all benefit from Python's clean bitwise operator syntax and the ability to work with integers of any size.

EXAMPLE 12.1 ‚Äî Real-World Numeric Patterns 
from decimal import Decimal, ROUND_HALF_EVEN
import statistics, math, struct

# ‚îÄ‚îÄ Pattern 1: Money arithmetic pipeline ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
def process_invoice(items: list[tuple[str, str, str]]) -> dict:
    """items: [(description, unit_price_str, qty_str)]"""
    TWO_DP = Decimal('0.01')
    TAX    = Decimal('0.20')  # 20% VAT
    lines  = []
    subtotal = Decimal('0')
    for desc, price_s, qty_s in items:
        price = Decimal(price_s)
        qty   = Decimal(qty_s)
        line  = (price * qty).quantize(TWO_DP, ROUND_HALF_EVEN)
        lines.append({'desc': desc, 'line_total': line})
        subtotal += line
    tax   = (subtotal * TAX).quantize(TWO_DP, ROUND_HALF_EVEN)
    total = subtotal + tax
    return {'lines': lines, 'subtotal': subtotal, 'tax': tax, 'total': total}

invoice = process_invoice([
    ("Widget A", "9.99",  "3"),
    ("Widget B", "14.50", "2"),
])
print(f"Subtotal: ¬£{invoice['subtotal']}")  # ¬£58.97
print(f"VAT:      ¬£{invoice['tax']}")       # ¬£11.79
print(f"Total:    ¬£{invoice['total']}")     # ¬£70.76

# ‚îÄ‚îÄ Pattern 2: Statistics module ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
exam_scores = [72, 88, 91, 65, 94, 78, 83, 90, 71, 88]
print(f"Mean:   {statistics.mean(exam_scores):.1f}")
print(f"Median: {statistics.median(exam_scores)}")
print(f"StdDev: {statistics.stdev(exam_scores):.2f}")
print(f"Mode:   {statistics.mode(exam_scores)}")  # 88

# ‚îÄ‚îÄ Pattern 3: IPv4 address as 32-bit integer ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
def ip_to_int(ip: str) -> int:
    parts = [int(p) for p in ip.split('.')]
    return (parts[0] << 24) | (parts[1] << 16) | (parts[2] << 8) | parts[3]

def int_to_ip(n: int) -> str:
    return '.'.join(str((n >> ((3-i)*8)) & 0xFF) for i in range(4))

ip_n = ip_to_int("192.168.1.100")
print(f"{ip_n} = 0x{ip_n:08X}")   # 3232235876 = 0xC0A80164
print(int_to_ip(ip_n))             # 192.168.1.100

# ‚îÄ‚îÄ Pattern 4: Normalise a value to [0, 1] ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
def normalise(value: float, lo: float, hi: float) -> float:
    if math.isclose(lo, hi):
        raise ValueError("lo and hi must differ")
    return (value - lo) / (hi - lo)

print(normalise(75, 0, 100))    # 0.75
print(normalise(-10, -20, 20))  # 0.25

13.0 COMMON PITFALLS AND HOW TO AVOID THEM

MistakeProblem DescriptionCorrect Approach
Comparing Floats with == 0.1 + 0.2 == 0.3 is False due to IEEE 754 rounding. This catches many developers off-guard and can produce silent logic errors in conditional branches. Use math.isclose(a, b) for relative comparison, or abs(a-b) < epsilon for absolute tolerance. For exact equality, use Decimal or Fraction.
Using float for Money Storing currency as float accumulates rounding errors that compound across thousands of transactions. 0.1 + 0.2 is already off by 1e-16, and this error grows with each operation. Always use decimal.Decimal for financial calculations. Construct from strings: Decimal('19.99'). Store as integer cents or as a string in databases.
Constructing Decimal from float Decimal(0.1) does NOT give exact one-tenth ‚Äî it creates a Decimal from the float 0.1, which already carries IEEE 754 imprecision, producing a very long exact representation of that imprecise value. Always construct Decimal from strings: Decimal('0.1'). This is the single most important Decimal rule. If you receive a float, convert it via string with controlled precision: Decimal(f'{value:.2f}').
Expecting int Division to Return Float Coming from Python 2 or other languages, developers expect 7 / 2 to give 3 (integer division). In Python 3, / always returns a float: 7 / 2 == 3.5. Using the result in a list index raises TypeError. Use // for floor division when you need an integer result: 7 // 2 == 3. Use int(7 / 2) if you need a float division then truncation. Never use / when indexing.
Misunderstanding Negative Modulo Python's % operator returns a result with the same sign as the divisor, not the dividend. -7 % 3 is 2, not -1. Developers from C/Java expect -1 and get logically wrong results in modular arithmetic. Remember the invariant: a == (a // b) * b + (a % b) always holds. Python's modulo is mathematically correct for modular arithmetic. If you need C-style behaviour, use math.fmod(a, b) which preserves the sign of the dividend.
Operator Precedence with ** -3 ** 2 evaluates as -(3 ** 2) = -9, not (-3) ** 2 = 9. The unary minus has lower precedence than **. This is mathematically correct but surprises many developers. Always parenthesise the base when it is negative: (-3) ** 2. Make precedence explicit in any non-obvious power expression.
NaN Comparisons float('nan') == float('nan') is False. NaN is never equal to anything, including itself. Code that tests for NaN with == float('nan') silently fails to detect it. Use math.isnan(x) to test for NaN. Use math.isfinite(x) to test that a value is neither NaN nor infinite. Never use == to compare with float('nan').

15.0 KNOWLEDGE CHECK: PYTHON NUMBERS QUIZ

1. Why does 0.1 + 0.2 == 0.3 evaluate to False in Python? How do you correctly compare these values?
Click to reveal answer
Python floats are IEEE 754 double-precision binary floating-point numbers. The decimal values 0.1 and 0.2 cannot be represented exactly in binary ‚Äî they are stored as the closest 64-bit approximation. When added, the accumulated rounding error makes the result slightly different from the float representation of 0.3. This is not a Python bug; it is universal across all IEEE 754 implementations. The correct approach is math.isclose(0.1 + 0.2, 0.3), which uses a relative tolerance to check if two floats are "close enough." For exact decimal arithmetic, use Decimal('0.1') + Decimal('0.2') == Decimal('0.3'), which is True.
2. What is the result of -7 // 2 and -7 % 2 in Python? Why is this different from C?
Click to reveal answer
-7 // 2 is -4 (not -3) because Python's floor division always rounds toward negative infinity. -7 % 2 is 1 (not -1) because the result has the same sign as the divisor. The invariant a == (a // b) * b + (a % b) always holds: -7 == (-4) √ó 2 + 1 == -8 + 1 == -7. In C, integer division truncates toward zero, so -7 / 2 == -3 and -7 % 2 == -1. Python's behaviour is mathematically more consistent for modular arithmetic. Use math.fmod(a, b) if you need C-style truncation-based remainder.
3. What is wrong with Decimal(0.1)? How should you construct a Decimal representing one-tenth?
Click to reveal answer
Decimal(0.1) first evaluates the float literal 0.1, which is already the closest IEEE 754 double approximation to one-tenth ‚Äî not exactly one-tenth. Decimal then converts that imprecise float to an exact Decimal representation of its actual stored value, giving something like Decimal('0.1000000000000000055511151231257827021181583404541015625'). The correct way is Decimal('0.1') ‚Äî constructing from a string bypasses the float entirely and gives the exact decimal representation. This is the most important rule when using the Decimal type.
4. What does ~42 evaluate to in Python? What is the general formula?
Click to reveal answer
~42 evaluates to -43. The general formula is ~n == -(n + 1) for all integers. This follows from two's complement representation: bitwise NOT flips every bit, which in two's complement arithmetic increments the value by 1 and flips the sign. In practical terms: ~0 == -1, ~1 == -2, ~255 == -256. This identity is useful for creating bit mask complements: to clear bits specified by mask M, use flags & ~M.
5. What does Python's round(2.5) return? What about round(3.5)? Why?
Click to reveal answer
round(2.5) returns 2 and round(3.5) returns 4. Python uses "banker's rounding" (round half to even), also known as IEEE 754 rounding mode ROUND_HALF_EVEN. When the value is exactly halfway between two integers, it rounds to the nearest even number. This eliminates the systematic upward bias that "round half up" introduces in large datasets ‚Äî in a uniform distribution of values, about half of the 0.5 cases round up and half round down. School arithmetic uses "round half up" (2.5 ‚Üí 3), which is what most people expect but is statistically biased. Use Decimal.quantize(rounding=ROUND_HALF_UP) if you need school-style rounding.
6. What is the difference between math.sqrt(x) and x ** 0.5? Is there ever a reason to prefer one over the other?
Click to reveal answer
Both math.sqrt(x) and x ** 0.5 compute the square root of x as a float. math.sqrt(x) is implemented using the CPU's native square root instruction, while x ** 0.5 uses general exponentiation which may be slightly slower. Additionally, math.sqrt(x) raises ValueError for negative inputs, while (-1) ** 0.5 also raises ValueError (for float inputs). For complex inputs, use cmath.sqrt(x) which handles negative values correctly (cmath.sqrt(-1) == 1j). There is also math.isqrt(n) for integer square root (floor of the square root) without any floating-point arithmetic, which is exact for large integers.
7. Python's int can represent arbitrarily large numbers. Does this mean there is never any overflow risk in Python?
Click to reveal answer
For pure Python int arithmetic, yes ‚Äî there is no overflow. However, overflow risk returns in several important scenarios. First, when using float: 1e308 * 10 gives inf, and operations on inf may silently propagate through calculations. Second, when interfacing with C extensions (NumPy, ctypes, cffi, database drivers): these use fixed-width C integers and overflow silently according to C rules. numpy.int32(2147483647) + 1 gives -2147483648. Third, when converting a Python int to a C type via ctypes, values larger than the C type supports are truncated. Always be aware of the type boundaries when crossing the Python/C boundary.
8. What is the result of True + True + False + True? What does this reveal about Python's type hierarchy?
Click to reveal answer
The result is 3 ‚Äî an int. This reveals that bool is a proper subclass of int in Python, not merely a type that can convert to int. True has the integer value 1 and False has the integer value 0. They participate directly in arithmetic as integers. The type of the result is int (not bool) because the arithmetic operations return int. This design enables elegant counting patterns: sum(x > 0 for x in data) counts positive values by summing boolean results as integers. You can also use any() and all() for boolean logic across sequences.
9. When would you choose Fraction over Decimal, and vice versa?
Click to reveal answer
Choose Decimal when you need exact decimal arithmetic with a fixed number of significant digits ‚Äî financial calculations, currency, tax, accounting. Decimal uses a base-10 representation, so decimal fractions like 0.1 are stored exactly. It also supports configurable precision and rounding modes. Choose Fraction when you need exact rational arithmetic ‚Äî recipe scaling, unit conversions where you manipulate fractions explicitly (e.g. 3/4 cup √ó 1.5 servings), educational tools, or symbolic computation. Fraction represents numerator/denominator pairs and is always exactly correct but can grow denominators to be very large. Neither is appropriate for high-performance numerical computing ‚Äî use NumPy float64 arrays for that. For anything involving money, always choose Decimal.
