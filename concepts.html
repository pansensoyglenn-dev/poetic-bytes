Variables, Types & Mutation

What a variable actually is in memory, and why mutability matters.

variablesmutabilitytypes
Fundamental+
Analogy

A variable is a label on a sticky note attached to a box. The box holds the value. Two labels can point to the same box — changing the contents affects anyone reading either label.

In Python, a variable is not a container — it is a name that references an object. The assignment x = 42 creates an integer object 42 and attaches the name x to it. Assigning y = x gives y a reference to the same object, not a copy.

# Names and objects
x = [1, 2, 3]
y = x           # y points to the SAME list
y.append(4)
print(x)        # [1, 2, 3, 4] — x changed too!

# Immutable objects: int, float, str, tuple, frozenset
a = "hello"
b = a
b = b.upper()   # creates a NEW string; a is unaffected
print(a)        # "hello" — unchanged

# Check if two names point to the same object
print(x is y)   # True  — same object
print(a is b)   # False — different objects
💡 Mutable objects (list, dict, set, custom classes) can be changed in place. Immutable objects (int, str, tuple) cannot — operations always return a new object. This distinction is crucial for understanding function arguments, caching, and thread safety.

Scope & Namespaces (LEGB)

How Python finds variable names — Local, Enclosing, Global, Built-in.

scopeLEGBnamespaceclosure
Fundamental+
Analogy

Looking for your keys: first check your pocket (Local), then your bag (Enclosing), then the kitchen counter (Global), then the lost-and-found (Built-in). You stop at the first place you find them.

x = "global"

def outer():
    x = "enclosing"

    def inner():
        x = "local"
        print(x)   # "local"   — L
    inner()
    print(x)       # "enclosing" — E

outer()
print(x)           # "global"  — G
print(len)         # built-in  — B

# Closure captures enclosing scope
def make_adder(n):
    def add(x):
        return x + n  # n is captured from enclosing scope
    return add

add5 = make_adder(5)
print(add5(10))  # 15 — n=5 is remembered even after make_adder returned
💡 Python looks up names at runtime (late binding). This causes the famous "closure in a loop" bug where all closures capture the same loop variable reference, not its value at creation time.

First-Class Functions

Functions are objects — they can be passed, returned, stored, and assigned like any other value.

first-classhigher-ordercallable
Fundamental+
Analogy

A function being first-class is like a hammer that can also be placed in a toolbox, handed to someone else, or returned after use — it's not fixed to one wall. Most languages nail functions to the wall.

# Functions can be assigned to variables
greet = print
greet("Hello!")

# Functions can be passed as arguments
def apply(func, value):
    return func(value)

print(apply(str.upper, "python"))  # PYTHON
print(apply(abs, -42))              # 42

# Functions can be stored in data structures
ops = {"double": lambda x: x*2, "square": lambda x: x**2}
print(ops["square"](7))  # 49

# Functions can be returned from other functions
def multiplier(n):
    return lambda x: x * n

triple = multiplier(3)
print(list(map(triple, [1,2,3,4])))  # [3, 6, 9, 12]
💡 This is what enables Python's decorators, callbacks, higher-order functions like map()/filter(), and the entire functional programming style. Every decorator is just a function that takes and returns a function.

Null, None & Sentinel Values

Representing "nothing" safely — and why null is called the billion-dollar mistake.

Nonenullsentineloptional
Fundamental+
Analogy

None is like an empty envelope — it exists as an object, but contains nothing. Forgetting to check if an envelope is empty before reading it causes the dreaded NullPointerException in other languages.

Tony Hoare, who invented null references in 1965, called it his "billion-dollar mistake" because of the crashes it caused. Python's None is safer — it's a real object you can test against. The Pythonic habit: use is None / is not None, never == None.

# None is a singleton — always use 'is', never '=='
result = None
if result is None:
    print("No result yet")

# Sentinel pattern — a unique object marking "missing"
_MISSING = object()

def get_config(key, default=_MISSING):
    config = {"host": "localhost"}
    if key in config:
        return config[key]
    if default is _MISSING:
        raise KeyError(f"Missing key: {key}")
    return default

print(get_config("host"))           # localhost
print(get_config("port", default=8080))  # 8080
# get_config("missing")  # raises KeyError
⚠️ Never use a mutable object (like [] or {}) as a default argument — it's created once and shared across calls. Use None as the sentinel and create the mutable object inside the function body.

Big O Notation

Measuring how an algorithm scales as input grows — the universal language of algorithm efficiency.

Big Otime complexityspace complexity
Algorithms+
Analogy

Big O is like describing a road trip by "it takes about 2 hours regardless of how many passengers" (O(1)) versus "it takes 1 hour per person because you pick everyone up" (O(n)) versus "every person has to greet every other person first" (O(n²)).

Big O describes the upper bound on growth rate — ignoring constants and lower-order terms. O(2n) and O(n) are both O(n). What matters is how the function grows as n approaches infinity.

Complexity ladder (from fastest to slowest): O(1) Constant — Hash table lookup, array access by index O(log n) Logarithmic — Binary search, balanced BST operations O(n) Linear — Scanning an array, linear search O(n log n) Linearithmic — Merge sort, FFT, TimSort O(n²) Quadratic — Bubble sort, nested loops over same array O(2ⁿ) Exponential — Naive fibonacci, subset enumeration O(n!) Factorial — Brute-force permutation search
import time

def o1(lst):    return lst[0]               # O(1)
def on(lst):    return sum(lst)              # O(n)
def on2(lst):                                # O(n²)
    count = 0
    for i in lst:
        for j in lst: count += 1
    return count

for n in [100, 1000, 5000]:
    lst = list(range(n))
    t1 = time.perf_counter(); o1(lst); ms1=(time.perf_counter()-t1)*1000
    t2 = time.perf_counter(); on(lst); ms2=(time.perf_counter()-t2)*1000
    t3 = time.perf_counter(); on2(lst); ms3=(time.perf_counter()-t3)*1000
    print(f"n={n:>5}: O(1)={ms1:.3f}ms O(n)={ms2:.3f}ms O(n²)={ms3:.2f}ms")
💡 The difference between O(n log n) and O(n²) for sorting 1 million items: about 20 million vs 1 trillion operations. At 10⁹ ops/sec: 0.02 seconds vs 1,000 seconds. Algorithm choice dwarfs hardware.

Space Complexity & Memory Trade-offs

Every algorithm trades time for space — understanding when each matters.

space complexitymemorytrade-off
Algorithms+
Analogy

A memoization table is like keeping a notebook of solved maths problems. You use more paper (space), but never solve the same problem twice (time). The classic time-space trade-off.

import sys
from functools import lru_cache

# Space O(1) — generator uses constant memory regardless of n
def count_gen(n):
    i = 0
    while i < n:
        yield i; i += 1

# Space O(n) — list allocates all n integers at once
def count_list(n):
    return list(range(n))

n = 1_000_000
gen  = count_gen(n)
lst  = count_list(n)
print(f"Generator size: {sys.getsizeof(gen):>10,} bytes")
print(f"List size:      {sys.getsizeof(lst):>10,} bytes")

# Memoization: trade space for time
@lru_cache(maxsize=None)
def fib_memo(n):  # O(n) space, O(n) time
    return n if n < 2 else fib_memo(n-1) + fib_memo(n-2)

def fib_dp(n):     # O(1) space, O(n) time
    a, b = 0, 1
    for _ in range(n): a, b = b, a+b
    return a
✓ Generators (lazy evaluation) are the Python idiom for O(1) space iteration over large sequences. range(), file iteration, zip(), and map() all return lazy iterators — they never build the full sequence in memory.

Stack vs Heap Memory

Where different kinds of data live — and why it matters for performance and bugs.

stackheapcall stackallocation
Memory+
Analogy

The stack is a notepad you add to from the top and always take from the top (LIFO). The heap is a warehouse where you can store anything anywhere, but you need to remember the address — and clean up when done.

Call Stack Heap ┌──────────────────┐ ┌────────────────────────┐ │ main() │ │ list [1,2,3,4] ◄──┐ │ │ x = ref ───────┼──────────┼──────────────────┘ │ │ ├──────────────────┤ │ dict {"a":1} ◄──┐│ │ │ greet(name) │ │ str "hello" ◄─┐ ││ │ │ msg = ref ─────┼──────────┼──────────────────┘ ││ │ ├──────────────────┤ │ ... ││ │ │ (grows downward) │ └────────────────────┘│ │ └──────────────────┘ │ │ Stack: fast, automatic, fixed-size, LIFO, stores references & primitives Heap: slower, manual/GC, dynamic-size, stores all Python objects

In Python, all objects live on the heap. The call stack holds local variable names (which are references to heap objects). A stack overflow occurs when recursion is too deep — Python's default limit is 1,000 frames.

import sys

# Check Python's recursion limit
print("Recursion limit:", sys.getrecursionlimit())  # 1000

# Each function call adds a frame to the call stack
def depth(n):
    if n == 0: return 0
    return depth(n - 1) + 1

print("Stack depth test:", depth(500))  # ok

# id() returns the object's memory address on the heap
x = [1, 2, 3]
print(f"List lives at heap address: {id(x):#x}")
💡 Python stores small integers (−5 to 256) and short strings in a cache — they're reused rather than newly allocated. That's why a = 256; b = 256; a is b is True but a = 257; b = 257; a is b is False.

Garbage Collection & Reference Counting

How Python automatically reclaims memory — and what can still go wrong.

garbage collectionreference countingmemory leakgc
Memory+
Analogy

Reference counting is like a shared umbrella with a ticket system. Each person who borrows it takes a ticket (increments count). When they return it the count drops. When the count reaches zero, nobody needs it — it's safely put away (freed).

import sys, gc

x = [1, 2, 3]
print("Ref count:", sys.getrefcount(x))  # 2 (x + getrefcount arg)

y = x
print("After y=x:", sys.getrefcount(x))  # 3

del y
print("After del y:", sys.getrefcount(x)) # 2

# Circular reference — refcount never hits 0!
class Node:
    def __init__(self): self.ref = None

a, b = Node(), Node()
a.ref = b  # a → b
b.ref = a  # b → a  (cycle!)
del a, b   # refcounts drop to 1 — NOT freed by refcount alone

collected = gc.collect()  # cyclic GC detects and clears it
print(f"GC collected {collected} objects")
⚠️ Python can "leak" memory if you hold global references to large objects, accumulate callbacks with captured references, or use C extensions that don't properly decrement reference counts. Use gc.collect() and tracemalloc to diagnose memory growth.

CPU Cache & Memory Locality

Why sequential access to memory is often 100× faster than random access.

CPU cachelocalityL1/L2/L3performance
Memory+
Analogy

The CPU cache is your desk — fast to access but tiny. Main memory (RAM) is a filing cabinet across the room. A cache miss is like having to walk to the filing cabinet for every document instead of having it on your desk.

Memory Hierarchy (fastest → slowest, smallest → largest): CPU Registers ~0.3 ns ~1 KB — Variables in current instruction L1 Cache ~1 ns ~32 KB — Hottest code and data L2 Cache ~3 ns ~256 KB — Recent accesses L3 Cache ~10 ns ~8 MB — Shared across cores RAM ~60 ns ~16 GB — All running program data Disk (SSD) ~100 µs ~1 TB — Persistent storage Disk (HDD) ~5 ms ~4 TB — Slow persistent storage

Python's list stores object references contiguously in memory, but the objects themselves are scattered on the heap. NumPy arrays store actual values contiguously — this cache-friendly layout is why NumPy operations are 10–100× faster than Python loops for numerical work.

💡 Accessing data in order (row-major for 2D arrays) is cache-friendly. Random access causes cache misses. This is why iterating a list is faster than accessing a dict at the same scale — list elements are predictably positioned.

Hash Tables & Hashing

How Python's dict achieves O(1) lookup — and what can break it.

hash tabledictO(1)collision
Data Structures+
Analogy

A hash table is a library with a magic indexing system. You give the librarian a book title (key), they run a formula (hash function) and tell you instantly which shelf (bucket) it's on — no searching required.

# Python's dict is a hash table — O(1) average lookup
d = {"alice": 90, "bob": 85, "carol": 92}

# Under the hood: hash("alice") → bucket index → value
print(hash("alice"))   # some integer (e.g. 3713082716806266542)
print(hash(42))        # 42 (ints hash to themselves for small values)
print(hash((1,2,3)))   # tuples are hashable (immutable)

# Lists are NOT hashable — can't be dict keys
try:
    d[[1,2]] = "test"
except TypeError as e:
    print(f"Error: {e}")

# Custom hashable class — must implement __hash__ AND __eq__
class Point:
    def __init__(self, x, y): self.x, self.y = x, y
    def __hash__(self): return hash((self.x, self.y))
    def __eq__(self, o): return (self.x, self.y) == (o.x, o.y)

seen = {Point(0, 0), Point(1, 2)}
print(Point(0, 0) in seen)  # True — O(1) lookup
💡 A hash collision is when two different keys produce the same bucket index. Python handles this with open addressing (probing). In the worst case (all keys collide), dict lookup degrades to O(n) — but Python's hash functions make this essentially impossible in practice.

Stack, Queue & Priority Queue

LIFO vs FIFO — and when heap-based priority ordering matters.

stackqueuedequeheapLIFOFIFO
Data Structures+
Analogy

Stack: a stack of plates — you only add/remove from the top (LIFO). Queue: a checkout line — first in, first out (FIFO). Priority queue: an emergency room — the most critical patient goes next, regardless of arrival order.

from collections import deque
import heapq

# Stack — use list, O(1) push/pop from right end
stack = []
stack.append("a"); stack.append("b"); stack.append("c")
print(stack.pop())  # "c" — LIFO

# Queue — use deque, O(1) append right + popleft
queue = deque()
queue.append("first"); queue.append("second"); queue.append("third")
print(queue.popleft())  # "first" — FIFO

# Priority Queue — heapq is a min-heap
tasks = []
heapq.heappush(tasks, (3, "low priority"))
heapq.heappush(tasks, (1, "critical"))
heapq.heappush(tasks, (2, "medium"))
while tasks:
    priority, task = heapq.heappop(tasks)
    print(f"  [{priority}] {task}")
💡 Never use list.insert(0, x) or list.pop(0) for a queue — both are O(n) because they shift all elements. deque.appendleft()/deque.popleft() are O(1) — that's the right tool.

Trees & Graphs

Hierarchical and networked structures — and how to traverse them.

treeBSTgraphBFSDFS
Data Structures+
Analogy

A tree is your family tree — one root, branching down with no cycles. A graph is a road map — any city can connect to any other, and you can go in circles. BFS explores all nearby nodes first (ripple in a pond). DFS dives deep down one path first (exploring a maze).

from collections import deque

class Node:
    def __init__(self, val):
        self.val = val; self.left = self.right = None

def bfs(root):  # Breadth-First Search — level by level
    if not root: return []
    queue, result = deque([root]), []
    while queue:
        node = queue.popleft()
        result.append(node.val)
        if node.left:  queue.append(node.left)
        if node.right: queue.append(node.right)
    return result

def dfs(root):  # Depth-First (in-order) — left, root, right
    if not root: return []
    return dfs(root.left) + [root.val] + dfs(root.right)

root = Node(4)
root.left = Node(2); root.right = Node(6)
root.left.left = Node(1); root.left.right = Node(3)
root.right.left = Node(5); root.right.right = Node(7)

print("BFS (level order):", bfs(root))   # [4,2,6,1,3,5,7]
print("DFS (in-order):   ", dfs(root))   # [1,2,3,4,5,6,7] sorted!
💡 BFS uses a queue and finds shortest paths in unweighted graphs. DFS uses a stack (or recursion) and is better for exploring all paths, detecting cycles, and topological sorting. Both are O(V + E) — vertices plus edges.

Recursion

A function calling itself — the elegance, the danger, and when to use iteration instead.

recursionbase casecall stackstack overflow
Algorithms+
Analogy

Standing between two facing mirrors — each reflection contains another mirror, which contains another. The base case is when you stop looking (the reflections get too small to see). Without a base case, you'd look forever.

# Every recursive function needs: base case + smaller subproblem
def factorial(n: int) -> int:
    if n <= 1: return 1      # base case
    return n * factorial(n - 1)  # recursive case

print(factorial(6))  # 720

# Same logic, iteratively (avoids stack depth limit)
def factorial_iter(n: int) -> int:
    result = 1
    for i in range(2, n + 1):
        result *= i
    return result

# Recursion shines for naturally recursive structures
def flatten(items):
    for item in items:
        if isinstance(item, list):
            yield from flatten(item)  # recurse on sublists
        else:
            yield item

nested = [1, [2, [3, 4], 5], [6, [7]]]
print(list(flatten(nested)))  # [1,2,3,4,5,6,7]
⚠️ Python doesn't have tail-call optimisation (TCO). A recursive function called 1,001 times deep hits the 1,000-frame limit and raises RecursionError. For deep recursion, convert to iteration with an explicit stack.

Dynamic Programming

Solving complex problems by breaking them into overlapping subproblems — and never solving the same one twice.

dynamic programmingmemoizationsubproblemsDP
Algorithms+
Analogy

You're driving across the country and need the cheapest route. Instead of recalculating the cost of reaching Chicago every time it appears in different route combinations, you write it down once. The notebook is your DP table.

DP requires two properties: overlapping subproblems (the same smaller problems appear repeatedly) and optimal substructure (the optimal solution is built from optimal solutions of subproblems).

from functools import lru_cache

# Coin change — minimum coins to make amount
def min_coins_dp(coins: list[int], amount: int) -> int:
    dp = [float('inf')] * (amount + 1)
    dp[0] = 0  # base: 0 coins needed to make 0

    for a in range(1, amount + 1):
        for coin in coins:
            if coin <= a:
                dp[a] = min(dp[a], dp[a - coin] + 1)

    return dp[amount] if dp[amount] != float('inf') else -1

coins = [1, 5, 10, 25]
print(f"Min coins for $0.41: {min_coins_dp(coins, 41)}")  # 4
print(f"Min coins for $0.99: {min_coins_dp(coins, 99)}")  # 9

# Longest Common Subsequence
@lru_cache(maxsize=None)
def lcs(a: str, b: str) -> int:
    if not a or not b: return 0
    if a[-1] == b[-1]: return 1 + lcs(a[:-1], b[:-1])
    return max(lcs(a[:-1], b), lcs(a, b[:-1]))

print(f"LCS('ABCBDAB','BDCABA'): {lcs('ABCBDAB','BDCABA')}")  # 4
💡 @lru_cache converts any recursive function into a memoised (top-down DP) version automatically. It caches results keyed by arguments. For bottom-up DP, build a table iteratively from smallest to largest subproblem.

Concurrency vs Parallelism

Doing multiple things — the critical difference between interleaving and truly simultaneous execution.

concurrencyparallelismthreadGIL
Concurrency+
Analogy

Concurrency: one barista taking orders from multiple customers, switching between them (one person, interleaved). Parallelism: four baristas each serving a different customer simultaneously (four people, truly simultaneous). Concurrency is about structure; parallelism is about execution.

Concurrency (one core, interleaved) Parallelism (multiple cores) Core 1: Task A ─── Task B ─── Task A ─── Task B Core 1: Task A ────────────────── Core 2: Task B ────────────────── Python Threading: Concurrency, NOT parallelism for CPU-bound work (GIL) Python asyncio: Concurrency via cooperative multitasking (single thread) Python multiprocessing: True parallelism (separate processes, no GIL)
from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor
import time

def io_task(n):       # I/O-bound: waiting dominates
    time.sleep(0.1); return n * 2

def cpu_task(n):      # CPU-bound: computation dominates
    return sum(i**2 for i in range(n))

# I/O-bound → use threads (GIL released during sleep/IO)
t = time.perf_counter()
with ThreadPoolExecutor(10) as p:
    results = list(p.map(io_task, range(10)))
print(f"Threads (I/O): {(time.perf_counter()-t):.2f}s")  # ~0.1s

# CPU-bound → use processes (bypasses GIL)
t = time.perf_counter()
with ProcessPoolExecutor() as p:
    results = list(p.map(cpu_task, [100_000]*4))
print(f"Processes (CPU): {(time.perf_counter()-t):.2f}s")
💡 Rule of thumb: I/O-bound tasks (network calls, file reads, DB queries) → asyncio or ThreadPoolExecutor. CPU-bound tasks (number crunching, image processing) → ProcessPoolExecutor or NumPy (which releases the GIL). Threads for CPU-bound work gain nothing in CPython.

Race Conditions & Locks

What goes wrong when threads share state — and how locks prevent it.

race conditionmutexlockdeadlockthread-safe
Concurrency+
Analogy

A race condition is two people editing the same Google Doc at the exact same moment — one's changes might overwrite the other's. A lock (mutex) is like a "currently editing" flag — one person holds it, everyone else waits.

import threading

# Race condition — two threads increment the same counter
counter = 0

def unsafe_increment():
    global counter
    for _ in range(100_000):
        counter += 1  # NOT atomic: read → add → write

threads = [threading.Thread(target=unsafe_increment) for _ in range(2)]
for t in threads: t.start()
for t in threads: t.join()
print(f"Unsafe: {counter} (expected 200000)")  # likely less!

# Fix: use a Lock to make the operation atomic
counter = 0
lock = threading.Lock()

def safe_increment():
    global counter
    for _ in range(100_000):
        with lock:      # acquire → increment → release
            counter += 1

threads = [threading.Thread(target=safe_increment) for _ in range(2)]
for t in threads: t.start()
for t in threads: t.join()
print(f"Safe:   {counter} (expected 200000)")  # exactly 200000
⚠️ Deadlock occurs when Thread A holds Lock 1 waiting for Lock 2, while Thread B holds Lock 2 waiting for Lock 1 — both wait forever. Prevention: always acquire locks in the same order, or use threading.RLock for re-entrant locking.

Async / Await & Event Loops

Single-threaded concurrency — how one thread handles thousands of I/O operations without blocking.

asyncawaitevent loopcoroutineasyncio
Concurrency+
Analogy

A single waiter at a restaurant: they take an order, submit it to the kitchen (await), and while it cooks they take another table's order. They're not cooking multiple dishes simultaneously — they're efficiently switching tasks during waiting periods.

import asyncio, time async def fetch(name: str, delay: float) -> str: print(f"Starting {name}…") await asyncio.sleep(delay) # yields control; doesn't block print(f"Done {name}") return f"{name} result" async def main(): start = time.perf_counter() # Sequential: 0.1 + 0.2 + 0.3 = 0.6s # r1 = await fetch("A", 0.1) # r2 = await fetch("B", 0.2) # Concurrent: max(0.1, 0.2, 0.3) = 0.3s results = await asyncio.gather( fetch("A", 0.1), fetch("B", 0.2), fetch("C", 0.3), ) elapsed = time.perf_counter() - start print(f"All done in {elapsed:.2f}s") # ~0.3s print(results) asyncio.run(main())
💡 The event loop runs one coroutine at a time. When a coroutine hits await, it pauses and the loop runs another ready coroutine. No threads, no locks needed — cooperative multitasking. This is how Node.js works, and why it can handle thousands of concurrent connections on a single thread.

Object-Oriented Programming (OOP)

Organising code around objects that bundle data and behaviour — the four pillars.

OOPclassencapsulationinheritancepolymorphism
Paradigm+
The Four Pillars of OOP: Encapsulation Bundle data + methods; hide internals with _private/__mangled Abstraction Expose only what's needed; hide complexity behind interfaces (ABC) Inheritance Reuse and extend behaviour; subclass inherits from superclass Polymorphism Same interface, different behaviour — duck typing + dunder methods
from abc import ABC, abstractmethod class Shape(ABC): # Abstraction @abstractmethod def area(self) -> float: ... class Circle(Shape): # Inheritance def __init__(self, r): self._r = r # Encapsulation def area(self): return 3.14159 * self._r**2 class Rectangle(Shape): def __init__(self, w, h): self._w, self._h = w, h def area(self): return self._w * self._h shapes = [Circle(5), Rectangle(4, 6), Circle(3)] for s in shapes: print(f"{type(s).__name__}: area = {s.area():.2f}") # Polymorphism — same .area() call, different behaviour per type total = sum(s.area() for s in shapes) print(f"Total area: {total:.2f}")

Functional Programming

Functions as the primary building block — pure, side-effect-free, composable.

functionalpure functionimmutabilitymap/filter/reduce
Paradigm+
Analogy

A pure function is like a vending machine: same money in always produces the same snack out — no side effects, no surprises. An impure function is like asking a friend for advice — the outcome depends on their mood (hidden state).

from functools import reduce # Pure function — no side effects, same input = same output def double(x): return x * 2 def is_even(x): return x % 2 == 0 def add(a, b): return a + b numbers = [1, 2, 3, 4, 5, 6] # Functional pipeline — no mutation of original data result = reduce(add, filter(is_even, map(double, numbers))) print(result) # 24 — (4+8+12) — doubled evens summed # Equivalent with comprehensions (more Pythonic) result2 = sum(x*2 for x in numbers if x % 2 == 0) # Function composition def compose(*fns): def composed(x): for fn in reversed(fns): x = fn(x) return x return composed process = compose(str.upper, str.strip, str.lower) print(process(" Hello World ")) # "HELLO WORLD"
💡 Python is multi-paradigm — you can use OOP, functional, and procedural styles freely. Pure functions are easiest to test (no setup/teardown needed), cache safely with @lru_cache, and run in parallel (no shared state to protect).

SOLID Principles

Five design principles that make software easier to understand, maintain, and extend.

SOLIDSRPOCPDIPdesign principles
Paradigm+
S — Single Responsibility: a class should have ONE reason to change O — Open/Closed: open for extension, closed for modification L — Liskov Substitution: subclasses must be usable wherever superclass is expected I — Interface Segregation: many specific interfaces beat one general interface D — Dependency Inversion: depend on abstractions, not concrete implementations
from abc import ABC, abstractmethod # D — Dependency Inversion: depend on the abstraction, not the class class Notifier(ABC): # abstraction @abstractmethod def send(self, msg: str): ... class EmailNotifier(Notifier): # concrete def send(self, msg): print(f"Email: {msg}") class SMSNotifier(Notifier): def send(self, msg): print(f"SMS: {msg}") # OrderProcessor depends on the abstraction, not EmailNotifier class OrderProcessor: def __init__(self, notifier: Notifier): self._notifier = notifier # injected dependency def process(self, order): # ... process order ... self._notifier.send(f"Order {order} confirmed") # Swap notifier without changing OrderProcessor OrderProcessor(EmailNotifier()).process("#1001") OrderProcessor(SMSNotifier()).process("#1002")

HTTP & REST APIs

How the web communicates — methods, status codes, and RESTful conventions.

HTTPRESTAPIstatus codesJSON
Networking+
Analogy

HTTP is like sending letters: GET asks "send me this document", POST says "here's a new document to file", PUT says "replace this document entirely", PATCH says "change just this part", DELETE says "destroy this document". The status code is the reply stamp: 200 "got it", 404 "no such document", 500 "we made an error".

HTTP Methods and their semantics: GET /users → list all users (read, safe, idempotent) GET /users/42 → get user 42 (read, safe, idempotent) POST /users → create a new user (write, not idempotent) PUT /users/42 → replace user 42 entirely (write, idempotent) PATCH /users/42 → update specific fields (write, may not be idempotent) DELETE /users/42 → delete user 42 (write, idempotent) Common Status Codes: 200 OK 201 Created 204 No Content 301 Moved Permanently 302 Found (redirect) 400 Bad Request 401 Unauthorized 403 Forbidden 404 Not Found 422 Unprocessable 500 Internal Server Error 502 Bad Gateway 503 Service Unavailable

💡 Idempotent means calling the same operation multiple times has the same effect as calling it once. GET, PUT, and DELETE are idempotent. POST is not — calling it twice creates two resources. This matters for retry logic and distributed systems.

Databases & ACID Properties

What makes a database trustworthy — Atomicity, Consistency, Isolation, Durability.

databaseACIDtransactionSQLNoSQL
Networking+
Analogy

Transferring money between bank accounts: you debit one and credit the other. ACID guarantees that either both happen (Atomicity), the balance stays consistent (Consistency), other transactions don't see the intermediate state (Isolation), and a power outage can't corrupt the result (Durability).

ACID Properties: Atomic All operations in a transaction succeed together, or all fail together Consistent The database moves from one valid state to another; constraints always hold Isolated Concurrent transactions behave as if they ran sequentially Durable Committed transactions survive crashes, power failures, and bugs SQL vs NoSQL trade-offs: SQL (PostgreSQL, MySQL) NoSQL (MongoDB, Redis, DynamoDB) Schema Enforced, rigid Flexible, schema-less ACID Full ACID support Often BASE (eventual consistency) Joins Native, efficient Manual, expensive Scale Vertical scaling Horizontal scaling Query SQL (declarative, powerful) Document/key-value queries
