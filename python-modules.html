💻 Running Code Locally – Step-by-Step

Many examples in this lesson involve creating multiple files and importing between them. The in‑browser runner works for simple snippets, but for full module and package demonstrations, you need to run the code on your own machine.

How to run these examples on your computer

  1. Install Python 3.10+ – Download from python.org and verify with python --version in your terminal.
  2. Create a project folder – e.g., mkdir myproject && cd myproject.
  3. Create a virtual environment – python -m venv .venv and activate it (see the venv section below).
  4. Create the files – Copy each code snippet into a .py file (e.g., math_utils.py, greetings.py, etc.) inside your project folder, respecting the package structure shown (folders with __init__.py).
  5. Run your main script – From the terminal, execute python main.py (or whichever file contains your entry point).
  6. Install external packages – If you need packages like requests or pandas, install them inside your virtual environment with pip install requests.

💡 The browser‑based “Run” button works for single‑file code, but imports that reference other files or package structures will likely fail because the runner does not have access to your file system. Always use your local Python environment for module and package exercises.

📦 What Are Modules & Packages?

As your Python programs grow beyond a single file, you need a way to organise code into logical, reusable units. Modules and packages are Python's answer.

🧩 Module = One .py File

A module is simply a file containing Python definitions and statements. The file name is the module name with the .py suffix. Modules allow you to group related functions, classes, and constants together.

# math_utils.py — a module
def add(a, b): return a + b
def subtract(a, b): return a - b
PI = 3.14159
📁 Save this as math_utils.py in your project folder.

Any Python file you create can be imported as a module. The standard library itself is a collection of modules (os, sys, json, etc.).

📁 Package = Folder with __init__.py

A package is a directory that contains a special __init__.py file (which can be empty). Packages allow you to organise related modules into a hierarchy.

# myproject/
# ├── __init__.py        # marks this as a package
# ├── main.py
# ├── utils/
# │   ├── __init__.py
# │   ├── string_helpers.py
# │   └── math_helpers.py
# └── models/
#     ├── __init__.py
#     └── user.py
📁 Create these folders and files in your project directory.

Packages let you structure large codebases cleanly. They also enable relative imports and control the public API via __all__.

Why This Matters

Without modules and packages, every Python project would be a single, unmaintainable file. With them, you can:

  • ✓ Reuse code across multiple files and projects
  • ✓ Keep your code organised and easy to navigate
  • ✓ Share your work by publishing packages to PyPI
  • ✓ Isolate dependencies with virtual environments

↗️ Import Statements — The Gateway to Modules

Every Python file that wants to use another module must import it. The import system is flexible — import whole modules, specific names, or even rename them.

📥 Basic Import Forms

# 1. Import the whole module
import math
print(math.sqrt(16))

# 2. Import with an alias
import math as m
print(m.pi)

# 3. Import specific names from a module
from math import sqrt, pi
print(sqrt(25))
print(pi)

# 4. Import everything (generally discouraged)
from math import *
print(sin(0))   # messy, pollutes namespace

# 5. Import a module with a shorter alias
import pandas as pd
import numpy as np
🔁 These imports work directly in any Python file. No extra files needed.
💡 The import * form is strongly discouraged in most codebases because it pollutes the global namespace and makes it unclear which names come from which module. Be explicit — import only what you need.

🔍 How Python Finds Modules

When you write import something, Python searches for something in this order:

  • 1. The directory containing the script you're running
  • 2. PYTHONPATH environment variable directories
  • 3. The standard library directories
  • 4. Site-packages (where pip installs third-party packages)
import sys
print(sys.path)   # List of directories Python searches
👀 Run this to see where Python looks for modules on your system.
✨ You can add custom paths to sys.path at runtime, but it's cleaner to structure your project properly and install your own packages in development mode.

🔄 Relative Imports

Inside a package, you can import other modules from the same package using relative imports with dots.

# In myproject/utils/string_helpers.py
from . import math_helpers       # import from same package
from ..models import user         # import from sibling package
from .math_helpers import clamp  # import specific name

# Relative imports only work inside packages.
# They are resolved relative to the current module's package.
📁 This file must be inside a package (folder with __init__.py). The browser runner will not execute this correctly – run locally.
⚠️ Relative imports only work inside a package. They cannot be used in a script that is executed directly (__name__ == "__main__"). For scripts, use absolute imports.

📄 Creating Your Own Modules

Any .py file is a module. Here's how to write, import, and use your own modules effectively.

📝 A Simple Module

Create a file called greetings.py with the following content:

# greetings.py
"""A simple module for greeting people."""

def hello(name: str) -> str:
    """Return a friendly greeting."""
    return f"Hello, {name}!"

def goodbye(name: str) -> str:
    return f"Goodbye, {name}!"

# This code runs only when the module is executed directly
if __name__ == "__main__":
    print(hello("World"))
📁 Save as greetings.py and run it with python greetings.py to test.

Now import and use it in another file (e.g., main.py):

# main.py
import greetings
print(greetings.hello("Alice"))
print(greetings.goodbye("Bob"))
📁 Keep main.py in the same folder as greetings.py and run python main.py.
💡 The if __name__ == "__main__" guard allows a module to be both imported and run as a script. Code inside this block executes only when you run the file directly (python greetings.py), not when it's imported.

📂 Module Search Path

If your module is not in the same directory as your script, you have three options:

  • 1. Place it in the same directory as the script
  • 2. Add its directory to PYTHONPATH
  • 3. Install it as a package using pip install -e . (editable install)

📁 Packages — Organising Modules into Hierarchies

Packages are how you structure large projects. A package is a directory with an __init__.py file.

📐 Package Structure

Here's a typical package structure for a project called analytics:

# analytics/
# ├── __init__.py
# ├── core/
# │   ├── __init__.py
# │   ├── stats.py
# │   └── utils.py
# ├── visualization/
# │   ├── __init__.py
# │   ├── charts.py
# │   └── tables.py
# └── data/
#     ├── __init__.py
#     └── loaders.py
📁 Create these folders and files locally. Each folder must contain an __init__.py (can be empty).

To import from this package:

from analytics.core.stats import mean, median
from analytics.visualization.charts import bar_chart
from analytics.data.loaders import load_csv
📁 Run this from a script outside the analytics folder, or use pip install -e . to install the package locally.

📄 The __init__.py File

The __init__.py file marks a directory as a Python package. It can be empty, but it can also contain code that runs when the package is imported — for example, to expose a clean public API.

# analytics/__init__.py
"""
Analytics package — data analysis and visualisation tools.
"""

from .core.stats import mean, median, standard_deviation
from .visualization.charts import bar_chart, line_chart
from .data.loaders import load_csv, load_json

__all__ = ["mean", "median", "standard_deviation",
           "bar_chart", "line_chart", "load_csv", "load_json"]
📁 This file goes inside the analytics/ folder.

Now users can do:

import analytics
print(analytics.mean([1,2,3,4,5]))   # clean and explicit
📁 Run this from a script after installing the package or placing it in the same directory.
💡 The __init__.py file is executed when the package is imported. This is where you can import submodules to expose them at the package level, making the API cleaner for users.

🎯 __all__ — Controlling What Gets Imported

The __all__ variable in a module or package defines which names are exported when someone uses from module import *.

# mymodule.py
def public_func(): pass
def _private_func(): pass    # convention: underscore means private

__all__ = ["public_func"]   # only public_func is exported

# In another file
from mymodule import *   # only public_func is imported
📁 Save as mymodule.py and test the import in a separate script.
💡 Even without __all__, names starting with an underscore are not imported by import *. But __all__ gives you explicit control and documents the public API.

In a package, __all__ in __init__.py controls what is available when someone imports the package directly.

🧪 Virtual Environments & pip — Dependency Management

Every Python project should have its own isolated environment. Virtual environments prevent dependency conflicts and keep your global Python clean.

📦 Creating a Virtual Environment

# Create a virtual environment (Python 3.3+)
python -m venv .venv

# Activate it
# On macOS/Linux:
source .venv/bin/activate

# On Windows:
.venv\Scripts\activate

# Once activated, your prompt changes to show (.venv)
# Now install packages — they go into this environment only
pip install requests pandas flask
💻 Run these commands in your terminal inside your project folder.
💡 Always create a virtual environment for every Python project. It's the single most important habit for maintaining reproducible, conflict-free projects.

📋 requirements.txt — Locking Dependencies

Once you have installed packages, freeze the exact versions so anyone else (or your future self) can reproduce the environment.

# Generate a requirements file
pip freeze > requirements.txt

# Install from a requirements file
pip install -r requirements.txt
📁 This creates a requirements.txt file in your project root.

requirements.txt looks like this:

requests==2.31.0
pandas==2.1.4
flask==3.0.0
✨ For larger projects, consider using poetry or uv for more robust dependency management, including development vs. production dependencies.

📥 Installing Your Own Package in Development Mode

If you're building a package, you can install it in "editable" mode so changes to the source are immediately reflected without reinstalling.

# In your project root (with setup.py or pyproject.toml)
pip install -e .

# Now you can import your package from anywhere
import mypackage
📁 Requires a setup.py or pyproject.toml in your project root.

✅ Module & Package Best Practices

Write maintainable, shareable Python code with these proven conventions.

📁 Use a Clear Project Structure

Place your modules in a src/ directory or use a flat structure. Keep tests in a tests/ directory. Name your package meaningfully.

src/tests/README.md

📝 Document Every Module

Every module, class, and function should have a docstring. Use __all__ to document the public API.

docstrings__all__Sphinx

🧪 Use Absolute Imports

In most cases, absolute imports are clearer and more maintainable than relative imports. Use relative imports only within a package.

absolute importsexplicit

🔒 Always Use a Virtual Environment

Never install packages globally. Each project gets its own .venv and requirements.txt or pyproject.toml.

venvrequirements.txtpip

📦 Name Your Package Clearly

Choose a unique, memorable name for your package. If publishing to PyPI, check availability first. Use lowercase with underscores.

namingPyPI

🔄 Use import at the Top

Put all imports at the top of the file, grouped by standard library, third-party, and local imports. This makes dependencies easy to see.

PEP 8import order

Frequently Asked Questions

Common questions about Python modules and packages

What is the difference between a module and a package?

+

A module is a single .py file. A package is a directory containing multiple modules and a special __init__.py file. Packages are used to organise related modules into a hierarchy.

Why do we need __init__.py?

+

__init__.py marks a directory as a Python package. It can be empty, but it can also contain code that runs when the package is imported — for example, to expose a clean public API or configure logging. Without it, Python treats the directory as a regular folder, not a package.

What is the difference between import module and from module import name?

+

import module imports the entire module and you access its contents with module.name. from module import name imports only the specified name directly into your namespace, so you can use name without the module prefix. The latter is cleaner when you use a specific name many times, but can lead to name collisions.

How do I import a module from a parent directory?

+

Use relative imports: from .. import parent_module. This only works inside a package. For scripts, add the parent directory to sys.path or better, restructure your project as an installable package and use absolute imports.

What is if __name__ == "__main__" used for?

+

This guard prevents code from running when the module is imported. Code inside the guard runs only when the module is executed directly (python module.py). This is useful for testing, running a demo, or providing a CLI entry point while keeping the module reusable.

Should I use relative or absolute imports?

+

PEP 8 recommends absolute imports for readability: from mypackage.submodule import func. Relative imports (from . import submodule) are acceptable inside a package, especially to avoid repeating the package name. But absolute imports are generally clearer and more maintainable.

How do I share my Python package with others?

+

Write a setup.py or pyproject.toml, then build and upload to PyPI using build and twine. Or share the source via GitHub and instruct users to install with pip install git+https://....
