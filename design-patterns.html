Creational Patterns

Creational patterns abstract the instantiation process — they help make systems independent of how objects are created, composed, and represented.

1️⃣

Singleton

Ensure a class has only one instance and provide a global access point to it.

CreationalGoF+
Code
When to Use
Real World
import threading

class Singleton:
    _instance = None
    _lock = threading.Lock()

    def __new__(cls, *args, **kwargs):
        if cls._instance is None:
            with cls._lock:
                if cls._instance is None:
                    cls._instance = super().__new__(cls)
        return cls._instance

class AppConfig(Singleton):
    def __init__(self):
        if not hasattr(self, '_init'):
            self.debug = False
            self.db_url = "postgresql://localhost/app"
            self._init = True

a = AppConfig()
b = AppConfig()
print(a is b)  # True — same object
💡 The double-checked locking (checking _instance twice — outside and inside the lock) prevents multiple threads from each creating their own instance while the first is being constructed.
✓ Use When
  • Exactly one instance must coordinate actions across the system
  • Config, logging, or connection pool objects
  • Shared mutable state needs a single source of truth
✗ Avoid When
  • Testing — singletons make unit tests hard to isolate
  • You just want global variables (use module-level instead)
  • The "single instance" constraint might change later
Real-World Usage
  • Python's logging.getLogger() returns the same logger for the same name
  • Django's database connection pool
  • Flask's application context (current_app)
  • Python module objects are themselves singletons — importing a module twice returns the cached version

Factory Method

Define an interface for creating objects, but let subclasses decide which class to instantiate.

CreationalGoF+
Code
When to Use
Real World
from abc import ABC, abstractmethod

class Notification(ABC):
    @abstractmethod
    def send(self, message: str) -> None: ...

class EmailNotification(Notification):
    def send(self, message): print(f"📧 Email: {message}")

class SMSNotification(Notification):
    def send(self, message): print(f"📱 SMS: {message}")

class PushNotification(Notification):
    def send(self, message): print(f"🔔 Push: {message}")

class NotificationFactory:
    _map = {"email": EmailNotification, "sms": SMSNotification, "push": PushNotification}

    @classmethod
    def create(cls, channel: str) -> Notification:
        klass = cls._map.get(channel.lower())
        if not klass:
            raise ValueError(f"Unknown channel: {channel}")
        return klass()

for ch in ["email", "sms", "push"]:
    NotificationFactory.create(ch).send("Hello!")
✓ Use When
  • A class can't anticipate the class of objects it must create
  • The creation logic should be in subclasses, not the base
  • You want to decouple client code from concrete classes
✗ Avoid When
  • Only one type of object will ever be created — use a constructor
  • The hierarchy would become complex for simple creation needs
Real-World Usage
  • Django ORM's objects.create() and manager methods
  • SQLAlchemy's create_engine() creates different engine types from a URL string
  • pytest's fixture system — fixtures are factory functions

Abstract Factory

Provide an interface for creating families of related objects without specifying their concrete classes.

CreationalGoF+
Code
When to Use
Real World
from abc import ABC, abstractmethod

class Button(ABC):
    @abstractmethod
    def render(self): ...

class Checkbox(ABC):
    @abstractmethod
    def render(self): ...

class UIFactory(ABC):
    @abstractmethod
    def create_button(self) -> Button: ...
    @abstractmethod
    def create_checkbox(self) -> Checkbox: ...

class DarkButton(Button):
    def render(self): print("[Dark Button]")

class DarkCheckbox(Checkbox):
    def render(self): print("[Dark Checkbox]")

class DarkThemeFactory(UIFactory):
    def create_button(self): return DarkButton()
    def create_checkbox(self): return DarkCheckbox()

def build_ui(factory: UIFactory):
    factory.create_button().render()
    factory.create_checkbox().render()

build_ui(DarkThemeFactory())
✓ Use When
  • System must be independent of how products are created
  • You need families of related products to work together
  • Swapping entire product families (e.g. themes, platforms) at runtime
✗ Avoid When
  • Only one product family — Factory Method suffices
  • Adding new product types requires changing all factory interfaces
Real-World Usage
  • GUI toolkits: wxPython, Qt — platform-specific widget factories
  • Database abstraction layers — different adapters per DB engine
  • Theme engines in design systems

Builder

Separate the construction of a complex object from its representation using a step-by-step interface.

CreationalGoF+
Code
When to Use
Real World
from dataclasses import dataclass, field

@dataclass
class Query:
    table: str
    columns: list[str]
    conditions: list[str]
    order: str | None
    limit_val: int | None

    def to_sql(self) -> str:
        cols = ", ".join(self.columns) or "*"
        sql  = f"SELECT {cols} FROM {self.table}"
        if self.conditions:
            sql += f" WHERE {' AND '.join(self.conditions)}"
        if self.order:
            sql += f" ORDER BY {self.order}"
        if self.limit_val:
            sql += f" LIMIT {self.limit_val}"
        return sql

class QueryBuilder:
    def __init__(self, table: str):
        self._table = table
        self._cols, self._conds = [], []
        self._order = self._limit = None

    def select(self, *cols): self._cols.extend(cols); return self
    def where(self, cond):  self._conds.append(cond); return self
    def order_by(self, col): self._order = col; return self
    def limit(self, n):     self._limit = n; return self
    def build(self):
        return Query(self._table, self._cols, self._conds, self._order, self._limit)

q = (QueryBuilder("users")
     .select("id", "name", "email")
     .where("active = true")
     .order_by("name")
     .limit(10)
     .build())
print(q.to_sql())
💡 The Builder pattern shines when an object requires many parameters, especially optional ones. It prevents the "telescoping constructor" anti-pattern where you have constructors with 8+ parameters.
✓ Use When
  • Complex objects with many optional parameters
  • Different representations of the same construction process
  • Step-by-step construction with validation at each step
✗ Avoid When
  • Simple objects — use a constructor or @dataclass
  • The number of steps is fixed and small
Real-World Usage
  • SQLAlchemy's query API: session.query(User).filter(...).order_by(...).limit(10)
  • Python's pathlib.Path chaining
  • HTTP clients: httpx.Client().get(url).headers(...)
  • Python's unittest.mock.patch builder

Prototype

Create new objects by cloning an existing object rather than creating from scratch.

CreationalGoF+
Code
When to Use
Real World
import copy
from dataclasses import dataclass, field

@dataclass
class GameCharacter:
    name: str
    hp: int
    stats: dict[str, int] = field(default_factory=dict)
    inventory: list[str] = field(default_factory=list)

    def clone(self) -> "GameCharacter":
        return copy.deepcopy(self)

    def with_name(self, name: str) -> "GameCharacter":
        c = self.clone()
        c.name = name
        return c

# Base template — build once, clone many times
warrior_template = GameCharacter(
    name="Warrior", hp=200,
    stats={"str": 18, "dex": 10, "int": 6},
    inventory=["sword", "shield"]
)

player1 = warrior_template.with_name("Aragorn")
player2 = warrior_template.with_name("Boromir")
player2.inventory.append("horn")

print(player1)
print(player2)
print("Template unchanged:", warrior_template.inventory)
✓ Use When
  • Object creation is expensive (DB queries, heavy computation)
  • You need many slightly-different variants of an object
  • Classes to instantiate are specified at runtime
✗ Avoid When
  • Objects are cheap to create from scratch
  • Deep copying creates unexpected side effects with circular references
Real-World Usage
  • Python's copy.copy() and copy.deepcopy() built-ins implement this pattern
  • Django model instances: obj.pk = None; obj.save() clones a record
  • Game engines for spawning enemy variants from templates
Structural Patterns

Structural patterns deal with object composition — creating relationships between objects to form larger, flexible structures.

🔌

Adapter

Convert one interface into another that clients expect — lets incompatible classes work together.

StructuralGoF+
Code
When to Use
Real World
class LegacyPayment:
    # Old system — can't change this
    def do_payment(self, amount_cents: int, currency_code: str) -> bool:
        print(f"Legacy: paying {amount_cents}¢ in {currency_code}")
        return True

class ModernPaymentGateway:
    # New interface all new code expects
    def charge(self, amount: float, currency: str = "USD") -> bool: ...

class LegacyPaymentAdapter(ModernPaymentGateway):
    # Wraps the old system in the new interface
    def __init__(self):
        self._legacy = LegacyPayment()

    def charge(self, amount: float, currency: str = "USD") -> bool:
        cents = int(amount * 100)
        return self._legacy.do_payment(cents, currency.upper())

def process_order(gateway: ModernPaymentGateway, total: float):
    print(f"Processing ${total:.2f}")
    gateway.charge(total)

process_order(LegacyPaymentAdapter(), 29.99)
✓ Use When
  • Integrating a legacy system with a new interface
  • Using a third-party library with an incompatible interface
  • Reusing existing functionality without modifying source code
✗ Avoid When
  • You can change the original class — prefer direct refactoring
  • The translation logic is so complex it obscures the intent
Real-World Usage
  • Python's io.TextIOWrapper adapts a binary stream to a text stream
  • Django REST Framework serializers adapt model instances to JSON-serializable dicts
  • Database drivers: psycopg2, cx_Oracle all adapt different DB APIs to Python's DB-API 2.0

Decorator (Structural)

Attach additional responsibilities to an object dynamically — a flexible alternative to subclassing.

StructuralGoF+
Code
When to Use
Real World
from abc import ABC, abstractmethod

class DataSource(ABC):
    @abstractmethod
    def write(self, data: str): ...
    @abstractmethod
    def read(self) -> str: ...

class FileDataSource(DataSource):
    def __init__(self, name): self._data = ""; self.name = name
    def write(self, data): self._data = data; print(f"File({self.name}): written")
    def read(self): return self._data

class DataSourceDecorator(DataSource):
    def __init__(self, wrapped: DataSource): self._wrapped = wrapped
    def write(self, data): self._wrapped.write(data)
    def read(self): return self._wrapped.read()

class EncryptionDecorator(DataSourceDecorator):
    def write(self, data):
        encrypted = data[::-1]  # toy encryption
        print(f"Encrypted: {encrypted}")
        super().write(encrypted)
    def read(self): return super().read()[::-1]

class CompressionDecorator(DataSourceDecorator):
    def write(self, data):
        print(f"Compressed ({len(data)} → {len(data)//2} bytes)")
        super().write(data)

source = CompressionDecorator(EncryptionDecorator(FileDataSource("data.txt")))
source.write("Hello, Design Patterns!")
print("Read back:", source.read())
💡 Note the difference from Python's function decorator (@decorator syntax). The GoF Decorator wraps an object at runtime; Python's @decorator wraps a function at definition time. Both follow the same structural idea.
✓ Use When
  • Adding behaviour to individual objects without affecting others
  • Inheritance would cause a class explosion of combinations
  • Behaviour needs to be added and removed at runtime
✗ Avoid When
  • The order of decorators matters in unexpected ways
  • Simple extension via inheritance is cleaner
Real-World Usage
  • Python's functools.lru_cache, property, staticmethod are function decorators
  • Python's io module: BufferedReader(FileIO(...))
  • Django middleware wraps request/response handlers
  • Flask's @app.route decorator

Facade

Provide a simplified interface to a complex subsystem — hide complexity behind a clean API.

StructuralGoF+
Code
When to Use
Real World
class VideoDecoder:
    def decode(self, f): print(f"Decoded video: {f}")

class AudioMixer:
    def mix(self, f): print(f"Mixed audio: {f}")

class SubtitleParser:
    def parse(self, f): print(f"Parsed subtitles: {f}")

class VideoRenderer:
    def render(self, v, a): print("Rendered final frame buffer")

class VideoPlayerFacade:
    # One simple method hides 4 subsystems
    def __init__(self):
        self._decoder  = VideoDecoder()
        self._mixer    = AudioMixer()
        self._subs     = SubtitleParser()
        self._renderer = VideoRenderer()

    def play(self, filename: str):
        print(f"▶ Playing {filename}")
        self._decoder.decode(filename)
        self._mixer.mix(filename)
        self._subs.parse(filename)
        self._renderer.render("video", "audio")
        print("✓ Playback started\n")

player = VideoPlayerFacade()
player.play("movie.mp4")
✓ Use When
  • A simple interface is needed for a complex subsystem
  • Layering subsystems — each layer communicates via facade
  • Reducing dependencies on internal subsystem classes
✗ Avoid When
  • Clients need fine-grained control over the subsystem
  • The facade becomes a "god object" owning too much logic
Real-World Usage
  • Python's requests library is a facade over urllib3, SSL, encodings, and HTTP parsing
  • Django's ORM is a facade over raw SQL and DB drivers
  • boto3 (AWS SDK) facades dozens of AWS service APIs

Proxy

Provide a surrogate or placeholder for another object to control access to it.

StructuralGoF+
Code
When to Use
Real World
from abc import ABC, abstractmethod class Database(ABC): @abstractmethod def query(self, sql: str) -> list: ... class RealDatabase(Database): def query(self, sql: str) -> list: print(f"[DB] Executing: {sql}") return [{"id": 1, "name": "Ana"}] class CachedDatabaseProxy(Database): def __init__(self, db: Database): self._db = db self._cache: dict = {} self._hits = 0 def query(self, sql: str) -> list: if sql in self._cache: self._hits += 1 print(f"[Cache HIT #{self._hits}] {sql}") return self._cache[sql] result = self._db.query(sql) self._cache[sql] = result return result db = CachedDatabaseProxy(RealDatabase()) db.query("SELECT * FROM users") db.query("SELECT * FROM users") # cache hit db.query("SELECT * FROM users") # cache hit
✓ Use When
  • Lazy initialisation of expensive objects
  • Caching results of expensive operations
  • Access control — check permissions before delegating
  • Logging or auditing access to an object
✗ Avoid When
  • The overhead of the proxy outweighs its benefits
  • The response time increase isn't acceptable
Real-World Usage
  • Python's unittest.mock.MagicMock is a proxy that records all calls
  • SQLAlchemy lazy-loaded relationships — a proxy that queries on first access
  • Python's weakref.proxy()
  • CDNs and reverse proxies (Nginx, Cloudflare) follow this exact pattern

Composite

Compose objects into tree structures. Let clients treat individual objects and compositions uniformly.

StructuralGoF+
Code
When to Use
Real World
from abc import ABC, abstractmethod

class FileSystemItem(ABC):
    def __init__(self, name): self.name = name

    @abstractmethod
    def size(self) -> int: ...

    @abstractmethod
    def display(self, indent=0): ...

class File(FileSystemItem):  # Leaf
    def __init__(self, name, size):
        super().__init__(name); self._size = size
    def size(self): return self._size
    def display(self, indent=0):
        print(" " * indent + f"📄 {self.name} ({self._size}B)")

class Directory(FileSystemItem):  # Composite
    def __init__(self, name):
        super().__init__(name); self._children = []
    def add(self, item: FileSystemItem):
        self._children.append(item); return self
    def size(self): return sum(c.size() for c in self._children)
    def display(self, indent=0):
        print(" " * indent + f"📁 {self.name}/ ({self.size()}B)")
        for c in self._children: c.display(indent + 2)

root = Directory("project")
src  = Directory("src")
src.add(File("main.py", 1200)).add(File("utils.py", 800))
root.add(src).add(File("README.md", 300))
root.display()
✓ Use When
  • Representing part-whole hierarchies (file systems, UI trees, org charts)
  • Client code should treat individual objects and groups uniformly
  • Recursive tree structures are natural for the domain
✗ Avoid When
  • Leaves and composites need very different interfaces
  • The hierarchy is shallow — a list suffices
Real-World Usage
  • Python's AST (Abstract Syntax Tree) — expressions contain sub-expressions
  • HTML/XML DOM — every node is either a leaf (text) or a composite (element)
  • GUI frameworks: widget trees, layout containers
  • pytest's test collection: TestSuite contains TestCases

Bridge

Decouple an abstraction from its implementation so both can vary independently.

StructuralGoF+
Code
When to Use
Real World
from abc import ABC, abstractmethod

class Renderer(ABC):  # Implementation
    @abstractmethod
    def render_circle(self, x, y, r): ...

class SVGRenderer(Renderer):
    def render_circle(self, x, y, r):
        print(f'<circle cx="{x}" cy="{y}" r="{r}"/>')

class CanvasRenderer(Renderer):
    def render_circle(self, x, y, r):
        print(f"ctx.arc({x},{y},{r},0,2*Math.PI)")

class Shape(ABC):  # Abstraction
    def __init__(self, renderer: Renderer):
        self.renderer = renderer

class Circle(Shape):
    def __init__(self, x, y, r, renderer):
        super().__init__(renderer)
        self.x, self.y, self.r = x, y, r

    def draw(self):
        self.renderer.render_circle(self.x, self.y, self.r)

Circle(10, 20, 5, SVGRenderer()).draw()
Circle(10, 20, 5, CanvasRenderer()).draw()
✓ Use When
  • Avoiding permanent binding between abstraction and implementation
  • Both abstraction and implementation should be extensible independently
  • Implementation changes shouldn't affect client code
✗ Avoid When
  • Only one implementation exists — needless complexity
  • The abstraction and implementation rarely change independently
Real-World Usage
  • Python's logging module: Logger (abstraction) + Handler implementations (FileHandler, StreamHandler)
  • Database drivers: the DB-API 2.0 spec is an abstraction; each vendor provides the implementation

Flyweight

Use sharing to efficiently support a large number of fine-grained objects — minimise memory by sharing common state.

StructuralGoF+
Code
When to Use
Real World
import sys

class TreeType:
    # Intrinsic state — shared, immutable
    def __init__(self, name, color, texture):
        self.name, self.color, self.texture = name, color, texture
    def draw(self, x, y):  # x,y are extrinsic state — passed in
        print(f"Drawing {self.name} at ({x},{y})")

class TreeFactory:
    _types: dict[tuple, TreeType] = {}

    @classmethod
    def get(cls, name, color, texture) -> TreeType:
        key = (name, color, texture)
        if key not in cls._types:
            cls._types[key] = TreeType(name, color, texture)
            print(f"Created new TreeType: {name}")
        return cls._types[key]

forest = []
for i in range(1000):
    t = TreeFactory.get("Pine", "green", "rough")
    forest.append((t, i, i*2))  # extrinsic state stored outside

print(f"Trees in forest: {len(forest)}")
print(f"Unique TreeType objects: {len(TreeFactory._types)}")
forest[42][0].draw(*forest[42][1:])
✓ Use When
  • Huge numbers of similar objects are consuming too much RAM
  • Most object state can be made extrinsic (passed in)
  • Many identical objects can be replaced by a few shared ones
✗ Avoid When
  • Objects have little shared state — sharing saves nothing
  • The complexity outweighs the memory savings
Real-World Usage
  • Python interns small integers (−5 to 256) and common strings automatically
  • Font rendering: glyph objects (shape data) are shared; position is extrinsic
  • Game engines: terrain tiles, particle effects share type objects
  • Python's __slots__ reduces per-instance memory similarly
Behavioral Patterns

Behavioral patterns are concerned with algorithms and the assignment of responsibilities between objects — how objects collaborate and distribute work.

📡

Observer

Define a one-to-many dependency so when one object changes state, all dependents are notified automatically.

BehavioralGoF+
Code
When to Use
Real World
from abc import ABC, abstractmethod
from dataclasses import dataclass

class Observer(ABC):
    @abstractmethod
    def update(self, event: str, data) -> None: ...

class Subject:
    def __init__(self):
        self._observers: list[Observer] = []

    def subscribe(self, obs): self._observers.append(obs)
    def unsubscribe(self, obs): self._observers.remove(obs)

    def _notify(self, event, data=None):
        for obs in self._observers:
            obs.update(event, data)

class StockMarket(Subject):
    def __init__(self): super().__init__(); self._prices = {}
    def set_price(self, ticker, price):
        old = self._prices.get(ticker)
        self._prices[ticker] = price
        self._notify("price_change", {"ticker": ticker, "old": old, "new": price})

class AlertObserver(Observer):
    def update(self, event, data):
        if data["old"] and abs(data["new"]-data["old"])/data["old"] > 0.05:
            print(f"🚨 {data['ticker']} moved {(data['new']-data['old'])/data['old']:.1%}!")

class LogObserver(Observer):
    def update(self, event, data):
        print(f"[LOG] {data['ticker']}: {data['old']} → {data['new']}")

market = StockMarket()
market.subscribe(LogObserver())
market.subscribe(AlertObserver())
market.set_price("AAPL", 190.0)
market.set_price("AAPL", 202.0)  # triggers alert
✓ Use When
  • A change in one object requires updating others — number unknown
  • Decoupled publish/subscribe without tight coupling
  • Event-driven architectures, UI updates, reactive systems
✗ Avoid When
  • Observers are called in a specific required order
  • Unexpected updates cascade and cause performance issues
Real-World Usage
  • Django signals: post_save.connect(handler)
  • Python's tkinter event binding: widget.bind('<Click>', callback)
  • JavaScript's EventEmitter (same pattern)
  • SQLAlchemy ORM events: event.listen()
  • FastAPI's dependency injection follows observer-like patterns

Strategy

Define a family of algorithms, encapsulate each, and make them interchangeable at runtime.

BehavioralGoF+
Code
When to Use
Real World
from abc import ABC, abstractmethod
from typing import Callable

# Strategy using Protocol / Callable (idiomatic Python)
SortStrategy = Callable[[list], list]

def bubble_sort(data: list) -> list:
    d = data.copy()
    for i in range(len(d)):
        for j in range(len(d) - i - 1):
            if d[j] > d[j+1]: d[j], d[j+1] = d[j+1], d[j]
    return d

def quick_sort(data: list) -> list:
    if len(data) <= 1: return data
    pivot = data[len(data) // 2]
    left  = [x for x in data if x < pivot]
    mid   = [x for x in data if x == pivot]
    right = [x for x in data if x > pivot]
    return quick_sort(left) + mid + quick_sort(right)

class DataProcessor:
    def __init__(self, strategy: SortStrategy = sorted):
        self._sort = strategy

    def set_strategy(self, strategy: SortStrategy):
        self._sort = strategy

    def process(self, data: list) -> list:
        print(f"Using: {self._sort.__name__}")
        return self._sort(data)

data = [64, 34, 25, 12, 22]
proc = DataProcessor(bubble_sort)
print(proc.process(data))
proc.set_strategy(quick_sort)
print(proc.process(data))
💡 In Python, strategies are naturally just callables (functions or lambdas). No need for abstract base classes — pass the function directly. This is idiomatic Python Strategy.
✓ Use When
  • Multiple algorithms for the same task, switchable at runtime
  • Eliminating large conditionals that select behaviour
  • Isolating algorithm implementation details from the code that uses it
✗ Avoid When
  • Only one or two algorithms — simple if/else is clearer
  • Clients never need to be aware of which strategy is used
Real-World Usage
  • Python's sorted(key=...) — the key function is a strategy
  • scikit-learn estimators: swap LinearRegression for RandomForest with the same .fit()/.predict() interface
  • Password hashing: bcrypt, argon2, scrypt are interchangeable strategies
  • Compression: gzip, bz2, lzma follow the same interface

Command

Encapsulate a request as an object — enabling undo/redo, queuing, logging, and transactional operations.

BehavioralGoF+
Code
When to Use
Real World
from abc import ABC, abstractmethod
from collections import deque

class Command(ABC):
    @abstractmethod
    def execute(self): ...
    @abstractmethod
    def undo(self): ...

class TextEditor:
    def __init__(self): self.text = ""
    def insert(self, s): self.text += s
    def delete(self, n): self.text = self.text[:-n]

class InsertCommand(Command):
    def __init__(self, editor, text):
        self.editor, self.text = editor, text
    def execute(self): self.editor.insert(self.text)
    def undo(self):    self.editor.delete(len(self.text))

class CommandHistory:
    def __init__(self): self._stack = deque()
    def execute(self, cmd: Command):
        cmd.execute(); self._stack.append(cmd)
    def undo(self):
        if self._stack: self._stack.pop().undo()

editor  = TextEditor()
history = CommandHistory()

history.execute(InsertCommand(editor, "Hello"))
history.execute(InsertCommand(editor, " World"))
print(editor.text)   # Hello World
history.undo()
print(editor.text)   # Hello
history.undo()
print(editor.text)   # (empty)
✓ Use When
  • Undo/redo functionality is required
  • Operations need to be queued, scheduled, or logged
  • Implementing transactional behaviour (rollback)
✗ Avoid When
  • Simple, non-reversible operations — over-engineering
  • The command hierarchy grows to hundreds of classes
Real-World Usage
  • Django database migrations: each migration is a Command with forwards() and backwards()
  • Celery tasks: tasks are serialised commands queued for async execution
  • Git commits are commands with undo (revert) operations
  • Python's asyncio.Task encapsulates a coroutine as a schedulable command
⛓️

Chain of Responsibility

Pass requests along a chain of handlers — each handler decides to process or forward the request.

BehavioralGoF+
Code
When to Use
Real World
from abc import ABC, abstractmethod
from dataclasses import dataclass

@dataclass
class Request:
    user_role:   str
    amount:      float
    description: str

class Approver(ABC):
    def __init__(self, name, limit):
        self.name, self.limit = name, limit
        self._next: Approver | None = None

    def set_next(self, handler: "Approver") -> "Approver":
        self._next = handler; return handler

    def handle(self, req: Request):
        if req.amount <= self.limit:
            print(f"✅ {self.name} approved ${req.amount:.0f} — {req.description}")
        elif self._next:
            print(f"⬆  {self.name} escalating ${req.amount:.0f}…")
            self._next.handle(req)
        else:
            print(f"❌ ${req.amount:.0f} exceeds all approval limits")

manager = Approver("Manager",   1_000)
director = Approver("Director", 10_000)
cfo      = Approver("CFO",       100_000)
manager.set_next(director).set_next(cfo)

for amt in [500, 5_000, 50_000, 150_000]:
    manager.handle(Request("employee", amt, "Expense"))
    print()
✓ Use When
  • More than one object may handle a request, and the handler isn't known a priori
  • Processing pipeline where each stage passes to the next
  • The set of handlers should be specifiable dynamically
✗ Avoid When
  • Every request must be handled — chain might drop requests silently
  • Performance-critical paths — traversing a long chain has overhead
Real-World Usage
  • Django/Flask middleware stacks: each middleware can handle or pass the request
  • Python's logging: loggers pass records up to parent handlers
  • Exception handling: try/except chains bubble up through the call stack
  • Click command groups route CLI args through a handler chain

Template Method

Define the skeleton of an algorithm in a base class — let subclasses override specific steps without changing the algorithm's structure.

BehavioralGoF+
Code
When to Use
Real World
from abc import ABC, abstractmethod

class DataMigration(ABC):
    # Template method — the invariant skeleton
    def run(self):
        self.connect()
        data = self.extract()
        data = self.transform(data)
        self.load(data)
        self.disconnect()

    @abstractmethod
    def connect(self): ...
    @abstractmethod
    def extract(self) -> list: ...
    @abstractmethod
    def transform(self, data) -> list: ...
    @abstractmethod
    def load(self, data): ...
    def disconnect(self): print("Disconnected")  # optional hook

class CSVToJSON(DataMigration):
    def connect(self):     print("Opening CSV file")
    def extract(self):    return ["row1", "row2", "row3"]
    def transform(self, d): return [{"data": r} for r in d]
    def load(self, d):      print(f"Writing {len(d)} JSON records")

CSVToJSON().run()
✓ Use When
  • The overall algorithm structure is fixed but certain steps vary
  • Avoiding code duplication of the common skeleton across subclasses
  • Controlling extension points precisely
✗ Avoid When
  • The algorithm varies too much — Strategy is more flexible
  • Subclasses must override too many methods — hints at wrong abstraction
Real-World Usage
  • Django's class-based views: View.dispatch() calls get()/post() — subclasses override
  • Python's unittest.TestCase: setUp() and tearDown() hooks
  • SQLAlchemy ORM: __init__ hooks, before_flush events

State

Allow an object to alter its behaviour when its internal state changes — the object will appear to change its class.

BehavioralGoF+
Code
When to Use
Real World
from abc import ABC, abstractmethod

class OrderState(ABC):
    @abstractmethod
    def confirm(self, order): ...
    @abstractmethod
    def ship(self, order): ...
    @abstractmethod
    def deliver(self, order): ...
    def __str__(self): return type(self).__name__

class PendingState(OrderState):
    def confirm(self, o): print("✅ Confirmed"); o.state = ConfirmedState()
    def ship(self, o):    print("❌ Must confirm first")
    def deliver(self, o): print("❌ Must confirm first")

class ConfirmedState(OrderState):
    def confirm(self, o): print("Already confirmed")
    def ship(self, o):    print("🚚 Shipped"); o.state = ShippedState()
    def deliver(self, o): print("❌ Must ship first")

class ShippedState(OrderState):
    def confirm(self, o): print("Already done")
    def ship(self, o):    print("Already shipped")
    def deliver(self, o): print("📦 Delivered!"); o.state = DeliveredState()

class DeliveredState(OrderState):
    def confirm(self, o): print("Order complete")
    def ship(self, o):    print("Order complete")
    def deliver(self, o): print("Order complete")

class Order:
    def __init__(self): self.state = PendingState()
    def confirm(self): self.state.confirm(self)
    def ship(self):    self.state.ship(self)
    def deliver(self): self.state.deliver(self)

o = Order()
o.ship()     # blocked
o.confirm()
o.ship()
o.deliver()
✓ Use When
  • Object behaviour depends heavily on its state, which changes at runtime
  • Replacing large conditionals that check object state
  • Finite state machines: orders, workflows, game entities
✗ Avoid When
  • Few states and simple transitions — if/else is clearer
  • State changes are infrequent and behaviour rarely varies
Real-World Usage
  • Python's generators: yield implements a cooperative state machine
  • asyncio's event loop tracks coroutine states (pending, running, done)
  • Django's order/payment status workflows
  • Network connections: closed → connecting → connected → closing

Iterator

Provide a way to sequentially access elements of a collection without exposing its underlying representation.

BehavioralGoF+
Code
When to Use
Real World
class BinaryTreeNode:
    def __init__(self, val, left=None, right=None):
        self.val, self.left, self.right = val, left, right

class InOrderIterator:
    def __init__(self, root):
        self._stack = []
        self._push_left(root)

    def _push_left(self, node):
        while node:
            self._stack.append(node)
            node = node.left

    def __iter__(self): return self

    def __next__(self):
        if not self._stack:
            raise StopIteration
        node = self._stack.pop()
        self._push_left(node.right)
        return node.val

tree = BinaryTreeNode(
    4,
    left  = BinaryTreeNode(2, BinaryTreeNode(1), BinaryTreeNode(3)),
    right = BinaryTreeNode(6, BinaryTreeNode(5), BinaryTreeNode(7))
)

print("In-order:", list(InOrderIterator(tree)))
💡 Python's __iter__ and __next__ protocol IS the Iterator pattern built into the language. Every for loop calls these methods. Generators are the most Pythonic way to implement iterators.
✓ Use When
  • Hiding the internal representation of a collection
  • Supporting multiple simultaneous traversals
  • Uniform interface across different collection types
✗ Avoid When
  • Python's built-in iteration protocol already handles this — just implement __iter__
Real-World Usage
  • Python's for loop, iter(), next() — the iterator protocol is native
  • All Python collections implement __iter__: list, dict, set, str, range
  • SQLAlchemy query results are iterators over rows
  • Django querysets are lazy iterators that only hit the DB when iterated

Mediator

Define an object that encapsulates how a set of objects interact — promotes loose coupling by preventing direct references.

BehavioralGoF+
Code
When to Use
Real World
class ChatRoom:
    def __init__(self, name):
        self.name = name
        self._members: list["User"] = []

    def join(self, user: "User"):
        self._members.append(user)
        self.broadcast(f"{user.name} joined", sender=user)

    def broadcast(self, msg, sender):
        for m in self._members:
            if m is not sender:
                m.receive(f"[{self.name}] {sender.name}: {msg}")

class User:
    def __init__(self, name):
        self.name = name
        self._room: ChatRoom | None = None

    def join(self, room): self._room = room; room.join(self)
    def say(self, msg):   self._room.broadcast(msg, self)
    def receive(self, msg): print(f"  → {self.name} receives: {msg}")

room = ChatRoom("#python")
alice, bob, carol = User("Alice"), User("Bob"), User("Carol")
for u in [alice, bob, carol]: u.join(room)
alice.say("Hello everyone!")
✓ Use When
  • Many objects communicate in complex, tangled ways
  • Reusing an object is hard because it refers to too many others
  • Decoupling components that would otherwise be tightly coupled
✗ Avoid When
  • The mediator itself becomes a "god object" with too much logic
  • Components are few and communication is simple
Real-World Usage
  • Django signals: the signal dispatcher is a mediator between senders and receivers
  • Message brokers: RabbitMQ, Kafka act as mediators between producers and consumers
  • Air traffic control: planes talk to the tower, not each other

When to Use

import copy from dataclasses import dataclass @dataclass(frozen=True) class Memento: state: dict # immutable snapshot class GameCharacter: def __init__(self, name): self._state = {"name": name, "hp": 100, "level": 1, "xp": 0} def save(self) -> Memento: return Memento(copy.deepcopy(self._state)) def restore(self, m: Memento): self._state = copy.deepcopy(m.state) def fight(self): self._state["hp"] -= 30; self._state["xp"] += 50 def __str__(self): return f"HP={self._state['hp']} XP={self._state['xp']}" hero = GameCharacter("Aria") checkpoint = hero.save() print("Before boss:", hero) hero.fight(); hero.fight(); hero.fight() print("After losses:", hero) hero.restore(checkpoint) print("Restored:", hero)

Visitor

Represent an operation to be performed on elements of an object structure — define new operations without changing the classes.

BehavioralGoF+
Code
When to Use
Real World
from abc import ABC, abstractmethod
from functools import singledispatch

# Pythonic Visitor using singledispatch
class Number:
    def __init__(self, v): self.v = v

class Add:
    def __init__(self, l, r): self.left, self.right = l, r

class Mul:
    def __init__(self, l, r): self.left, self.right = l, r

@singledispatch
def evaluate(node): raise TypeError(f"Unknown: {type(node)}")

@evaluate.register(Number)
def _(node): return node.v

@evaluate.register(Add)
def _(node): return evaluate(node.left) + evaluate(node.right)

@evaluate.register(Mul)
def _(node): return evaluate(node.left) * evaluate(node.right)

@singledispatch
def to_str(node): raise TypeError()
@to_str.register(Number)
def _(n): return str(n.v)
@to_str.register(Add)
def _(n): return f"({to_str(n.left)} + {to_str(n.right)})"
@to_str.register(Mul)
def _(n): return f"({to_str(n.left)} * {to_str(n.right)})"

expr = Add(Mul(Number(2), Number(3)), Number(4))
print(to_str(expr), "=", evaluate(expr))  # (2*3)+4 = 10
✓ Use When
  • Many distinct and unrelated operations on an object structure
  • Object classes rarely change but operations on them change frequently
  • Gathering related operations across class hierarchies
✗ Avoid When
  • The object hierarchy changes frequently — all visitors must be updated
  • Classes in the hierarchy need to be kept encapsulated from visitors
Real-World Usage
  • Python's ast.NodeVisitor — traverse and operate on Python AST nodes
  • functools.singledispatch — idiomatic Python dispatch on type
  • Compiler passes: type checking, optimisation, code generation are separate visitors over the same AST
