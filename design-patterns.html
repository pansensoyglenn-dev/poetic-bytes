What are Design Patterns?

In 1994, Erich Gamma, Richard Helm, Ralph Johnson, and John Vlissides — known as the Gang of Four (GoF) — published Design Patterns: Elements of Reusable Object-Oriented Software. It catalogued 23 patterns that solve recurring design problems. These patterns are language-agnostic blueprints, and Python's flexibility makes many of them especially clean to implement. Understanding them separates junior developers from senior ones.

🟢 Creational Patterns

Control how objects are created — decouple construction from usage

Creational patterns abstract the instantiation process. They make a system independent of how its objects are created, composed, and represented. When simple ClassName() construction becomes a problem — because the class name is unknown, construction is complex, or only one instance must exist — creational patterns provide the answer.

1 Singleton

🔒 Singleton — One Instance, Global Access

The Singleton pattern ensures a class has only one instance and provides a global access point to it. It is used for shared resources like database connections, configuration managers, and logging systems where multiple instances would cause conflicts or waste resources.

Python's module system is itself a natural singleton — a module is only imported once. For class-based singletons, the cleanest approach overrides __new__.

# Singleton using __new__
class DatabaseConnection:
    _instance = None

    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance.connected = False
            cls._instance.host = "localhost"
        return cls._instance

    def connect(self, host: str):
        self.host = host
        self.connected = True
        print(f"Connected to {self.host}")

    def status(self) -> str:
        return f"{'Connected' if self.connected else 'Disconnected'} — {self.host}"

# Both variables point to the exact same object
db1 = DatabaseConnection()
db2 = DatabaseConnection()

db1.connect("prod-server-01")
print(db2.status())          # Connected — prod-server-01
print(db1 is db2)           # True — same instance
print(id(db1) == id(db2))   # True
✅ Real-world use: database connection pool, application config, logger, thread pool, cache manager.
⚠️ Singletons make unit testing harder because shared state bleeds between tests. Prefer dependency injection in testable code.
2 Factory Method

🏭 Factory Method — Delegate Object Creation to Subclasses

The Factory Method defines an interface for creating an object but lets subclasses decide which class to instantiate. The creator class calls its own factory method rather than calling a constructor directly, allowing subclasses to override what gets created. This pattern appears everywhere in Python frameworks — Django's form fields, SQLAlchemy's column types, and pytest fixtures all use factory methods.

from abc import ABC, abstractmethod

# Abstract creator
class Notification(ABC):
    def send(self, message: str):
        channel = self.create_channel()   # factory method call
        channel.deliver(message)
        print(f"Sent via {channel.__class__.__name__}: {message}")

    @abstractmethod
    def create_channel(self):   # the factory method
        pass

# Concrete products
class EmailChannel:
    def deliver(self, msg): print(f"  📧 Email: {msg}")

class SMSChannel:
    def deliver(self, msg): print(f"  📱 SMS: {msg}")

class PushChannel:
    def deliver(self, msg): print(f"  🔔 Push: {msg}")

# Concrete creators — each overrides the factory method
class EmailNotification(Notification):
    def create_channel(self): return EmailChannel()

class SMSNotification(Notification):
    def create_channel(self): return SMSChannel()

class PushNotification(Notification):
    def create_channel(self): return PushChannel()

# Client code — works with any Notification subclass
for notifier in [EmailNotification(), SMSNotification(), PushNotification()]:
    notifier.send("Your order has shipped!")
✅ Real-world use: Django ORM field types, logging handlers, serializer fields in DRF, plugin systems.
3 Abstract Factory

🏗️ Abstract Factory — Families of Related Objects

Abstract Factory provides an interface for creating families of related or dependent objects without specifying their concrete classes. Where Factory Method creates one product, Abstract Factory creates an entire suite of related products. Think of it as a factory of factories — useful when your system must work with multiple product variants (e.g. light/dark UI theme, Windows/macOS widgets).

from abc import ABC, abstractmethod

# Abstract products
class Button(ABC):
    @abstractmethod
    def render(self) -> str: pass

class Checkbox(ABC):
    @abstractmethod
    def render(self) -> str: pass

# Concrete products — Light theme
class LightButton(Button):
    def render(self): return "[ Light Button ]"

class LightCheckbox(Checkbox):
    def render(self): return "☐ Light Checkbox"

# Concrete products — Dark theme
class DarkButton(Button):
    def render(self): return "[ Dark Button ]"

class DarkCheckbox(Checkbox):
    def render(self): return "■ Dark Checkbox"

# Abstract factory
class UIFactory(ABC):
    @abstractmethod
    def create_button(self) -> Button: pass
    @abstractmethod
    def create_checkbox(self) -> Checkbox: pass

# Concrete factories
class LightThemeFactory(UIFactory):
    def create_button(self): return LightButton()
    def create_checkbox(self): return LightCheckbox()

class DarkThemeFactory(UIFactory):
    def create_button(self): return DarkButton()
    def create_checkbox(self): return DarkCheckbox()

def render_ui(factory: UIFactory):
    btn = factory.create_button()
    chk = factory.create_checkbox()
    print(btn.render(), chk.render())

render_ui(LightThemeFactory())   # [ Light Button ]  ☐ Light Checkbox
render_ui(DarkThemeFactory())    # [ Dark Button ]   ■ Dark Checkbox
✅ Real-world use: cross-platform UI toolkits, database driver families, cloud provider SDK wrappers.
4 Builder

🧱 Builder — Construct Complex Objects Step by Step

The Builder pattern separates the construction of a complex object from its representation. It is ideal when an object requires many optional parameters or when construction involves multiple steps that must be performed in a specific order. Python's fluent interface style makes Builders particularly elegant — each method returns self, enabling method chaining.

from dataclasses import dataclass, field
from typing import List, Optional

@dataclass
class QueryConfig:
    table: str
    columns: List[str]
    conditions: List[str]
    order_by: Optional[str]
    limit: Optional[int]

class QueryBuilder:
    def __init__(self, table: str):
        self._table      = table
        self._columns    = ["*"]
        self._conditions = []
        self._order_by   = None
        self._limit      = None

    def select(self, *columns: str) -> "QueryBuilder":
        self._columns = list(columns)
        return self

    def where(self, condition: str) -> "QueryBuilder":
        self._conditions.append(condition)
        return self

    def order_by(self, column: str) -> "QueryBuilder":
        self._order_by = column
        return self

    def limit(self, n: int) -> "QueryBuilder":
        self._limit = n
        return self

    def build(self) -> str:
        sql = f"SELECT {', '.join(self._columns)} FROM {self._table}"
        if self._conditions:
            sql += f" WHERE {' AND '.join(self._conditions)}"
        if self._order_by:
            sql += f" ORDER BY {self._order_by}"
        if self._limit:
            sql += f" LIMIT {self._limit}"
        return sql

# Fluent method chaining
query = (
    QueryBuilder("users")
    .select("id", "name", "email")
    .where("active = 1")
    .where("age >= 18")
    .order_by("name")
    .limit(25)
    .build()
)
print(query)
# SELECT id, name, email FROM users WHERE active = 1 AND age >= 18 ORDER BY name LIMIT 25
✅ Real-world use: ORM query builders (SQLAlchemy, Django ORM), HTTP request builders, test data factories, document builders.
5 Prototype

🧬 Prototype — Clone Objects Instead of Creating from Scratch

The Prototype pattern creates new objects by copying (cloning) an existing object. It is used when object creation is expensive — involving database queries, network calls, or complex computation — and you want to create new instances quickly by cloning a pre-built template. Python's copy module makes this pattern trivially easy.

import copy
from dataclasses import dataclass, field
from typing import List

@dataclass
class GameCharacter:
    name:   str
    hp:     int
    mp:     int
    skills: List[str] = field(default_factory=list)
    level:  int = 1

    def clone(self) -> "GameCharacter":
        """Deep clone — fully independent copy."""
        return copy.deepcopy(self)

    def __str__(self):
        return f"{self.name} Lv{self.level} HP:{self.hp} MP:{self.mp} Skills:{self.skills}"

# Expensive base template (imagine loading from a database)
warrior_template = GameCharacter(
    name="Warrior", hp=500, mp=100,
    skills=["Slash", "Block", "Battle Cry"]
)

# Clone and customise — no expensive re-construction
player1 = warrior_template.clone()
player1.name  = "Aragorn"
player1.level = 15

player2 = warrior_template.clone()
player2.name  = "Boromir"
player2.level = 12
player2.skills.append("Shield Bash")   # only affects player2

print(warrior_template)
print(player1)
print(player2)
💡 Use copy.copy() for a shallow clone (nested objects are shared) and copy.deepcopy() for a fully independent copy. For most prototype use cases, deepcopy is the safe choice.

🔵 Structural Patterns

Compose classes and objects into larger structures while keeping them flexible

Structural patterns deal with how classes and objects are composed to form larger structures. They use inheritance and composition to create new functionality without changing existing code. These patterns are essential for integrating incompatible interfaces, reducing memory overhead, and simplifying complex subsystems.

6 Adapter

🔌 Adapter — Make Incompatible Interfaces Work Together

The Adapter pattern converts the interface of a class into another interface that clients expect. It lets classes work together that couldn't otherwise because of incompatible interfaces. It is the most commonly used structural pattern — every time you wrap a third-party library to match your application's expected interface, you are using the Adapter pattern.

# Legacy payment system — old interface
class LegacyPaymentGateway:
    def make_payment(self, amount_cents: int, card_number: str) -> bool:
        print(f"Legacy: charging {amount_cents} cents to card {card_number[-4:]}")
        return True

# Modern interface your application expects
class PaymentProcessor:
    def charge(self, amount_dollars: float, token: str) -> dict: ...

# Adapter — wraps legacy gateway, exposes modern interface
class LegacyPaymentAdapter(PaymentProcessor):
    def __init__(self, gateway: LegacyPaymentGateway):
        self._gateway = gateway

    def charge(self, amount_dollars: float, token: str) -> dict:
        # Convert dollars → cents, token → card number (mocked)
        cents = int(amount_dollars * 100)
        card  = f"4111111111111{token[-3:]}"   # token mapping
        success = self._gateway.make_payment(cents, card)
        return {"success": success, "amount": amount_dollars, "token": token}

# Client uses modern interface — unaware of legacy internals
gateway   = LegacyPaymentGateway()
processor = LegacyPaymentAdapter(gateway)
result    = processor.charge(29.99, "tok_visa_abc123")
print(result)
✅ Real-world use: wrapping third-party APIs, database driver abstraction, file format converters, unit testing mocks.
7 Bridge

🌉 Bridge — Decouple Abstraction from Implementation

The Bridge pattern separates an abstraction from its implementation so that the two can vary independently. Without Bridge, adding a new abstraction variant multiplies the number of classes needed. With Bridge, you compose abstraction and implementation independently, avoiding a class explosion.

from abc import ABC, abstractmethod

# Implementation side
class Renderer(ABC):
    @abstractmethod
    def render_circle(self, x, y, radius): pass
    @abstractmethod
    def render_square(self, x, y, side): pass

class VectorRenderer(Renderer):
    def render_circle(self, x, y, r):
        print(f"Vector circle at ({x},{y}) r={r}")
    def render_square(self, x, y, s):
        print(f"Vector square at ({x},{y}) side={s}")

class RasterRenderer(Renderer):
    def render_circle(self, x, y, r):
        print(f"Raster circle: drawing {2*r*r} pixels at ({x},{y})")
    def render_square(self, x, y, s):
        print(f"Raster square: drawing {s*s} pixels at ({x},{y})")

# Abstraction side — holds a reference to implementation
class Shape(ABC):
    def __init__(self, renderer: Renderer):
        self.renderer = renderer   # bridge
    @abstractmethod
    def draw(self): pass

class Circle(Shape):
    def __init__(self, renderer, x, y, radius):
        super().__init__(renderer)
        self.x, self.y, self.radius = x, y, radius
    def draw(self):
        self.renderer.render_circle(self.x, self.y, self.radius)

class Square(Shape):
    def __init__(self, renderer, x, y, side):
        super().__init__(renderer)
        self.x, self.y, self.side = x, y, side
    def draw(self):
        self.renderer.render_square(self.x, self.y, self.side)

# Mix any shape with any renderer — 2×2 = 4 combos, only 4 classes
vector = VectorRenderer()
raster = RasterRenderer()

Circle(vector, 0, 0, 5).draw()
Circle(raster, 0, 0, 5).draw()
Square(vector, 1, 1, 10).draw()
Square(raster, 1, 1, 10).draw()
✅ Real-world use: graphics rendering engines, device driver abstraction, cross-platform UI, notification systems with multiple backends.
8 Composite

🌲 Composite — Tree Structures of Objects

The Composite pattern composes objects into tree structures to represent part-whole hierarchies. It lets clients treat individual objects and compositions of objects uniformly. File systems, UI component trees, and organisation charts are all natural composites — a folder contains files and other folders; a UI panel contains buttons and other panels.

from abc import ABC, abstractmethod
from typing import List

class FileSystemItem(ABC):
    def __init__(self, name: str):
        self.name = name

    @abstractmethod
    def size(self) -> int: pass

    @abstractmethod
    def display(self, indent: int = 0): pass

# Leaf node
class File(FileSystemItem):
    def __init__(self, name: str, size_kb: int):
        super().__init__(name)
        self._size = size_kb

    def size(self) -> int: return self._size

    def display(self, indent=0):
        print(" " * indent + f"📄 {self.name} ({self._size}KB)")

# Composite node
class Folder(FileSystemItem):
    def __init__(self, name: str):
        super().__init__(name)
        self._children: List[FileSystemItem] = []

    def add(self, item: FileSystemItem) -> "Folder":
        self._children.append(item)
        return self

    def size(self) -> int:
        return sum(child.size() for child in self._children)

    def display(self, indent=0):
        print(" " * indent + f"📁 {self.name}/ ({self.size()}KB)")
        for child in self._children:
            child.display(indent + 4)

# Build a file tree
root = Folder("project")
src  = Folder("src")
src.add(File("main.py", 12)).add(File("utils.py", 8))

docs = Folder("docs")
docs.add(File("README.md", 4)).add(File("CHANGELOG.md", 6))

root.add(src).add(docs).add(File("requirements.txt", 2))
root.display()
print(f"Total: {root.size()}KB")
✅ Real-world use: file systems, HTML/DOM trees, GUI widget hierarchies, organisation charts, menu systems.
9 Decorator

🎀 Decorator — Add Behaviour Without Changing the Class

The Decorator pattern attaches additional responsibilities to an object dynamically. Decorators provide a flexible alternative to subclassing for extending functionality. Python has built-in syntactic support for this pattern with the @ decorator syntax — though the GoF Decorator wraps objects rather than functions. Both are conceptually the same: wrap the original, add behaviour, delegate to original.

from abc import ABC, abstractmethod

# Component interface
class Coffee(ABC):
    @abstractmethod
    def cost(self) -> float: pass
    @abstractmethod
    def description(self) -> str: pass

# Concrete component
class SimpleCoffee(Coffee):
    def cost(self): return 1.00
    def description(self): return "Coffee"

# Base decorator
class CoffeeDecorator(Coffee):
    def __init__(self, coffee: Coffee):
        self._coffee = coffee
    def cost(self): return self._coffee.cost()
    def description(self): return self._coffee.description()

# Concrete decorators — each adds cost and description
class Milk(CoffeeDecorator):
    def cost(self): return super().cost() + 0.25
    def description(self): return super().description() + ", Milk"

class Vanilla(CoffeeDecorator):
    def cost(self): return super().cost() + 0.50
    def description(self): return super().description() + ", Vanilla"

class WhippedCream(CoffeeDecorator):
    def cost(self): return super().cost() + 0.75
    def description(self): return super().description() + ", Whipped Cream"

# Stack decorators dynamically
coffee = SimpleCoffee()
coffee = Milk(coffee)
coffee = Vanilla(coffee)
coffee = WhippedCream(coffee)

print(coffee.description())   # Coffee, Milk, Vanilla, Whipped Cream
print(f"${coffee.cost():.2f}")  # $2.50
✅ Real-world use: Python's own @property, @lru_cache, @dataclass. Flask routes (@app.route), Django middleware, logging wrappers.
10 Facade

🏛️ Facade — Simple Interface to a Complex Subsystem

The Facade pattern provides a simplified interface to a complex body of code — a library, framework, or complex set of classes. It does not encapsulate the subsystem but provides a convenient entry point, reducing the learning curve and coupling for client code. Django's render() function is a classic facade — it hides template loading, context processing, and HTTP response construction behind one call.

# Complex subsystem classes
class VideoDecoder:
    def decode(self, file: str):
        print(f"  Decoding video: {file}")

class AudioMixer:
    def mix(self, tracks: list):
        print(f"  Mixing {len(tracks)} audio tracks")

class SubtitleParser:
    def parse(self, file: str):
        print(f"  Parsing subtitles: {file}")

class VideoEncoder:
    def encode(self, output: str, codec: str):
        print(f"  Encoding to {output} using {codec}")

class ThumbnailGenerator:
    def generate(self, file: str):
        print(f"  Generating thumbnail for {file}")

# Facade — one simple method hides all of the above
class VideoConverterFacade:
    def __init__(self):
        self._decoder   = VideoDecoder()
        self._mixer     = AudioMixer()
        self._subtitles = SubtitleParser()
        self._encoder   = VideoEncoder()
        self._thumbs    = ThumbnailGenerator()

    def convert(self, input_file: str, output_file: str, codec="H.264"):
        print(f"Converting {input_file} → {output_file}")
        self._decoder.decode(input_file)
        self._mixer.mix(["track1", "track2"])
        self._subtitles.parse(input_file.replace(".mp4", ".srt"))
        self._encoder.encode(output_file, codec)
        self._thumbs.generate(output_file)
        print("Done!")

# Client only talks to the facade
converter = VideoConverterFacade()
converter.convert("lecture.mp4", "lecture_web.mp4")
✅ Real-world use: Django render(), requests.get() (hides sockets, HTTP, encoding), pandas.read_csv(), any SDK wrapper.
11 Flyweight

🪶 Flyweight — Share State to Support Many Fine-Grained Objects

The Flyweight pattern minimises memory usage by sharing as much data as possible with similar objects. It separates intrinsic state (shared, context-independent) from extrinsic state (unique per object). Essential when you need to create thousands or millions of objects that share significant common data — characters in a text editor, particles in a game engine, icons in a file manager.

import sys
from dataclasses import dataclass

# Flyweight — intrinsic (shared) state
class CharacterStyle:
    """Shared style data — font, size, bold, italic."""
    _cache: dict = {}

    def __new__(cls, font: str, size: int, bold: bool, italic: bool):
        key = (font, size, bold, italic)
        if key not in cls._cache:
            instance = super().__new__(cls)
            instance.font   = font
            instance.size   = size
            instance.bold   = bold
            instance.italic = italic
            cls._cache[key] = instance
        return cls._cache[key]

    def __repr__(self):
        return f"Style({self.font} {self.size}pt {'B' if self.bold else ''}{'I' if self.italic else ''})"

# Context — extrinsic (unique) state: character + position
@dataclass
class Character:
    char:  str
    x:     int
    y:     int
    style: CharacterStyle   # shared flyweight

# Build 1000 characters — only unique styles are stored
chars = []
for i, ch in enumerate("Hello, World! " * 50):
    style = CharacterStyle("Arial", 12, i % 5 == 0, i % 7 == 0)
    chars.append(Character(ch, i * 8, 0, style))

print(f"Characters created : {len(chars)}")
print(f"Unique styles cached: {len(CharacterStyle._cache)}")
print(f"First char style   : {chars[0].style}")
✅ Real-world use: text rendering engines, game particle systems, icon caches in file explorers, connection pools.
12 Proxy

🛡️ Proxy — Control Access to Another Object

The Proxy pattern provides a surrogate or placeholder for another object to control access to it. There are four main proxy types: Virtual (lazy initialisation), Remote (local representation of a remote object), Protection (access control), and Caching (memoised results). Python's __getattr__ magic method makes proxies especially clean to write.

import time
from functools import wraps

# Real subject — expensive operation
class SlowDatabaseQuery:
    def fetch(self, query: str) -> list:
        time.sleep(0.01)   # simulate DB latency
        return [f"row_{i}_for_{query}" for i in range(3)]

# Caching Proxy — same interface, adds transparent caching
class CachingQueryProxy:
    def __init__(self):
        self._db    = SlowDatabaseQuery()
        self._cache = {}

    def fetch(self, query: str) -> list:
        if query not in self._cache:
            print(f"  Cache MISS — querying DB for '{query}'")
            self._cache[query] = self._db.fetch(query)
        else:
            print(f"  Cache HIT  — returning cached '{query}'")
        return self._cache[query]

# Protection Proxy — adds authorisation check
class SecureQueryProxy:
    def __init__(self, user_role: str):
        self._db   = CachingQueryProxy()
        self._role = user_role

    def fetch(self, query: str) -> list:
        if "DROP" in query.upper() and self._role != "admin":
            raise PermissionError(f"Role '{self._role}' cannot run destructive queries")
        return self._db.fetch(query)

db = SecureQueryProxy("reader")
print(db.fetch("SELECT users"))
print(db.fetch("SELECT users"))   # cache hit

try:
    db.fetch("DROP TABLE users")
except PermissionError as e:
    print(f"Blocked: {e}")
✅ Real-world use: ORM lazy loading (SQLAlchemy deferred columns), API rate limiting, access control layers, logging proxies, CDN caching.

🟣 Behavioral Patterns

Define how objects communicate and distribute responsibility

Behavioral patterns are concerned with algorithms and the assignment of responsibilities between objects. They describe patterns of communication between objects and classes. The 11 behavioral patterns cover everything from event handling and undo systems to language interpreters and traversal algorithms.

13 Chain of Responsibility

⛓️ Chain of Responsibility — Pass Requests Along a Chain

The Chain of Responsibility pattern passes a request along a chain of handlers. Each handler decides to process the request or pass it to the next handler in the chain. This decouples the sender from the receiver and allows multiple objects a chance to handle the request. Django middleware and Python's logging system both use this pattern directly.

from abc import ABC, abstractmethod
from typing import Optional

class SupportHandler(ABC):
    def __init__(self):
        self._next: Optional[SupportHandler] = None

    def set_next(self, handler: "SupportHandler") -> "SupportHandler":
        self._next = handler
        return handler   # allows chaining: a.set_next(b).set_next(c)

    def handle(self, severity: int, message: str):
        if self._next:
            self._next.handle(severity, message)

class Level1Support(SupportHandler):
    def handle(self, severity: int, message: str):
        if severity == 1:
            print(f"L1 Support resolved: {message}")
        else:
            print(f"L1 escalating severity {severity}…")
            super().handle(severity, message)

class Level2Support(SupportHandler):
    def handle(self, severity: int, message: str):
        if severity == 2:
            print(f"L2 Support resolved: {message}")
        else:
            print(f"L2 escalating severity {severity}…")
            super().handle(severity, message)

class EngineeringTeam(SupportHandler):
    def handle(self, severity: int, message: str):
        print(f"Engineering team handling severity {severity}: {message}")

# Build the chain
l1 = Level1Support()
l2 = Level2Support()
eng = EngineeringTeam()
l1.set_next(l2).set_next(eng)

for sev, msg in [(1, "Password reset"), (2, "Account locked"), (3, "Data corruption")]:
    l1.handle(sev, msg)
    print()
✅ Real-world use: Django middleware pipeline, Python logging handlers, request validation chains, event bubbling in UI.
14 Command

📋 Command — Encapsulate Requests as Objects

The Command pattern encapsulates a request as an object, letting you parameterise methods with different requests, queue or log requests, and support undoable operations. It is the pattern behind undo/redo systems, transaction managers, task queues, and macro recording — any situation where an action needs to be stored, scheduled, or reversed.

from abc import ABC, abstractmethod
from collections import deque

class Command(ABC):
    @abstractmethod
    def execute(self): pass
    @abstractmethod
    def undo(self): pass

# Receiver
class TextDocument:
    def __init__(self): self.text = ""
    def insert(self, text: str): self.text += text
    def delete(self, n: int):    self.text = self.text[:-n]
    def __str__(self): return f"Doc: '{self.text}'"

# Concrete commands
class InsertCommand(Command):
    def __init__(self, doc: TextDocument, text: str):
        self._doc, self._text = doc, text
    def execute(self): self._doc.insert(self._text)
    def undo(self):    self._doc.delete(len(self._text))

# Invoker — manages history for undo/redo
class Editor:
    def __init__(self):
        self._history: deque = deque()

    def execute(self, command: Command):
        command.execute()
        self._history.append(command)

    def undo(self):
        if self._history:
            self._history.pop().undo()

doc    = TextDocument()
editor = Editor()

editor.execute(InsertCommand(doc, "Hello"))
editor.execute(InsertCommand(doc, ", World"))
editor.execute(InsertCommand(doc, "!"))
print(doc)          # Doc: 'Hello, World!'

editor.undo()
print(doc)          # Doc: 'Hello, World'

editor.undo()
print(doc)          # Doc: 'Hello'
✅ Real-world use: text editor undo/redo, database transaction rollback, Celery task queues, GUI button actions, macro recorders.
15 Iterator

🔄 Iterator — Sequential Access Without Exposing Structure

The Iterator pattern provides a way to sequentially access elements of a collection without exposing its underlying structure. Python has this pattern built directly into the language — the iterator protocol (__iter__ and __next__) is how for loops work. Every object that supports for x in obj implements the Iterator pattern.

from typing import Iterator

# Custom collection implementing the iterator protocol
class NumberRange:
    """An inclusive range that iterates in steps."""
    def __init__(self, start: int, stop: int, step: int = 1):
        self.start = start
        self.stop  = stop
        self.step  = step

    def __iter__(self) -> Iterator[int]:
        current = self.start
        while current <= self.stop:
            yield current
            current += self.step

    def __contains__(self, item: int) -> bool:
        return self.start <= item <= self.stop and (item - self.start) % self.step == 0

    def __len__(self) -> int:
        return (self.stop - self.start) // self.step + 1

# Works with all Python iteration protocols
evens = NumberRange(2, 20, 2)

print("Even numbers:", list(evens))
print("Count:", len(evens))
print("14 in range:", 14 in evens)
print("Sum:", sum(evens))
print("First 3:", [x for x in evens if x < 8])

# Tree iterator — pre-order traversal via generator
class TreeNode:
    def __init__(self, val, left=None, right=None):
        self.val, self.left, self.right = val, left, right

    def __iter__(self):   # pre-order
        yield self.val
        if self.left:  yield from self.left
        if self.right: yield from self.right

tree = TreeNode(1, TreeNode(2, TreeNode(4), TreeNode(5)), TreeNode(3))
print("Tree pre-order:", list(tree))
✅ Real-world use: Python's for loop, range(), file iteration, Django QuerySets, SQLAlchemy result sets.
16 Mediator

🕊️ Mediator — Centralise Complex Communications

The Mediator pattern defines an object that encapsulates how a set of objects interact. It promotes loose coupling by keeping objects from referring to each other explicitly. Instead of N objects communicating with each other in N² ways, they all communicate through one mediator. Django's signal system and an air traffic control tower are both classic mediators.

from typing import List

class ChatRoom:
    """Mediator — manages all chat participants."""
    def __init__(self, name: str):
        self.name = name
        self._members: List["User"] = []

    def join(self, user: "User"):
        self._members.append(user)
        user.room = self
        self.broadcast(f"{user.name} joined the room.", sender=None)

    def leave(self, user: "User"):
        self._members.remove(user)
        self.broadcast(f"{user.name} left the room.", sender=None)

    def broadcast(self, message: str, sender: "User"):
        for member in self._members:
            if member is not sender:
                member.receive(self.name, message)

    def direct(self, to_name: str, message: str, sender: "User"):
        for m in self._members:
            if m.name == to_name:
                m.receive(f"DM from {sender.name}", message)
                return

class User:
    def __init__(self, name: str):
        self.name = name
        self.room: ChatRoom = None

    def say(self, message: str):
        print(f"{self.name}: {message}")
        self.room.broadcast(f"{self.name}: {message}", sender=self)

    def dm(self, to: str, message: str):
        self.room.direct(to, message, self)

    def receive(self, source: str, message: str):
        print(f"  [{self.name} receives from {source}] {message}")

room = ChatRoom("python-help")
alice, bob, carol = User("Alice"), User("Bob"), User("Carol")
for u in [alice, bob, carol]: room.join(u)
alice.say("Hello everyone!")
bob.dm("Carol", "Hey, check your PR")
✅ Real-world use: Django signals, event buses, MVC controllers, chat systems, air traffic control simulations.
17 Memento

📸 Memento — Capture and Restore Object State

The Memento pattern captures and externalises an object's internal state so it can be restored later, without violating encapsulation. The three roles are: Originator (the object whose state is saved), Memento (the snapshot), and Caretaker (stores mementos, knows when to save and restore). This is the pattern behind save games, undo history, and database transaction snapshots.

from dataclasses import dataclass
from typing import List
import copy

# Memento — immutable snapshot
@dataclass(frozen=True)
class EditorMemento:
    content: str
    cursor:  int
    label:   str = ""

# Originator — the object being saved
class CodeEditor:
    def __init__(self):
        self._content = ""
        self._cursor  = 0

    def type(self, text: str):
        self._content += text
        self._cursor   = len(self._content)

    def save(self, label: str = "") -> EditorMemento:
        return EditorMemento(self._content, self._cursor, label)

    def restore(self, memento: EditorMemento):
        self._content = memento.content
        self._cursor  = memento.cursor

    def __str__(self):
        return f"Content: '{self._content}' | Cursor: {self._cursor}"

# Caretaker — manages the history stack
class History:
    def __init__(self): self._stack: List[EditorMemento] = []
    def push(self, m): self._stack.append(m)
    def pop(self):    return self._stack.pop() if self._stack else None
    def __len__(self): return len(self._stack)

editor  = CodeEditor()
history = History()

editor.type("def hello():");        history.push(editor.save("v1"))
editor.type("\n    pass");           history.push(editor.save("v2"))
editor.type("\n    print('hi')");   history.push(editor.save("v3"))

print("Current:", editor)

editor.restore(history.pop())   # back to v3 (itself)
editor.restore(history.pop())   # back to v2
print("After 2x undo:", editor)
✅ Real-world use: text editor undo, game save states, database transaction rollback, browser history, version control snapshots.
18 Observer

👁️ Observer — Subscribe to Events Automatically

The Observer pattern defines a one-to-many dependency between objects so that when one object (the Subject) changes state, all its dependents (Observers) are notified and updated automatically. This is the foundation of event-driven programming, reactive systems, and the Model-View-Controller architecture. Django signals, Python's asyncio event system, and every GUI framework use this pattern.

from abc import ABC, abstractmethod
from typing import List, Dict, Any
from dataclasses import dataclass

class Observer(ABC):
    @abstractmethod
    def update(self, event: str, data: Dict[str, Any]): pass

class EventBus:
    """Subject — maintains and notifies observers."""
    def __init__(self):
        self._listeners: Dict[str, List[Observer]] = {}

    def subscribe(self, event: str, observer: Observer):
        self._listeners.setdefault(event, []).append(observer)

    def unsubscribe(self, event: str, observer: Observer):
        if event in self._listeners:
            self._listeners[event].remove(observer)

    def publish(self, event: str, data: Dict[str, Any] = {}):
        print(f"\n[EventBus] '{event}' → {len(self._listeners.get(event, []))} listeners")
        for obs in self._listeners.get(event, []):
            obs.update(event, data)

# Concrete observers
class EmailService(Observer):
    def update(self, event, data):
        print(f"  EmailService: sending '{event}' email to {data.get('email','?')}")

class AnalyticsService(Observer):
    def update(self, event, data):
        print(f"  Analytics: tracking '{event}' — {data}")

class AuditLogger(Observer):
    def update(self, event, data):
        print(f"  AuditLog: [{event}] {data}")

bus       = EventBus()
email     = EmailService()
analytics = AnalyticsService()
audit     = AuditLogger()

bus.subscribe("user.registered", email)
bus.subscribe("user.registered", analytics)
bus.subscribe("user.registered", audit)
bus.subscribe("user.login",      analytics)
bus.subscribe("user.login",      audit)

bus.publish("user.registered", {"email": "ana@example.com", "id": 42})
bus.publish("user.login",      {"user_id": 42, "ip": "192.168.1.1"})
✅ Real-world use: Django signals (post_save, pre_delete), GUI event listeners, reactive state management (Redux, MobX), pub/sub messaging systems.
19 State

🔁 State — Change Behaviour When State Changes

The State pattern allows an object to alter its behaviour when its internal state changes. The object will appear to change its class. Instead of a massive if/elif chain checking a status field, the State pattern encapsulates each state in its own class. Python 3.10+ match statements are a lighter alternative for simple state machines, but the full pattern is needed when states have complex behaviour.

from abc import ABC, abstractmethod

class OrderState(ABC):
    @abstractmethod
    def pay(self, order) -> str: pass
    @abstractmethod
    def ship(self, order) -> str: pass
    @abstractmethod
    def cancel(self, order) -> str: pass
    def __str__(self): return self.__class__.__name__

class PendingState(OrderState):
    def pay(self, order):
        order.state = PaidState()
        return "Payment accepted → Paid"
    def ship(self, order): return "Cannot ship unpaid order"
    def cancel(self, order):
        order.state = CancelledState()
        return "Order cancelled"

class PaidState(OrderState):
    def pay(self, order): return "Already paid"
    def ship(self, order):
        order.state = ShippedState()
        return "Order shipped → Shipped"
    def cancel(self, order):
        order.state = CancelledState()
        return "Order refunded and cancelled"

class ShippedState(OrderState):
    def pay(self, order): return "Already shipped and paid"
    def ship(self, order): return "Already shipped"
    def cancel(self, order): return "Cannot cancel shipped order"

class CancelledState(OrderState):
    def pay(self, order): return "Cannot pay cancelled order"
    def ship(self, order): return "Cannot ship cancelled order"
    def cancel(self, order): return "Already cancelled"

class Order:
    def __init__(self): self.state = PendingState()
    def pay(self):    print(f"[{self.state}] pay()    → {self.state.pay(self)}")
    def ship(self):   print(f"[{self.state}] ship()   → {self.state.ship(self)}")
    def cancel(self): print(f"[{self.state}] cancel() → {self.state.cancel(self)}")

order = Order()
order.ship()    # blocked — not paid yet
order.pay()     # → Paid
order.ship()    # → Shipped
order.cancel()  # blocked — already shipped
✅ Real-world use: order management systems, TCP connection states, game character states, vending machines, document approval workflows.
20 Strategy

♟️ Strategy — Swap Algorithms at Runtime

The Strategy pattern defines a family of algorithms, encapsulates each one, and makes them interchangeable. It lets the algorithm vary independently from the clients that use it. In Python, strategies are often implemented as plain functions or lambdas passed as arguments — Python's first-class functions make the classic class-based strategy leaner. Python's built-in sorted(key=...) is the most famous strategy in the language.

from abc import ABC, abstractmethod
from typing import List
import re

class SortStrategy(ABC):
    @abstractmethod
    def sort(self, data: list) -> list: pass

class BubbleSort(SortStrategy):
    def sort(self, data: list) -> list:
        arr = data[:]
        n = len(arr)
        for i in range(n):
            for j in range(n - i - 1):
                if arr[j] > arr[j+1]:
                    arr[j], arr[j+1] = arr[j+1], arr[j]
        print(f"  BubbleSort used")
        return arr

class QuickSort(SortStrategy):
    def sort(self, data: list) -> list:
        if len(data) <= 1: return data
        pivot = data[len(data) // 2]
        left  = [x for x in data if x < pivot]
        mid   = [x for x in data if x == pivot]
        right = [x for x in data if x > pivot]
        result = self.sort(left) + mid + self.sort(right)
        print(f"  QuickSort used")
        return result

class BuiltinSort(SortStrategy):
    def sort(self, data: list) -> list:
        print(f"  Python built-in Timsort used")
        return sorted(data)

# Context — uses whichever strategy is injected
class Sorter:
    def __init__(self, strategy: SortStrategy):
        self._strategy = strategy

    def set_strategy(self, strategy: SortStrategy):
        self._strategy = strategy

    def sort(self, data: list) -> list:
        return self._strategy.sort(data)

data = [64, 34, 25, 12, 22, 11, 90]

sorter = Sorter(BubbleSort())
print(sorter.sort(data))

sorter.set_strategy(QuickSort())
print(sorter.sort(data))

sorter.set_strategy(BuiltinSort())
print(sorter.sort(data))
💡 In Python you can pass functions directly as strategies: sorted(data, key=lambda x: -x) is Strategy in action. The class-based form is needed when the strategy holds state or implements multiple related methods.
21 Template Method

📐 Template Method — Define the Skeleton, Fill in the Details

The Template Method defines the skeleton of an algorithm in a base class, deferring some steps to subclasses. It lets subclasses redefine certain steps without changing the overall algorithm's structure. Django's class-based views are the most famous Template Method implementation in Python — View.dispatch() is the template, and get(), post() are the hooks you override.

from abc import ABC, abstractmethod
import csv, json
from io import StringIO

class DataImporter(ABC):
    """Template method defines the import algorithm."""

    def import_data(self, raw: str) -> list:
        # Template method — fixed skeleton
        data   = self.parse(raw)
        data   = self.validate(data)
        data   = self.transform(data)
        self.save(data)
        return data

    @abstractmethod
    def parse(self, raw: str) -> list: pass

    def validate(self, data: list) -> list:
        # Default hook — subclasses may override
        return [row for row in data if row]

    def transform(self, data: list) -> list:
        return data   # default: no-op

    def save(self, data: list):
        print(f"  Saved {len(data)} records")

class CSVImporter(DataImporter):
    def parse(self, raw: str) -> list:
        reader = csv.DictReader(StringIO(raw))
        return list(reader)

    def transform(self, data: list) -> list:
        for row in data:
            row["name"] = row["name"].strip().title()
        return data

class JSONImporter(DataImporter):
    def parse(self, raw: str) -> list:
        return json.loads(raw)

    def validate(self, data: list) -> list:
        return [r for r in data if r.get("name")]

csv_data = "name,age\n  alice  ,30\nbob,25\n"
json_data = '[{"name":"Carol","age":28},{"name":"","age":0},{"name":"Dave","age":35}]'

print("CSV import:")
result = CSVImporter().import_data(csv_data)
print(result)

print("\nJSON import:")
result = JSONImporter().import_data(json_data)
print(result)
✅ Real-world use: Django class-based views, data import pipelines, test frameworks (setUp/tearDown), document generation, report builders.
22 Visitor

🚶 Visitor — Add Operations Without Changing Classes

The Visitor pattern lets you add further operations to objects without modifying them. You define a visitor class with a method for each element type, then the elements "accept" the visitor and call the appropriate method. It is the right pattern when you have a stable class hierarchy and need to add many unrelated operations over time, keeping them cleanly separated.

from abc import ABC, abstractmethod
from typing import List

# Element interface
class DocumentNode(ABC):
    @abstractmethod
    def accept(self, visitor: "DocumentVisitor"): pass

# Concrete elements
class Heading(DocumentNode):
    def __init__(self, text: str, level: int = 1):
        self.text, self.level = text, level
    def accept(self, v): v.visit_heading(self)

class Paragraph(DocumentNode):
    def __init__(self, text: str): self.text = text
    def accept(self, v): v.visit_paragraph(self)

class CodeBlock(DocumentNode):
    def __init__(self, code: str, lang: str = "python"):
        self.code, self.lang = code, lang
    def accept(self, v): v.visit_code_block(self)

# Visitor interface
class DocumentVisitor(ABC):
    @abstractmethod
    def visit_heading(self, h: Heading): pass
    @abstractmethod
    def visit_paragraph(self, p: Paragraph): pass
    @abstractmethod
    def visit_code_block(self, c: CodeBlock): pass

# Concrete visitor 1 — HTML export
class HTMLExporter(DocumentVisitor):
    def __init__(self): self.output = []
    def visit_heading(self, h):
        self.output.append(f"{h.text}")
    def visit_paragraph(self, p):
        self.output.append(f"
{p.text}
")
    def visit_code_block(self, c):
        self.output.append(f'
{c.code}
')
    def result(self): return "\n".join(self.output)

# Concrete visitor 2 — word count
class WordCounter(DocumentVisitor):
    def __init__(self): self.count = 0
    def visit_heading(self, h):   self.count += len(h.text.split())
    def visit_paragraph(self, p): self.count += len(p.text.split())
    def visit_code_block(self, c): pass   # skip code

doc: List[DocumentNode] = [
    Heading("Python Design Patterns", 1),
    Paragraph("Design patterns are reusable solutions to common problems."),
    CodeBlock("print('hello')"),
    Paragraph("The Visitor pattern separates operations from data structures."),
]

html    = HTMLExporter()
counter = WordCounter()
for node in doc:
    node.accept(html)
    node.accept(counter)

print(html.result())
print(f"\nWord count: {counter.count}")
✅ Real-world use: document export (HTML, PDF, Markdown), AST traversal in compilers, code linters, tax calculation systems.
23 Interpreter

📖 Interpreter — A Grammar for a Language

The Interpreter pattern defines a representation for a grammar of a language and provides an interpreter to deal with that grammar. Each grammar rule becomes a class. It is the most specialised GoF pattern and is used when you need a simple scripting language, query language, or expression evaluator. Python's ast module and SQL query parsers are full-scale interpreters.

from abc import ABC, abstractmethod

# Abstract expression
class Expression(ABC):
    @abstractmethod
    def interpret(self, context: dict) -> int: pass

# Terminal expressions
class Number(Expression):
    def __init__(self, value: int): self.value = value
    def interpret(self, ctx): return self.value

class Variable(Expression):
    def __init__(self, name: str): self.name = name
    def interpret(self, ctx): return ctx.get(self.name, 0)

# Non-terminal expressions
class Add(Expression):
    def __init__(self, left, right): self.left, self.right = left, right
    def interpret(self, ctx):
        return self.left.interpret(ctx) + self.right.interpret(ctx)

class Subtract(Expression):
    def __init__(self, left, right): self.left, self.right = left, right
    def interpret(self, ctx):
        return self.left.interpret(ctx) - self.right.interpret(ctx)

class Multiply(Expression):
    def __init__(self, left, right): self.left, self.right = left, right
    def interpret(self, ctx):
        return self.left.interpret(ctx) * self.right.interpret(ctx)

# Build expression tree for: (x + 5) * (y - 3)
context = {"x": 10, "y": 8}

expr = Multiply(
    Add(Variable("x"), Number(5)),
    Subtract(Variable("y"), Number(3))
)

result = expr.interpret(context)
print(f"(x + 5) * (y - 3) = ({context['x']} + 5) * ({context['y']} - 3) = {result}")
# (x + 5) * (y - 3) = (10 + 5) * (8 - 3) = 75

# Change variables and re-evaluate — same expression tree
context = {"x": 2, "y": 7}
print(f"With x=2, y=7: {expr.interpret(context)}")   # (2+5)*(7-3) = 28
✅ Real-world use: expression evaluators, SQL parsers, regex engines, configuration DSLs, template engines, rule engines.

❓ Design Patterns FAQ

Common questions about applying GoF patterns in Python

Do I need to memorise all 23 patterns?

+

No. Focus on the 8–10 most common ones first: Singleton, Factory Method, Builder, Adapter, Facade, Decorator, Observer, Strategy, Template Method, and Command. These appear in virtually every large Python codebase. The remaining patterns are worth knowing conceptually so you can recognise them when you encounter them and apply them when the problem fits.

Python already has many patterns built in — which ones are redundant?

+

Iterator is built into the language via __iter__/__next__ and yield. Decorator is a first-class language feature with @ syntax. Singleton is naturally implemented by modules. Prototype is trivially done with copy.deepcopy(). Strategy collapses to passing a function as an argument. Python's expressiveness means the class-heavy GoF solutions often reduce to a few lines, but the underlying pattern and its intent remain unchanged.

When should I NOT use a design pattern?

+

When the problem doesn't require it. Over-engineering is a real risk — applying a pattern to a problem it was not designed to solve adds complexity without benefit. The rule: start with the simplest working solution. Introduce a pattern only when you feel the pain the pattern was designed to solve — complexity, rigidity, or fragility in that specific area of code.

What is the difference between Strategy and Template Method?

+

Both vary parts of an algorithm. Template Method uses inheritance — the base class defines the skeleton and subclasses override specific steps. Strategy uses composition — the algorithm is encapsulated in a separate object that is injected. Strategy is more flexible (swap at runtime, pass as a function), while Template Method is simpler for fixed hierarchies. Prefer Strategy in Python because composition is generally more Pythonic than deep inheritance.

How are design patterns different from algorithms?

+

Algorithms solve computational problems (sorting, searching, graph traversal) with a defined set of steps. Design patterns solve software architecture problems (object creation, composition, communication) with a structural blueprint. An algorithm is like a cooking recipe — specific steps, specific output. A pattern is like a kitchen layout — it describes how things relate, not the exact sequence of operations.

Which patterns appear most in Python interview questions?

+

Singleton (understand the pitfalls), Observer (event systems), Strategy (dependency injection), Decorator (Python's own @ syntax), and Factory Method appear most frequently. Interviewers often ask you to implement a simple version from scratch or to identify which pattern is being used in a given code snippet. Understanding the intent of each pattern matters more than memorising their class diagrams.

Are there patterns beyond the original GoF 23?

+

Yes. The GoF book was written in 1994 and focused on OOP. Modern patterns include: Repository (data access layer), Dependency Injection (inversion of control), Event Sourcing (store state as events), CQRS (separate read/write models), Circuit Breaker (microservices resilience), and many more from enterprise and distributed systems contexts. The GoF patterns remain the foundational vocabulary that all of these build upon.
