VINO PH: A Python Learning Journey

🍷 Welcome to VINO PH

This page presents a complete, production-quality Python program for a Philippine wine shop ordering system called VINO PH. It demonstrates core Python concepts: nested dictionaries as a mini-database, f-strings with format specifiers for clean output, while True loops with try/except for robust input validation, arithmetic computation, and a senior citizen discount feature.

Every concept is introduced in the context of a real, working program — so you see why each design decision was made. By the end, you'll understand not just the code, but the thinking behind the code.

VINO PH is intentionally small enough to read in one sitting, yet it packs in almost every pattern a beginner will meet again and again in bigger, real-world systems: a data layer (the wines dictionary), a presentation layer (the formatted menu and receipt), a validation layer (the two while True loops), and a business-rules layer (the senior discount calculation). Professional applications — a point-of-sale system, an e-commerce checkout, a booking engine — are built from the exact same four layers, just at a much larger scale. Learning to recognize this shape in a 70-line script is what makes it possible to recognize it later inside a 70,000-line codebase.

Complete Python Script

wine_shop.py 
# ── VINO PH — Wine Shop (Philippines) ──
# Python 3 · Interactive · Poetic Bytes

wines = {
    1:  {"name": "Yellow Tail Cabernet Sauvignon (Australia)",          "price": 799},
    2:  {"name": "Jacob's Creek Shiraz (Australia)",                   "price": 899},
    3:  {"name": "Concha y Toro Frontera Cabernet Sauvignon (Chile)",   "price": 599},
    4:  {"name": "Beringer Founders' Estate Merlot (USA)",             "price": 1299},
    5:  {"name": "Robert Mondavi Private Selection Chardonnay (USA)",  "price": 1499},
    6:  {"name": "Gallo Family Vineyards Merlot (USA)",               "price": 649},
    7:  {"name": "Sutter Home Cabernet Sauvignon (USA)",              "price": 699},
    8:  {"name": "Lindemans Bin 65 Chardonnay (Australia)",           "price": 899},
    9:  {"name": "Penfolds Koonunga Hill Shiraz Cabernet (Australia)", "price": 1199},
    10: {"name": "Wolf Blass Yellow Label Cabernet Sauvignon",        "price": 1099},
    11: {"name": "Hardys Stamp Shiraz Cabernet (Australia)",          "price": 799},
    12: {"name": "Oxford Landing Cabernet Sauvignon (Australia)",     "price": 749},
    13: {"name": "Woodbridge by Robert Mondavi Cabernet (USA)",      "price": 999},
    14: {"name": "Kim Crawford Sauvignon Blanc (New Zealand)",       "price": 1699},
    15: {"name": "Oyster Bay Sauvignon Blanc (New Zealand)",         "price": 1499},
}

# ── Welcome banner ──
print("==================================================")
print("     WELCOME TO VINO PH - Wine Shop (Philippines)")
print("     Available Wines (Prices in PHP per bottle)")
print("==================================================")

# ── Display the menu via for loop ──
for num, wine in wines.items():
    print(f"{num:2d}. {wine['name']:50} ₱{wine['price']:,.0f}")

# ── First input loop: wine selection ──
while True:
    try:
        choice = int(input("\nEnter the number of the wine you want to buy (1-15): "))
        if choice in wines:
            selected_wine = wines[choice]
            print(f"\nYou selected: {selected_wine['name']}")
            print(f"Price per bottle: ₱{selected_wine['price']:,}")
            break
        else:
            print("Invalid number! Please choose between 1 and 15.")
    except ValueError:
        print("Please enter a valid number!")

# ── Second input loop: quantity ──
while True:
    try:
        quantity = int(input("\nEnter quantity (bottles): "))
        if quantity > 0:
            break
        else:
            print("Quantity must be at least 1!")
    except ValueError:
        print("Please enter a valid number!")

# ── Senior Citizen Discount ──
is_senior = input("\nAre you a senior citizen? (yes/no): ").strip().lower()
discount_rate = 0.10 if is_senior == "yes" else 0.00

# ── Computation ──
total_cost = selected_wine['price'] * quantity
discount_amount = total_cost * discount_rate
final_cost = total_cost - discount_amount

# ── Print receipt ──
print("\n" + "=" * 50)
print("           VINO PH - RECEIPT")
print("=" * 50)
print(f"Wine          : {selected_wine['name']}")
print(f"Price per bottle: ₱{selected_wine['price']:,.0f}")
print(f"Quantity      : {quantity}")
print(f"Subtotal      : ₱{total_cost:,.0f}")
if discount_amount > 0:
    print(f"Senior Discount (10%): -₱{discount_amount:,.0f}")
print(f"Total Cost    : ₱{final_cost:,.0f}")
print("=" * 50)
print("Thank you for shopping at VINO PH! 🍷")
03 — Logical Breakdown

Step-by-Step Explanation

01

The Wine Inventory — Dictionary Data Structure

Data Structure ▾ 
wines = {
    1: {"name": "Yellow Tail Cabernet Sauvignon", "price": 799},
    2: {"name": "Jacob's Creek Shiraz", "price": 899},
    ... (15 items total)
}
Why was this code used?
A dictionary (dict) is the ideal Python data structure when you need to look up values by a key. Here, each wine number (1–15) is a key, and each value is another dictionary containing the wine's name and price. This creates a nested dictionary — a dictionary inside a dictionary.
What is its significance?
It acts as the program's database — the single source of truth for all product information. Instead of scattering 15 wine names and prices across dozens of variables, everything is organized in one place.
How does it operate?
When the user types a number (e.g. 5), the program uses it as a key: wines[5] instantly returns {"name": "Robert Mondavi...", "price": 1499}. No looping required — dictionary lookup is O(1) — instant access.
How does this affect the program's output?
Every later line of output — the menu, the confirmation line, the receipt — reads directly from this dictionary. Change a price here and it ripples correctly through the whole program automatically, because nothing else hard-codes the numbers.
dict { } nested dict key-value pairs O(1) lookup
02

Welcome Banner — print() Statements

Output / UI ▾

print() is Python's built-in function for displaying text output to the terminal. Multiple calls construct a visual banner that greets the user and sets context.
What is its significance?
This is the program's user interface header. The rows of = signs create a visual border — a classic terminal UI technique.
How does this affect the program's output?
It is the very first thing the user sees. A clear banner signals "this program is organized and trustworthy" before a single wine is even listed — first impressions matter as much in a terminal as they do in a GUI.
print() string literals sequential execution
03

Displaying the Menu — for Loop with f-strings

Iteration / Formatting ▾ 
for num, wine in wines.items():
    print(f"{num:2d}. {wine['name']:50} ₱{wine['price']:,.0f}")
Why was this code used?
A for loop with .items() efficiently iterates over every wine in the dictionary without repeating code.
What is the significance of f-strings and format specs?
f-strings allow Python variables to be embedded directly inside a string using {} braces. Format specifiers control column widths and number display:

• {num:2d} — integer, at least 2 characters wide
• {wine['name']:50} — string padded to 50 characters
• {wine['price']:,.0f} — float with commas, 0 decimal places
How does this affect the program's output?
Without the format specifiers, every row of the menu would have a different length and the prices would drift out of alignment. With them, every row lines up into neat, readable columns — the same principle used in receipts, invoices, and spreadsheets.
for loop .items() f-string format specifier :2d :50 :,.0f
04

Wine Selection Loop — while True + try/except

Input Validation ▾ 
while True:
    try:
        choice = int(input("\nEnter the number of the wine (1-15): "))
        if choice in wines:
            selected_wine = wines[choice]
            break
        else:
            print("Invalid number! Please choose between 1 and 15.")
    except ValueError:
        print("Please enter a valid number!")
Why while True?
while True creates an infinite loop that runs until explicitly stopped with break. This pattern guarantees the program never proceeds with invalid input.
Why try / except ValueError?
input() always returns a string. Wrapping it in int() converts it to an integer — but if the user types "abc", int() raises a ValueError. The try/except block catches that crash and prints a friendly message instead.
How does this affect the program's output?
This is the difference between a program that survives a typo and one that crashes and loses the customer's order. Every downstream line — the receipt, the total — depends on selected_wine being valid, so this loop is the gatekeeper for everything that follows.
while True try / except ValueError int(input()) in operator
05

Quantity Input Loop — Positive Integer Validation

Input Validation ▾ 
while True:
    try:
        quantity = int(input("\nEnter quantity (bottles): "))
        if quantity > 0:
            break
        else:
            print("Quantity must be at least 1!")
    except ValueError:
        print("Please enter a valid number!")
Why is this loop separate?
Each loop has a single responsibility — a core principle of clean code. The first loop handles which wine, this loop handles how many.
Why is the condition quantity > 0?
The > form reads more naturally as "Must be a positive number." It explicitly rejects zero, negative numbers, and text.
How does this affect the program's output?
A quantity of zero or a negative number would corrupt the math later — total_cost = price * quantity would print a nonsensical or negative subtotal. Rejecting bad quantities here keeps every number in the receipt trustworthy.
while True quantity > 0 break single responsibility
06

Senior Citizen Discount & Total Cost Computation

Calculation ▾ 
is_senior = input("\nAre you a senior citizen? (yes/no): ").strip().lower()
discount_rate = 0.10 if is_senior == "yes" else 0.00
total_cost = selected_wine['price'] * quantity
discount_amount = total_cost * discount_rate
final_cost = total_cost - discount_amount
What's the significance of the conditional expression?
The conditional expression 0.10 if is_senior == "yes" else 0.00 sets the discount rate to 10% for seniors and 0% otherwise in one clear line.
How is the final cost computed?
Three steps: First, total_cost is the product of price and quantity. Second, discount_amount is the total multiplied by the rate. Third, final_cost subtracts the discount from the total.
Why .strip().lower() on the input?
.strip() removes accidental leading/trailing spaces and .lower() normalizes the answer, so "Yes", " YES ", and "yes" are all treated identically. Without it, a stray space or capital letter would silently deny a senior their discount.
conditional expression arithmetic * and - business logic string .strip() .lower()

Receipt Output — f-strings with Number Formatting

Why use f-strings again here?
The receipt must display dynamic values — the actual wine chosen, the real price, the user's quantity, and the computed total. The :, format spec adds thousands separators (e.g. 1699 → 1,699) for currency readability.
What is the significance of the conditional discount line?
The if discount_amount > 0: check ensures the discount line only appears when a discount was actually applied.
How does this affect the program's output?
This is the payoff moment: every earlier line of code — the dictionary, the loops, the discount math — exists purely to produce these final, correctly formatted lines. A bug anywhere upstream shows up here as a wrong price, a missing discount, or a crash, which is why testing usually starts by reading the receipt backwards.

04 — Deep Dive

Key Python Concepts Explained

Nested Dictionaries as Mini-Databases. In professional software, data is usually stored in a database. In a small Python program, a nested dictionary serves exactly the same purpose. The outer dictionary's keys (integers 1–15) act like row IDs. The inner dictionaries act like rows in a table, each containing a name column and a price column. This is the same mental model used in SQL, MongoDB, and most modern data systems.

The Defensive Input Pattern. The combination of while True + try/except + break is one of the most important patterns in Python programming. It is called defensive programming — you assume the user will make mistakes and design your code to handle them gracefully without crashing.

f-strings vs Older String Formatting. f-strings (introduced in Python 3.6) are now the preferred method because they are readable, concise, and support format specifiers directly inside the curly braces. VINO PH uses advanced format specs like :2d, :50, and :,.0f.

05 — The Bigger Picture

Why Every Programmer Should Learn This

🎯 It's Not About Wine — It's About Reliable Software

VINO PH looks like a small toy program, but the seven patterns inside it are the same seven patterns that show up in payroll systems, hospital scheduling software, banking apps, and e-commerce checkouts. A nested dictionary today, a full SQL table tomorrow. A while True + try/except loop today, a production form-validation layer tomorrow. Programmers who skip learning these fundamentals "because the real work uses frameworks" often struggle later, because every framework is ultimately built on top of these exact primitives.

There is also a quieter, more important lesson here: defensive programming is a form of respect for the user. The two while True loops in VINO PH exist so that a customer who fat-fingers a "6" as an "e" never sees a crash, a stack trace, or a lost order. Learning to anticipate human error — rather than assuming perfect input — is what separates code that works in a demo from code that survives contact with real people.

🤖 VINO PH in the Era of AI

It is tempting to think that in the age of AI coding assistants, understanding a program like VINO PH line-by-line is optional — that you can simply ask an AI to "write me a wine shop program" and move on. The opposite is true. AI tools like Claude are extremely good at generating code such as this, but they cannot substitute for a human who can read, verify, and reason about the code that gets generated. When an AI-written program produces a wrong total, silently swallows an exception, or applies a discount to the wrong customer, someone still has to trace the bug back through the dictionary lookups, the loops, and the arithmetic — exactly the skill this walkthrough builds.

In practice, the programmers who benefit most from AI assistance are the ones who already understand fundamentals like these. They can spot when generated code has an off-by-one error in {num:2d}, notice when a try/except block is silently hiding a real bug, or recognize when a "senior discount" conditional expression has its logic backwards. Fluency in fundamentals is what turns an AI from an autocomplete tool into a genuine collaborator — you direct it, review it, and correct it, rather than trusting it blindly.
