01 — The Essence

Introduction to the Zoo Manager Project

📘 What Is the Zoo Manager, Exactly?

Zoo Manager is a small, fully working application built to turn one idea from the Inheritance guide into something you can click, break, and rebuild: a single Animal base class defines what every animal in the zoo must be able to do — feed() and makeSound() — and three subclasses, Lion, Elephant, and Parrot, each supply their own version of those two behaviors.

The project exists in two parallel forms that mirror each other line for line: a JavaScript simulator you can click through right now in the Live Simulator section, and the actual runnable Python source underneath it, which you can copy, run in-browser via Pyodide, or drop straight into your own project. Seeing the identical class hierarchy expressed in two different languages is itself part of the lesson — inheritance is a concept, not a Python-only trick.

Nothing here is decorative. Every button you press calls a real method, on a real object, resolved through Python's (and JavaScript's) actual method-lookup rules. When the Lion card roars instead of trumpeting, that isn't a canned animation — it's Lion.makeSound() genuinely overriding Animal.makeSound() at runtime.

The Zookeeper's Dilemma

Imagine you actually managed a real zoo with three hundred animals across forty species. Every morning, a zookeeper walks the grounds with one clipboard that says exactly two instructions: "feed it" and "listen to it." The zookeeper doesn't carry three hundred different instruction manuals — they don't need to. Each animal already knows how to interpret "feed it" in its own way: the lion wants meat, the elephant wants hay, the parrot wants seeds.

That is precisely what this program models. The zookeeper in our code is the simulator's UI and the final for loop in the Python script — neither of them has any idea what species it's dealing with. They just call feed() and makeSound() and trust that whichever subclass is actually running will do the right, species-specific thing. That trust is what inheritance and method overriding make possible.

Why this matters beyond animals: Replace "zookeeper" with "payment processor," "notification dispatcher," or "AI agent orchestrator," and the exact same shape reappears — one caller, many interchangeable implementations, zero special-casing.

02 — Why This Is Worth Building

Why a Programmer Should Learn This

Four Reasons This Small Project Pays Off

A zoo simulator sounds like a toy exercise, but the pattern it teaches shows up in almost every serious codebase you will ever touch. Here's why it's worth the twenty minutes.

1. It turns theory into muscle memory. Reading that "super().feed() extends rather than replaces the parent method" is one kind of understanding. Clicking "Feed" on a Lion card and watching the hunger bar drop while a lion-specific sentence appears in the log is a different, stickier kind of understanding — you're no longer trusting the explanation, you're watching the mechanism.

2. It rehearses the "is-a" test in a domain everyone already understands. Before you apply inheritance to payment processors, UI components, or AI agents, it helps to practice the judgment call — "is a Lion genuinely a kind of Animal?" — somewhere the answer is obvious, so the instinct is already built when the domain gets more abstract.

3. It bridges two languages with one mental model. The JavaScript class Lion extends Animal in the simulator and the Python class Lion(Animal): in the source panel are doing the same conceptual work with different syntax. Recognizing that overlap is exactly what lets a developer move between a Python backend and a JavaScript frontend without relearning object-oriented design from scratch.

4. It's a realistic onramp to much larger systems. The "one shared interface, many overriding subclasses, one caller that never checks which subclass it got" shape is the exact skeleton behind game engines, plugin architectures, ORM model classes, and — as the AI-era section below explains — modern AI agent frameworks. Master it here, at three animals, and it scales to three hundred.

Significance in the Programming World

The "animal hierarchy" example is not unique to this site — it's one of the oldest and most widely used teaching devices in computer science, appearing in nearly every introductory OOP course for a reason: animals have obvious, intuitive "is-a" relationships that don't require domain expertise to evaluate. But the significance runs deeper than pedagogy.

Under the hood, what looks like a simple if-free, type-check-free dispatch (calling animal.makeSound() without asking "which animal is this?") is implemented by real compilers and interpreters using structures like virtual method tables in C++/Java or method resolution order (MRO) lookups in Python. Every time you click "Speak" in this simulator, you're triggering — at a much smaller scale — the exact same dispatch mechanism that lets massive production systems route millions of calls to the correct implementation without a single hardcoded type check.

Takeaway: Small, toy-sized examples like this one aren't a "beginner phase" you graduate out of — they're a compressed, inspectable version of a mechanism that scales, unmodified, all the way up to distributed, production-grade software.

05 — The Real Python

The Python Behind This App

This simulator's JavaScript logic is a direct mirror of the Python class hierarchy below. Copy it, run it right here in the browser via Pyodide, or drop it straight into your own project.

class Animal:
    def __init__(self, name: str):
        self.name = name
        self.hunger = 50

    def feed(self) -> str:
        self.hunger = max(0, self.hunger - 25)
        return f"{self.name} has been fed."

    def makeSound(self) -> str:
        raise NotImplementedError("Subclasses must implement makeSound()")

class Lion(Animal):
    def feed(self) -> str:
        base = super().feed()
        return f"{base} It devours a slab of meat."

    def makeSound(self) -> str:
        return f"{self.name} lets out a mighty ROAR!"

class Elephant(Animal):
    def feed(self) -> str:
        base = super().feed()
        return f"{base} It munches on leaves and hay."

    def makeSound(self) -> str:
        return f"{self.name} trumpets loudly!"

class Parrot(Animal):
    def feed(self) -> str:
        base = super().feed()
        return f"{base} It cracks open a sunflower seed."

    def makeSound(self) -> str:
        return f"{self.name} squawks: 'Polly wants a cracker!'"

# One list, three different subclasses — polymorphism via inheritance
zoo = [Lion("Leo"), Elephant("Ella"), Parrot("Polly")]
for animal in zoo:
    print(animal.makeSound())
    print(animal.feed())

Comprehensive Line-by-Line Walkthrough

Below is every meaningful line in the script, explained in the order Python actually executes it — from class definition to the final printed line.

  1. class Animal: Declares the base class with no parent of its own (it implicitly inherits from Python's built-in object). Nothing runs yet — Python is just registering the blueprint.
  2. def __init__(self, name: str): The constructor every animal will run first. self.name = name stores the given name; self.hunger = 50 gives every new animal an identical, deliberately mid-range starting hunger — hungry enough that feeding visibly matters, not so hungry it looks broken on arrival.
  3. def feed(self) -> str: The base implementation of feeding. max(0, self.hunger - 25) subtracts 25 points but clamps the floor at zero using max(), so hunger can never go negative no matter how many times an already-full animal is fed. It then returns a plain, species-neutral confirmation string.
  4. def makeSound(self) -> str: raise NotImplementedError(...) Deliberately provides no real sound. If any future subclass forgets to override this, calling it fails loudly and immediately, rather than silently returning a wrong or empty sound — the same contract-enforcement pattern used by Shape.area() in the main Inheritance guide.
  5. class Lion(Animal): The parentheses declare Animal as the parent. Lion now has __init__, feed(), and makeSound() automatically available, and will override the last two.
  6. Lion's feed(): base = super().feed() calls Animal.feed() first, running the exact same hunger-clamping math and capturing its returned sentence in a local variable named base. The next line returns f"{base} It devours a slab of meat." — the parent's sentence with a lion-specific sentence appended. Nothing about the hunger math was duplicated.
  7. Lion's makeSound(): Fully replaces the parent's version (no super() call here, since there's no shared sound logic worth reusing) and returns f"{self.name} lets out a mighty ROAR!".
  8. class Elephant(Animal): and class Parrot(Animal): Structurally identical to Lion — each calls super().feed() and appends its own diet-specific sentence, then fully overrides makeSound() with its own vocalization. The repetition across all three subclasses is intentional: it's what makes the "same shape, different content" nature of overriding so visible.
  9. zoo = [Lion("Leo"), Elephant("Ella"), Parrot("Polly")] Builds one Python list holding three objects of three different classes. Python lists don't enforce a single element type, so this is perfectly valid — the only thing all three objects share is that they're each some kind of Animal.
  10. for animal in zoo: On each pass, animal is bound to the next object in the list — first the Lion, then the Elephant, then the Parrot. The loop body never asks type(animal) or branches on species; it treats every iteration identically.
  11. print(animal.makeSound()) then print(animal.feed()): For each animal, Python looks up makeSound and feed starting from that specific object's actual class — not from whatever type the variable "looks like" in the code. This is why the identical two lines produce a roar-then-meat-sentence for Leo, a trumpet-then-hay-sentence for Ella, and a squawk-then-seed-sentence for Polly.
  12. Final output: Six printed lines total — two per animal — with zero if statements checking species anywhere in the loop. Every difference in the output comes entirely from which subclass's overridden method actually executed.

Same shape in JavaScript: The Live Simulator above runs a JavaScript class hierarchy — class Lion extends Animal, super.feed(), and so on — that mirrors this Python source line for line. Clicking "Feed" on the Lion card triggers the exact same override chain you just read, just in a different syntax.

07 — Naming What You Built

The Design Patterns Hiding Inside This Project

You didn't just write "some inheritance" — you actually implemented two named, well-studied design patterns without necessarily calling them that. Recognizing them by name makes it much easier to talk about your code with other engineers, and to recognize the same shape the next time it shows up in a library you're reading.

Pattern 1: Template Method

The Template Method pattern defines the skeleton of an operation in a base class, deferring some steps to subclasses. That's exactly what Animal.feed() does: the "reduce hunger, don't go below zero" skeleton lives in the parent, and each subclass supplies its own closing detail by calling super().feed() and appending to it.

# The parent defines the "skeleton" — clamp hunger, return a base sentence
class Animal:
    def feed(self) -> str:
        self.hunger = max(0, self.hunger - 25)
        return f"{self.name} has been fed."

# Every subclass fills in its own "final step" — the flavor text
class Lion(Animal):
    def feed(self) -> str:
        base = super().feed()   # run the shared skeleton first
        return f"{base} It devours a slab of meat."  # fill in the "hook"

Where you'll see this again: Django's class-based views, pytest's fixture teardown chains, and PyTorch's nn.Module.__call__ (which runs bookkeeping, then calls your overridden forward()) all follow this exact "fixed skeleton, overridable steps" shape.

Pattern 2 (the road not taken): Strategy

It's worth seeing the alternative you didn't choose, because the comparison sharpens your judgment about when inheritance is the right tool. The Strategy pattern achieves similar flexibility through composition instead of inheritance — you'd store a "feeding behavior" object as an attribute rather than overriding a method.

# Strategy version — composition instead of inheritance
class FeedingStrategy:
    def describe(self) -> str:
        raise NotImplementedError

class CarnivoreDiet(FeedingStrategy):
    def describe(self) -> str:
        return "devours a slab of meat"

class Animal:
    def __init__(self, name: str, diet: FeedingStrategy):
        self.name = name
        self.diet = diet  # the behavior is an object, not a class relationship
        self.hunger = 50

    def feed(self) -> str:
        self.hunger = max(0, self.hunger - 25)
        return f"{self.name} has been fed. It {self.diet.describe()}."

leo = Animal("Leo", CarnivoreDiet())

Why we still chose inheritance for this project: Strategy shines when the behavior needs to change at runtime (an animal switching diets, a payment processor switching providers). Inheritance shines when the behavior is fixed for the object's entire lifetime — a Lion is always a Lion. Since our animals never change species mid-simulation, inheritance is the simpler, more honest fit. Knowing both patterns is what lets you make that call deliberately instead of by habit.

Pattern 3: Factory Method

Notice that spawnAnimal('lion') in the JavaScript simulator takes a plain string and decides which class to instantiate — the calling code (the spawn buttons) never writes new Lion(...) directly. That indirection is the Factory Method pattern: centralizing "which concrete subclass do I create?" in one place instead of scattering constructor calls everywhere.

def make_animal(species: str, name: str) -> Animal:
    # One place decides which concrete class to build — callers never
    # need to import Lion, Elephant, or Parrot directly.
    factories = {
        "lion": Lion,
        "elephant": Elephant,
        "parrot": Parrot,
    }
    if species not in factories:
        raise ValueError(f"Unknown species: {species}")
    return factories[species](name)

# The caller only ever deals with strings and the Animal interface
zoo = [make_animal("lion", "Leo"), make_animal("parrot", "Polly")]

Why this matters: if a new species is added later, only make_animal's dictionary needs a new entry — every call site that already uses make_animal("...", "...") keeps working unchanged. This is precisely what the JavaScript simulator's makeAnimal(kind) helper function does, just with an if/else chain instead of a dictionary lookup.

08 — Prove It Works

Testing Your Inheritance Hierarchy

A class hierarchy you can't verify with tests is a class hierarchy you're afraid to refactor. Here's a real, runnable pytest suite for the exact Animal/Lion/Elephant/Parrot hierarchy from the Python Source section above — copy it into a file named test_zoo.py alongside your code and run pytest.

import pytest
from zoo import Animal, Lion, Elephant, Parrot

def test_base_animal_cannot_make_sound():
    # Animal.makeSound() must fail loudly — it has no sound of its own
    generic = Animal("Generic")
    with pytest.raises(NotImplementedError):
        generic.makeSound()

def test_feed_reduces_hunger_by_25():
    leo = Lion("Leo")
    assert leo.hunger == 50
    leo.feed()
    assert leo.hunger == 25

def test_hunger_never_goes_negative():
    leo = Lion("Leo")
    for _ in range(10):  # feed far more times than needed
        leo.feed()
    assert leo.hunger == 0

def test_each_subclass_returns_a_unique_sound():
    leo, ella, polly = Lion("Leo"), Elephant("Ella"), Parrot("Polly")
    sounds = {leo.makeSound(), ella.makeSound(), polly.makeSound()}
    assert len(sounds) == 3  # no two subclasses accidentally share an override

def test_feed_message_mentions_the_animals_name():
    ella = Elephant("Ella")
    assert "Ella" in ella.feed()

@pytest.mark.parametrize("cls,expected_fragment", [
    (Lion, "meat"),
    (Elephant, "hay"),
    (Parrot, "seed"),
])
def test_each_subclass_mentions_its_own_diet(cls, expected_fragment):
    # A parametrized test — one function, three subclasses, matching the
    # "one interface, many overrides" shape of the hierarchy it's testing.
    animal = cls("Test Animal")
    assert expected_fragment in animal.feed()

def test_all_animals_are_instances_of_animal():
    zoo = [Lion("Leo"), Elephant("Ella"), Parrot("Polly")]
    assert all(isinstance(a, Animal) for a in zoo)

@pytest.fixture
def sample_zoo():
    # A shared fixture — every test below gets a fresh, identical zoo
    return [Lion("Leo"), Elephant("Ella"), Parrot("Polly")]

def test_feeding_one_animal_does_not_affect_others(sample_zoo):
    sample_zoo[0].feed()  # feed only the Lion
    assert sample_zoo[0].hunger == 25
    assert sample_zoo[1].hunger == 50  # Elephant untouched
    assert sample_zoo[2].hunger == 50  # Parrot untouched

def test_zoo_size_matches_number_of_animals_created(sample_zoo):
    assert len(sample_zoo) == 3
📝 Why These Specific Tests
  1. test_base_animal_cannot_make_sound: Verifies the contract-enforcement itself — that instantiating the base class directly and calling the unimplemented method actually raises, rather than silently doing nothing.
  2. test_hunger_never_goes_negative: Directly tests the max(0, ...) clamp by feeding an animal far more times than its hunger can absorb — exactly the kind of edge case a code review comment would flag, now automated.
  3. test_each_subclass_returns_a_unique_sound: Uses a Python set to assert all three sounds are distinct — a copy-paste bug where two subclasses accidentally return the same string would fail this test immediately.
  4. The @pytest.mark.parametrize test: Runs the same assertion logic against all three subclasses without three copy-pasted test functions — the test suite itself follows the "one interface, many implementations" idea it's testing.
  5. The sample_zoo fixture: Centralizes zoo construction so every test starts from an identical, known state — the same reason the JavaScript simulator resets hunger to 50 for every new animal.
  6. test_feeding_one_animal_does_not_affect_others: Directly guards against Mistake 4 from the Debugging Guide — if hunger were ever accidentally declared as a shared class attribute instead of an instance attribute, this test would immediately fail.
09 — Debugging Guide

Common Mistakes When Writing This Kind of Hierarchy

Every mistake below is one a real beginner makes on their first attempt at this exact pattern. Recognizing the symptom is often faster than re-reading the theory.

Mistake 1: Forgetting super().__init__()

# ✗ Wrong — never calls the parent constructor
class Lion(Animal):
    def __init__(self, name: str):
        self.roar_volume = "loud"  # self.name and self.hunger are never set!

Symptom: AttributeError: 'Lion' object has no attribute 'name' the first time you call .feed() or .makeSound(). Fix: call super().__init__(name) as the first line of any overridden __init__.

Mistake 2: Misspelling the Method Name in the Override

# ✗ Wrong — "makeSound" vs "make_sound" silently creates a NEW method
class Lion(Animal):
    def make_sound(self) -> str:  # typo — doesn't override anything
        return "ROAR"

Symptom: No error at all — Python happily lets you define a brand-new method. Calling lion.makeSound() still raises the parent's NotImplementedError, which is confusing because you were sure you overrode it. Fix: the override's method name (and parameter signature) must match the parent's exactly — copy-paste the def line from the parent class instead of retyping it.

Mistake 3: Calling super() with the Wrong Method Name

# ✗ Wrong — calls feed() but you're inside makeSound()
class Lion(Animal):
    def makeSound(self) -> str:
        base = super().feed()  # mismatched — reduces hunger during a sound call!
        return f"{self.name} roars."

Symptom: No crash, but hunger mysteriously decreases every time an animal makes a sound — a hard-to-spot logic bug rather than a hard crash. Fix: super().<method>() should almost always call the same method name you're currently overriding.

Mistake 4: Mutating a Class Attribute Instead of an Instance Attribute

# ✗ Wrong — hunger defined on the class, shared by every instance
class Animal:
    hunger = 50  # class attribute, not self.hunger in __init__

    def feed(self) -> str:
        Animal.hunger -= 25  # mutates the SHARED class attribute
        return "fed"

Symptom: Feeding any animal reduces every animal's hunger, because they're all reading and writing the same shared class-level variable instead of their own instance state. Fix: initialize per-instance state inside __init__ as self.hunger = 50, never as a bare class-body assignment.

Mistake 5: Testing the Wrong Thing with type() Instead of isinstance()

# ✗ Fragile — breaks the moment you add a BigCat(Lion) subclass
if type(animal) == Lion:
    print("It's a lion!")

# ✓ Correct — respects the hierarchy, works for future subclasses too
if isinstance(animal, Lion):
    print("It's a lion (or some kind of lion)!")

Symptom: Code that works today silently stops recognizing an animal once you introduce a deeper hierarchy (Exercise 05's BigCat → Lion chain, for example). Fix: prefer isinstance() over type() == whenever you're checking "is this object part of this family," which is almost always what you actually mean.

Mistake 6: Overriding __init__ Signature Incompatibly

# ✗ Fragile — Parrot needs an extra required argument other animals don't
class Parrot(Animal):
    def __init__(self, name: str, vocabulary_size: int):  # no default!
        super().__init__(name)
        self.vocabulary_size = vocabulary_size

# Anywhere that builds animals generically now breaks for Parrot:
for cls in [Lion, Elephant, Parrot]:
    cls("Test")  # TypeError on Parrot — missing vocabulary_size

Symptom: Generic code that constructs any subclass the same way ("Liskov substitution" — a subclass should be usable anywhere the parent is expected) breaks specifically on the one subclass with a different constructor signature. Fix: give new required parameters a sensible default, or keep the constructor signature consistent across the whole family.

Mistake 7: Forgetting That print() Isn't the Same as return

# ✗ Wrong — prints instead of returning, so the caller gets None
class Lion(Animal):
    def makeSound(self) -> str:
        print(f"{self.name} roars")  # this prints immediately...
        # ...but there's no return statement, so the method returns None

# Somewhere else in the code:
message = leo.makeSound()
print(f"Logged: {message}")  # prints "Logged: None"

Symptom: The sound appears to work fine when called directly, but any code that tries to use the returned value — log it, display it in a UI, test it — gets None instead. This exact mismatch is precisely what a type checker catches automatically, as shown in the Type Hints section. Fix: decide once whether a method's job is to print (a side effect) or to return (a value the caller can use), and stay consistent with what the base class's signature promises.

Mistake 8: Catching Exceptions Too Broadly Around Polymorphic Calls

# ✗ Wrong — swallows the exact error you added NotImplementedError to surface
for animal in zoo:
    try:
        print(animal.makeSound())
    except Exception:
        pass  # a forgotten override now fails completely silently

Symptom: An animal that was never given a proper makeSound() override just... doesn't print anything, with no error anywhere in the logs. The deliberate, loud NotImplementedError from the base class — designed specifically to be impossible to miss — gets silently absorbed by an overly broad except clause several layers up. Fix: catch specific, expected exceptions only; never blanket-catch Exception around code whose entire safety net depends on a specific error actually surfacing.

Mistake 9: Not Checking the MRO Before Adding Multiple Inheritance

# ✗ Confusing — both parents define feed(), which one actually runs?
class LandAnimal:
    def feed(self) -> str:
        return "Grazes on land."

class WaterAnimal:
    def feed(self) -> str:
        return "Filters food from water."

class Hippo(LandAnimal, WaterAnimal):
    pass  # which feed() runs? Depends entirely on parent order.

print(Hippo().feed())      # "Grazes on land." — LandAnimal wins
print(Hippo.__mro__)       # reveals exactly why, in order

Symptom: Multiple inheritance silently picks one parent's method over the other based on the order they were listed in the class definition — class Hippo(LandAnimal, WaterAnimal) behaves differently from class Hippo(WaterAnimal, LandAnimal), and neither raises an error to warn you about the ambiguity. Fix: whenever two parent classes define the same method name, print Cls.__mro__ and confirm the resulting order actually matches your intent — or better, avoid the ambiguity entirely by giving each parent a distinctly named method instead of a colliding one.

A Full Debugging Session, Narrated

Here's what actually diagnosing one of these bugs looks like in practice, start to finish — useful to read once so the process feels familiar the first time it happens to you for real.

The report: "I added a Giraffe class and now feeding any animal in my zoo raises AttributeError: 'Giraffe' object has no attribute 'hunger' — but only for Giraffe, everything else still works."

Step 1 — Reproduce narrowly. Confirm it's specifically Giraffe("Gerald").feed() that fails, not the whole zoo. It is — Lion("Leo").feed() still works fine in isolation.

Step 2 — Read the traceback bottom to top. The error points at the line self.hunger = max(0, self.hunger - 25) inside Animal.feed(). That's shared code every subclass uses — so if only Giraffe fails there, the bug isn't in that line, it's in why Giraffe never got a hunger attribute in the first place.

Step 3 — Check the constructor. Sure enough:

# The actual bug — found by checking Giraffe's __init__
class Giraffe(Animal):
    def __init__(self, name: str, neck_length_m: float):
        self.neck_length_m = neck_length_m  # super().__init__() was never called!

Step 4 — Connect it to a known pattern. This is Mistake 1 from earlier on this exact page: a forgotten super().__init__() call. Animal.__init__ — the only place self.hunger gets set — never ran for this object.

Step 5 — Fix and verify with a test, not just a manual check. Add super().__init__(name) as the constructor's first line, then run (or write) a test like test_hunger_never_goes_negative from the Testing section against Giraffe specifically, so this exact regression can't silently return later.

The pattern behind the pattern: almost every bug in an inheritance hierarchy traces back to one of the numbered mistakes on this page. Building a mental index of "symptom → likely mistake number" is, in practice, most of what makes debugging these hierarchies fast instead of frustrating.

Real-World Case Study: A Notification System

Same Shape, Different Domain

To prove this isn't just an animal-specific trick, here's the identical structural pattern applied to a problem you're far more likely to actually build: a notification system that can send messages by email, SMS, or push notification. Read it side by side with the Zoo hierarchy — the shapes are line-for-line identical.

class Notifier:
    def __init__(self, recipient: str):
        self.recipient = recipient
        self.sent_count = 0

    def send(self, message: str) -> str:
        self.sent_count += 1
        return f"[{self.sent_count}] Queued for {self.recipient}: {message}"

    def channel_name(self) -> str:
        raise NotImplementedError("Subclasses must implement channel_name()")

class EmailNotifier(Notifier):
    def send(self, message: str) -> str:
        base = super().send(message)
        return f"{base} (via SMTP, HTML formatted)"

    def channel_name(self) -> str:
        return "Email"

class SMSNotifier(Notifier):
    def send(self, message: str) -> str:
        base = super().send(message[:160])  # SMS-specific truncation rule
        return f"{base} (via carrier gateway, 160-char limit)"

    def channel_name(self) -> str:
        return "SMS"

class PushNotifier(Notifier):
    def send(self, message: str) -> str:
        base = super().send(message)
        return f"{base} (via FCM/APNs, silent if app is foregrounded)"

    def channel_name(self) -> str:
        return "Push"

# A dispatcher that has NEVER heard of Email, SMS, or Push specifically
class AlertDispatcher:
    def __init__(self, channels: list):
        self.channels = channels

    def broadcast(self, message: str) -> None:
        for channel in self.channels:
            print(f"[{channel.channel_name()}] {channel.send(message)}")

dispatcher = AlertDispatcher([
    EmailNotifier("glenn@example.com"),
    SMSNotifier("+63-900-000-0000"),
    PushNotifier("device-token-abc123"),
])
dispatcher.broadcast("Server deploy completed successfully.")

Map it back to the zoo: Notifier ↔ Animal, channel_name() ↔ makeSound(), send() ↔ feed(), and AlertDispatcher.broadcast() ↔ the zookeeper's for animal in zoo loop. A dispatcher was written once and will keep working correctly the day someone adds a SlackNotifier or a WebhookNotifier — exactly the way the zoo's loop kept working the day you added a Giraffe in Exercise 01.

Second Example: A UI Widget Hierarchy

One more domain, because two examples make the generalization much harder to doubt. This time: rendering a form made of different input widgets, each of which knows how to render itself as HTML.

class FormField:
    def __init__(self, label: str, name: str):
        self.label = label
        self.name = name

    def render(self) -> str:
        raise NotImplementedError("Subclasses must implement render()")

class TextField(FormField):
    def render(self) -> str:
        return f'<label>{self.label}</label><input type="text" name="{self.name}">'

class CheckboxField(FormField):
    def render(self) -> str:
        return f'<label><input type="checkbox" name="{self.name}"> {self.label}</label>'

class DropdownField(FormField):
    def __init__(self, label: str, name: str, options: list):
        super().__init__(label, name)
        self.options = options

    def render(self) -> str:
        opts = "".join(f'<option>{o}</option>' for o in self.options)
        return f'<label>{self.label}</label><select name="{self.name}">{opts}</select>'

# The form itself never knows or cares which widget types it contains
class Form:
    def __init__(self, fields: list):
        self.fields = fields

    def render(self) -> str:
        return "".join(field.render() for field in self.fields)

signup_form = Form([
    TextField("Full Name", "name"),
    CheckboxField("Subscribe to newsletter", "newsletter"),
    DropdownField("Country", "country", ["PH", "US", "UK"]),
])
print(signup_form.render())

The same shape, a third time: FormField ↔ Animal, render() ↔ makeSound(), and Form.render() ↔ the zookeeper's loop. This is, structurally, how real UI frameworks like React's class components, Django's form widgets, and native mobile UI toolkits render heterogeneous collections of components without a giant if isinstance(field, TextField): ... elif isinstance(field, CheckboxField): ... chain anywhere in their rendering code.

13 — Talking About It

Interview Prep: Questions You Can Now Answer

Technical interviews love inheritance questions because they're easy to ask and hard to answer well without hands-on experience. Having built this project, you can now answer every one of these with a concrete example instead of a memorized definition.

Q1. What's the difference between inheritance and composition, and when would you choose one over the other?
Inheritance models an "is-a" relationship and is fixed for an object's lifetime — a Lion is always a Lion. Composition models a "has-a" relationship and can change at runtime — an object can swap out a strategy object it holds. Choose inheritance when the specialization is permanent and the hierarchy is shallow; choose composition when the behavior needs to be swappable or the relationships would require multiple, unrelated inheritance chains.
Q2. Walk me through what happens when you call an overridden method on a subclass instance.
Python looks up the method starting from the object's actual runtime class — not the variable's declared type — following the Method Resolution Order. It finds the closest override in that chain and executes it. If that override calls super().method(), the lookup continues up the chain from that point, running the parent's version before returning control to the subclass's code.
Q3. Why would you deliberately make a base class method raise NotImplementedError?
To enforce a contract: every subclass must supply its own implementation before the class can be meaningfully used. It converts a silent, hard-to-trace bug (a subclass forgetting to implement required behavior) into an immediate, loud failure at the exact call site where the missing piece matters.
Q4. What's the Liskov Substitution Principle, and can you give an example of violating it?
It states that a subclass should be usable anywhere its parent is expected without breaking the caller's assumptions. Mistake 6 in this project's Debugging Guide is a direct violation: giving Parrot a required constructor argument the other subclasses don't have means generic code that builds any animal the same way breaks specifically on Parrot.
Q5. How does Python resolve method lookups when a class inherits from multiple parents?
Via the C3 linearization algorithm, producing a single, consistent Method Resolution Order stored on Cls.__mro__. Python searches that order left to right and uses the first matching implementation it finds — this avoids the classic "diamond problem" ambiguity that plagues naive multiple-inheritance implementations in other languages.
Q6. What's a practical difference between an instance attribute and a class attribute, and how have you seen that go wrong?
An instance attribute (self.hunger, set in __init__) belongs to one object; a class attribute (declared directly in the class body) is shared across every instance of that class. Mistake 4 in this project shows the failure mode directly: defining hunger as a class attribute means feeding any one animal silently changes every animal's hunger, because they're all reading and writing the same shared value.
Q7. How would you unit test that a base class's contract is actually being enforced by its subclasses?
Write a parametrized test that instantiates each subclass and asserts the overridden method returns something meaningful (not the base class's fallback or an exception), plus a separate test confirming the base class itself still raises when called directly. The Testing section above shows exactly this: test_base_animal_cannot_make_sound and the parametrized test_each_subclass_mentions_its_own_diet test together cover both directions.
Q8. What's the Factory Method pattern, and why not just call constructors directly everywhere?
Factory Method centralizes the decision of which concrete subclass to instantiate in one place — like this project's make_animal(species, name) function — instead of scattering Lion(...), Elephant(...) constructor calls throughout the codebase. When a new subclass is added, only the factory needs a new entry; every existing call site that already goes through the factory keeps working unchanged.
Q9. If you had a hierarchy five levels deep, how would you refactor it?
Start by identifying which parts of the chain are genuine "is-a" relationships versus behaviors that got bolted on as convenient shortcuts. Behaviors used by only some branches of the hierarchy are strong candidates for extraction into mixins or composed strategy objects — flattening the inheritance chain back down to two or three levels while keeping the shared behavior available where it's actually needed.
Q10. How do type hints interact with inheritance and static type checkers?
A subclass's overridden method should return a type compatible with what the base class promises — narrowing the return type is generally fine, widening or changing it isn't. Tools like mypy check this automatically: if Animal.feed() -> str and a subclass override implicitly returns None because of a missing return statement, mypy flags it before the code ever runs, as demonstrated in the Type Hints section.
Q11. What's a mixin, and how is it different from a normal parent class?
A mixin is a class designed to be combined with others via multiple inheritance to add one specific, self-contained capability — it typically has no __init__ of its own and isn't meant to be instantiated alone. Unlike a normal parent class representing an "is-a" relationship, a mixin represents a "can-do" capability, like a hypothetical FlyableMixin that could be added to Parrot without implying every flying thing is the same kind of object.
Q12. Describe a bug you'd expect from forgetting super().__init__(), and how you'd diagnose it.
The subclass instance would be missing every attribute the parent constructor was responsible for setting — calling any method that reads those attributes raises AttributeError. Diagnosis: read the traceback for which attribute is missing, then check whether that subclass's __init__ actually delegates to super().__init__() — exactly the debugging session narrated in the Common Mistakes section above.
14 — Common Questions

Frequently Asked Questions

Why does Animal.makeSound() raise an error instead of just returning a generic sound?

+

Returning a generic sound like "..." would let a broken subclass silently pass — you'd only notice something was wrong by reading the output and thinking "that doesn't sound right." Raising NotImplementedError instead fails immediately and loudly, at the exact moment the missing override matters, which is far easier to debug.

What happens if a subclass forgets to call super().feed()?

+

Nothing crashes — Python doesn't require it. But the subclass's feed() would then be responsible for updating self.hunger itself, or hunger would silently stop decreasing whenever that animal is fed. This is a common real-world bug: overriding a method and forgetting that super() was doing important work you now have to replicate.

Isn't this the same as just writing three separate, unrelated feed_lion(), feed_elephant(), and feed_parrot() functions?

+

No — and that difference is the whole point. Three separate functions would each need their own hunger-clamping logic duplicated, and the calling code would need an if/elif chain to pick the right one. With inheritance, the hunger math exists exactly once, and the calling loop never checks species at all. Fixing a bug in the clamping logic means editing one method, not three.

Does the JavaScript simulator work identically to the Python source?

+

Structurally, yes — class Lion extends Animal with super.feed() follows the same override-and-extend pattern as Python's class Lion(Animal) with super().feed(). The syntax differs (JavaScript uses extends and getters for read-only properties like emoji), but the dispatch mechanism — call a method, let the runtime resolve it to the actual subclass — is conceptually identical in both languages.

Is my zoo roster saved anywhere if I refresh the page?

+

No. The simulator keeps your roster in an in-memory JavaScript array for the current page session only, so refreshing or navigating away resets it. This is intentional — the simulator is a teaching tool for watching dispatch happen, not a persistence demo.

Can I add a species that behaves like two different animals at once?

+

Yes, using multiple inheritance — class FlyingSquirrel(Mammal, Glider):, for example. Python resolves conflicts between the parents using the Method Resolution Order (MRO), covered in the main Inheritance guide's Types section. It's a legitimate pattern, but reach for it only when the animal genuinely needs behavior from two unrelated hierarchies — not as a shortcut around a messy single hierarchy.

Why start hunger at 50 instead of 0 or 100?

+

A deliberate design choice, not a technical requirement — starting at the midpoint means the hunger bar visibly moves the first time you click "Feed" without immediately hitting the floor. It's a small UX decision that makes the underlying mechanism easier to observe, which matters more here than realism.

Why does the Case Study use a totally different example (notifications) instead of more animals?

+

To prove the pattern generalizes. If the only example you'd ever seen was animals, it would be easy to mistake "inheritance" for "a trick that works for zoo simulators." Seeing the identical shape — base class, contract-enforcing method, several overriding subclasses, one caller that never checks type — reappear in a notification system is what proves you learned a transferable pattern, not a one-off recipe.

Do I need to know the Template Method and Strategy pattern names to use inheritance well?

+

No — you can write correct code without ever naming the pattern. But knowing the names lets you communicate design decisions precisely with other engineers ("I used Template Method here because the skeleton is fixed") instead of re-explaining the whole mechanism from scratch every time it comes up in a code review or design discussion.

What's the actual difference between type(x) == Lion and isinstance(x, Lion)?

+

type(x) == Lion only matches if x is exactly a Lion — a hypothetical WhiteLion(Lion) subclass would fail the check. isinstance(x, Lion) matches Lion and any of its subclasses, respecting the hierarchy. Mistake 5 in the Debugging Guide covers exactly this trap.

Can two different subclasses accidentally have identical makeSound() output, and would that be a bug?

+

Nothing stops it syntactically — Python doesn't enforce distinct return values. Whether it's a bug depends entirely on your domain: for a zoo simulator it would almost certainly be an accidental copy-paste, which is exactly what the test_each_subclass_returns_a_unique_sound test in the Testing section is designed to catch.

Why does the pytest suite use @pytest.mark.parametrize instead of three separate test functions?

+

Because writing test_lion_diet, test_elephant_diet, and test_parrot_diet as three near-identical functions would duplicate the same assertion logic three times — the exact kind of repetition inheritance itself exists to eliminate. Parametrizing keeps the test's own logic written once, run three times, one case per subclass.

Is it bad practice to have every subclass call super() in every overridden method?

+

Not inherently — it depends on whether the parent's logic is actually worth reusing. feed() calls super() in every subclass because the hunger-clamping math is genuinely shared. makeSound() deliberately does not call super() in any subclass, because the base version only raises an error — there's nothing useful to reuse. Call super() when there's real shared logic; skip it when the override is a full replacement.

How would this project change if two animals needed to interact with each other, like a predator and prey?

+

You'd likely introduce a separate method or a small coordinating class rather than cramming interaction logic into Animal itself — for example, a hunt(predator, prey) function or a PredatorMixin class that only certain subclasses inherit alongside Animal. This is a good moment to reach for multiple inheritance or composition rather than overloading the base class with a concern most animals don't share.

Does adding more subclasses ever make the base class itself need to change?

+

Ideally, no — that's the whole promise of the Open/Closed Principle mentioned in the main Inheritance guide: classes should be open for extension (new subclasses) but closed for modification (the base class shouldn't need edits). If adding a new animal type keeps forcing you back into Animal's code, that's a signal the base class's interface is missing something, or the new type doesn't actually belong in this hierarchy.

Where should I go next after finishing this project?

+

Back to the Inheritance guide's Projects section for the other nine planned builds (Vehicle Garage, RPG Character Builder, Org Chart, and more), or forward to the Polymorphism guide — since everything this project demonstrated about "one interface, many overriding subclasses" is, formally speaking, polymorphism in action.

16 — Under the Hood

Performance & Memory Notes

Is Method Lookup Through Inheritance Slow?

In CPython, resolving animal.makeSound() walks the object's Method Resolution Order until it finds a match, then caches that result. For a hierarchy this shallow — one level deep — the lookup cost is negligible, effectively a single dictionary access per call. Performance only becomes a real conversation at much deeper hierarchies (five-plus levels) or extremely hot loops calling millions of overridden methods per second, neither of which apply here.

Rule of thumb: Don't avoid inheritance for performance reasons at this scale — reach for flatter hierarchies when the design calls for it (as the Best Practices section of the main Inheritance guide argues), not because you're worried about lookup speed.

Memory: Every Instance Carries Its Own __dict__

By default, each Lion, Elephant, and Parrot instance stores its attributes (name, hunger) in a per-instance dictionary. For three animals in a browser demo, this is irrelevant. For a real system tracking hundreds of thousands of objects, that per-instance dictionary overhead adds up — which is exactly the problem __slots__ solves, as covered in the Encapsulation guide's cheat sheet.

# A memory-conscious version of Animal for large-scale use
class Animal:
    __slots__ = ('name', 'hunger')  # no per-instance __dict__ at all

    def __init__(self, name: str):
        self.name = name
        self.hunger = 50

# Subclasses that add NEW attributes must declare their own __slots__ too
class Parrot(Animal):
    __slots__ = ('vocabulary_size',)

    def __init__(self, name: str, vocabulary_size: int = 0):
        super().__init__(name)
        self.vocabulary_size = vocabulary_size

Every subclass in a __slots__-based hierarchy must declare its own __slots__ tuple — even an empty one, __slots__ = () — or Python silently gives that subclass a __dict__ anyway, quietly undoing the memory savings for that branch of the hierarchy.

The Real Cost Is Almost Always Human, Not Computational

In practice, the performance question that matters far more than CPU cycles is: how long does it take a new engineer to understand this hierarchy? A shallow, well-named hierarchy like this project's — one base class, three direct children, two overridden methods — costs almost nothing to read. Deep, tangled hierarchies cost real engineering time regardless of how fast the CPU resolves them.

17 — Reading the Signatures

A Closer Look at the Type Hints

Every method in the Python source uses type hints — name: str, -> str — that are easy to skim past. They're not decoration; they're a second, machine-checkable description of the contract every subclass has to honor.

def feed(self) -> str:  # promises every override also returns a str
    ...

def makeSound(self) -> str:  # the SAME promise, on the SAME contract
    raise NotImplementedError(...)

Because Animal.feed() is annotated -> str, a static type checker like mypy can flag any subclass override that returns something else — say, None because a developer forgot a return statement — before the code ever runs, not after a user notices a blank line in the Zoo Log.

21 — The Other Half

The JavaScript Behind the Live Simulator

The Python Source section above is the "official" teaching example, but the Live Simulator you clicked through earlier is powered by real JavaScript — not a scripted animation. Here it is in full, so you can compare it line-for-line against the Python version and see exactly where the two languages agree and where their syntax diverges.

class Animal {
  constructor(name) {
    this.name = name;
    this.hunger = 50;
  }
  feed() {
    this.hunger = Math.max(0, this.hunger - 25);
    return this.name + ' has been fed.';
  }
  makeSound() {
    throw new Error('Subclasses must implement makeSound()');
  }
}

class Lion extends Animal {
  feed() { return super.feed() + ' It devours a slab of meat.'; }
  makeSound() { return this.name + ' lets out a mighty ROAR!'; }
}

class Elephant extends Animal {
  feed() { return super.feed() + ' It munches on leaves and hay.'; }
  makeSound() { return this.name + ' trumpets loudly!'; }
}

class Parrot extends Animal {
  feed() { return super.feed() + ' It cracks open a sunflower seed.'; }
  makeSound() { return this.name + " squawks: 'Polly wants a cracker!'"; }
}
📝 Same Idea, Different Syntax — A Direct Comparison
  1. class Lion(Animal): vs. class Lion extends Animal {'{'}: Python uses parentheses after the class name; JavaScript uses the explicit keyword extends. Both declare the exact same relationship.
  2. super().feed() vs. super.feed(): Python calls super() as a function first, then accesses the method; JavaScript treats super as a direct reference to the parent's prototype methods. Same purpose, different grammar.
  3. self vs. this: Python requires self as an explicit first parameter on every method; JavaScript's this is implicit and bound automatically for class methods. Both refer to "the specific object this method was called on."
  4. raise NotImplementedError(...) vs. throw new Error(...): Python's exception hierarchy has a dedicated, semantically precise exception for this exact situation; JavaScript's generic Error class is used the same way, since JavaScript has no equivalent dedicated subclass built in by default.
  5. What's identical: the dispatch mechanism. In both languages, calling .makeSound() on an object resolves to that object's actual runtime class first — never the declared type, never a hardcoded check. That single fact is the entire lesson, expressed twice.

24 — Choosing the Right Tool

A Simple Decision Tree: Should This Be a Subclass?

Work Through These Questions, in Order

Every time you're tempted to write class NewThing(Animal): — or any new subclass of any hierarchy — walk through this sequence before committing to it.

QuestionIf YesIf No
1. Does the "is-a" test genuinely hold?Continue to question 2.Stop — use composition (a "has-a" attribute) instead.
2. Does the new type need to override at least one existing method?Continue to question 3.Consider whether you need a new class at all, or just a new instance of an existing class with different data.
3. Can the new type's behavior be fully expressed by overriding Animal's existing two methods?Write a normal subclass, exactly like Lion, Elephant, or Parrot.Consider whether Animal's interface itself needs a new method — added carefully, since every existing subclass will need to handle it too.
4. Does the behavior need to change at runtime, not just per-type?Reconsider Strategy (composition) instead of inheritance, even if question 1 passed.Inheritance remains the right, simpler choice.
5. Will this push the hierarchy past 2–3 levels deep?Pause and reconsider whether a mixin or composition would flatten the design.Proceed — the hierarchy is still shallow enough to reason about easily.

The point of this tree isn't to be followed mechanically every time — it's to make the judgment calls this page has been building toward (the "is-a" test, Strategy vs. inheritance, hierarchy depth) into a repeatable habit instead of a one-off insight you read once and forgot.

25 — Putting It Together Live

A Mock Live-Coding Prompt, Solved Step by Step

The Prompt

"Design a small class hierarchy for a zoo's feeding system. There should be a way to feed any animal in a collection without the calling code knowing which specific animal it's feeding. Each animal type should reduce hunger the same way, but describe the feeding differently. Add a test proving your solution actually differentiates between types."

Step 1: Restate the Requirements as a Checklist

  • Need a shared interface for feeding across all animal types.
  • The calling code must not branch on type — this is explicitly asking for polymorphism.
  • Hunger reduction logic should be shared, not duplicated per type.
  • Each type needs its own description text.
  • Needs at least one test proving the differentiation actually works.

Step 2: Recognize the Shape You Already Know

This is, almost word for word, the exact Animal/Lion/Elephant/Parrot hierarchy from this project. Naming that recognition out loud in a real interview is itself a good signal — it shows you can map an unfamiliar-sounding prompt back onto a pattern you've actually built before, rather than starting from a blank page.

Step 3: Write the Base Class First, Interface Only

class Animal:
    def __init__(self, name: str):
        self.name = name
        self.hunger = 50

    def feed(self) -> str:
        self.hunger = max(0, self.hunger - 25)
        return self._feed_message()

    def _feed_message(self) -> str:
        raise NotImplementedError

Notice the small deliberate change from the main project: splitting out a private _feed_message() hook instead of having subclasses override feed() directly. This is the Template Method pattern applied slightly more strictly — the hunger math in feed() is now completely un-overridable, which is worth mentioning out loud as a design tradeoff in a live interview.

Step 4: Fill In the Subclasses

class Lion(Animal):
    def _feed_message(self) -> str:
        return f"{self.name} devours a slab of meat."

class Elephant(Animal):
    def _feed_message(self) -> str:
        return f"{self.name} munches on leaves and hay."

Step 5: Write the Caller That Never Branches on Type

def feed_all(animals: list) -> list:
    return [a.feed() for a in animals]

Step 6: Write the Required Test

def test_feed_differentiates_between_types():
    lion, elephant = Lion("Leo"), Elephant("Ella")
    assert lion.feed() != elephant.feed()

Why walking through this matters: the technical content here is nothing new — it's the same hierarchy this whole page has been teaching. What's new is the process: restate requirements, recognize a known shape, build the interface before the implementation, and finish with a test that directly verifies the requirement that was asked for. That process transfers to prompts this page never covered.

27 — One More AI Angle

Safely Refactoring This Hierarchy with an AI Assistant

Prompting for Structural Changes Without Breaking the Contract

Given how naturally this kind of hierarchy comes up in AI-assisted coding sessions, it's worth naming a few habits that keep an AI assistant's refactor honest rather than accidentally destructive.

Do: paste the whole hierarchy, not one class. An assistant asked to "add a Giraffe" without seeing Animal, Lion, Elephant, and Parrot together has no way to match the existing pattern — it may invent a different shape entirely (a different constructor style, different method names) rather than extending the one you already have.

Do: ask it to run (or describe) the existing tests after any change. The pytest suite from the Testing section exists precisely so that "does this refactor still work?" has an objective, checkable answer — for a human or an AI assistant equally.

Don't: accept a rewrite that silently drops the NotImplementedError contract enforcement. It's a common, well-intentioned "simplification" an assistant might suggest — replacing it with a default sound "to make the base class more usable directly." That change removes the exact safety net Mistake 7's debugging story depended on; review structural suggestions like this with the same scrutiny you'd give a human teammate's pull request.

Don't: let a suggested refactor quietly break the Liskov Substitution Principle. If an AI-suggested change to one subclass's constructor adds a new required parameter the others don't have, that's Mistake 6 reappearing — worth catching in review, not after deployment.

Structuring This as a Real Multi-File Project

From One Script to a Small Package

Everything on this page lives in a single Python source block for teaching clarity, but a real codebase would split this hierarchy across files the moment more than one person needs to work on it. Here's a reasonable structure for exactly that:

zoo_project/
├── zoo/
│   ├── __init__.py
│   ├── animals.py        # Animal, Lion, Elephant, Parrot
│   ├── factory.py        # make_animal() — the Factory Method from the Design Patterns section
│   └── zookeeper.py      # Zookeeper class from Exercise 04
├── tests/
│   └── test_zoo.py       # the pytest suite from the Testing section
└── README.md

Splitting animals.py from zookeeper.py matters for a concrete reason: the zookeeper-facing code (the "caller" in every example on this page) should only ever need to import the Animal base class's public interface and the factory — never the individual subclasses directly. That's the multi-file version of the same discipline the loop demonstrates in a single file: depend on the shared interface, not on any specific concrete type.

# zoo/animals.py
class Animal:
    ...

class Lion(Animal):
    ...

# zoo/factory.py
from .animals import Animal, Lion, Elephant, Parrot

def make_animal(species: str, name: str) -> Animal:
    ...

# zoo/zookeeper.py — notice: no import of Lion, Elephant, or Parrot at all
from .animals import Animal

class Zookeeper:
    def rounds(self, zoo: list[Animal]) -> None:
        for animal in zoo:
            print(animal.makeSound())
            print(animal.feed())

The tell that your structure is right: if zookeeper.py never has to change or add an import when a new animal subclass is added to animals.py, your file boundaries are correctly following the same "depend on the interface, not the implementation" idea this entire page has been building toward.

How the Roster Cards Actually Get Drawn

One piece of this project hasn't been explained yet: how an Animal object turns into the visual card you clicked in the Live Simulator. It's worth walking through, because it's a second, smaller demonstration of polymorphism doing real work — this time in the rendering layer instead of the data layer.

function buildCard(animal) {
  const card = document.createElement('div');
  card.className = 'animal-card ' + animal.typeName.toLowerCase();

  const head = document.createElement('div');
  head.innerHTML = '<span class="ac-emoji">' + animal.emoji + '</span>' +
    '<div><div class="ac-name">' + animal.name + '</div>' +
    '<div class="ac-type">' + animal.typeName + ' extends Animal</div></div>';
  card.appendChild(head);

  const bar = document.createElement('div');
  const fill = document.createElement('div');
  fill.style.width = animal.hunger + '%';
  fill.style.background = animal.hunger > 70 ? 'var(--red)' : animal.barColor;
  bar.appendChild(fill);
  card.appendChild(bar);

  return card;
}
📝 Why This Function Is Polymorphic Too
  1. animal.typeName, animal.emoji, animal.barColor: Every one of these is a JavaScript getter — a property that looks like a plain attribute from the outside but actually runs a small overridden method under the hood. Lion, Elephant, and Parrot each define their own version, exactly like feed() and makeSound().
  2. buildCard() never checks the animal's type: Notice there is no if (animal instanceof Lion) anywhere in this function, even though the card visually looks different for every species — amber for Lion, blue for Elephant, teal for Parrot. The visual difference comes entirely from reading overridden getters, the same "one caller, many overriding implementations" shape as everything else on this page.
  3. The hunger bar's color logic: animal.hunger > 70 ? 'var(--red)' : animal.barColor layers one small conditional (an urgent-hunger warning) on top of the per-species color, which itself came from an overridden getter — a small example of combining a shared rule with subclass-specific data.
  4. Why this matters beyond this project: real UI frameworks constantly do exactly this — render a heterogeneous list of objects by calling the same handful of methods/getters on each one, never branching on concrete type. This function is a miniature, inspectable version of how a React or Vue component list renders a mix of different item types.
