PYTHON CLASSES: A COMPLETE GUIDE (2026)

1.0 THE THEORETICAL AND PRACTICAL FOUNDATIONS OF OBJECT-ORIENTED PROGRAMMING

Object-Oriented Programming (OOP) is one of the most influential paradigms in software engineering history. At its heart, OOP organises code around objects ‚Äî self-contained units that bundle data (attributes) and behaviour (methods) together. Python embraces OOP fully while remaining flexible enough to mix paradigms. Understanding classes is therefore not merely a language skill; it is a fundamental shift in how you model the real world in code.

A class is a blueprint ‚Äî a precise description of what instances of that type will look like and how they will behave. When you write class Dog:, you are not creating a dog; you are creating the concept of a dog, from which any number of individual dog objects can be stamped out. Each object (instance) carries its own state but shares the class's methods.

1.1 Why This Guide Exists ‚Äî Our Educational Philosophy

This resource has been designed as a comprehensive educational tool for computer science students, self-directed learners, and professional developers seeking to deepen their understanding of Python OOP. Every explanation has been crafted to meet university-level instructional standards: we emphasise not just the "how" but the "why" behind each concept, connecting theory to practical software design.

1.2 Prerequisites and Learning Path

This guide assumes familiarity with Python functions, variables, data types, conditionals, and loops. Some exposure to Python functions (including *args and **kwargs) is helpful. If you have not yet studied functions, we recommend reviewing the POETIC BYTES Python Functions Guide first.

2.0 CLASS BASICS ‚Äî ANATOMY OF A PYTHON CLASS

2.1 Defining a Class

Every class definition starts with the class keyword, followed by the class name (by convention in CamelCase), an optional inheritance list in parentheses, and a colon. The class body is indented and contains attribute assignments and method definitions.

The special method __init__ is the constructor ‚Äî it is called automatically when a new instance is created. Its first parameter is always self, which is a reference to the instance being constructed. You use self to attach attributes to the instance and to call other instance methods.

üí° Key Distinction: self is not a keyword in Python ‚Äî it is merely a naming convention. You could name it anything, but every Python programmer uses self so you should too. It is automatically passed by Python when you call an instance method.

2.2 Instance Attributes vs Class Attributes

Instance attributes are defined inside __init__ using self.name = value. They belong to one specific object ‚Äî each instance has its own copy.

Class attributes are defined at the class level, outside any method. They are shared across all instances. Modifying a class attribute on the class changes it everywhere; modifying it on an instance creates a new instance attribute that shadows the class attribute for that instance only.

‚ö†Ô∏è Critical Warning: Never use mutable objects (lists, dicts) as class attributes if you intend them to be per-instance. All instances will share the same object, causing very hard-to-debug side effects. Use self.items = [] inside __init__ instead.

EXAMPLE 2.1 ‚Äî Class Definition, Instantiation, and Attribute Access 
class BankAccount:
    """A simple bank account class demonstrating OOP fundamentals."""

    # Class attribute ‚Äî shared by ALL instances
    bank_name: str = "POETIC BANK"
    _total_accounts: int = 0  # tracks how many accounts exist

    def __init__(self, owner: str, balance: float = 0.0) -> None:
        """Initialise a new bank account."""
        # Instance attributes ‚Äî unique to each object
        self.owner   = owner
        self.balance = balance
        self._transactions: list[float] = []   # mutable ‚Üí always on self
        BankAccount._total_accounts += 1

    def deposit(self, amount: float) -> None:
        if amount <= 0:
            raise ValueError("Deposit amount must be positive.")
        self.balance += amount
        self._transactions.append(amount)
        print(f"Deposited ¬£{amount:.2f} ‚Äî new balance: ¬£{self.balance:.2f}")

    def withdraw(self, amount: float) -> None:
        if amount > self.balance:
            raise ValueError("Insufficient funds.")
        self.balance -= amount
        self._transactions.append(-amount)
        print(f"Withdrew ¬£{amount:.2f} ‚Äî new balance: ¬£{self.balance:.2f}")

    def get_statement(self) -> str:
        lines = [f"Account: {self.owner}"]
        for t in self._transactions:
            lines.append(f"  {'Credit' if t > 0 else 'Debit':6s}: ¬£{abs(t):.2f}")
        lines.append(f"Balance: ¬£{self.balance:.2f}")
        return "\n".join(lines)

# Instantiation
alice = BankAccount("Alice", 1000.0)
bob   = BankAccount("Bob")

alice.deposit(250.0)
alice.withdraw(75.0)
print(alice.get_statement())

# Class attributes are accessible on both the class and instances
print(f"Bank: {BankAccount.bank_name}")
print(f"Total accounts: {BankAccount._total_accounts}")  # 2

3.0 INHERITANCE AND THE METHOD RESOLUTION ORDER (MRO)

3.1 Single Inheritance

Inheritance allows a child class to acquire the attributes and methods of a parent class, extending or overriding them as needed. You specify the parent class in parentheses: class Child(Parent):. The child automatically has access to all of the parent's non-private members.

To call the parent's __init__ (or any overridden method), use super(). This is essential when the child adds new attributes ‚Äî you want to call the parent's init to set up inherited attributes before adding the child's own.

3.2 Multiple Inheritance and the C3 MRO

Python supports multiple inheritance ‚Äî a class can inherit from more than one parent. Python resolves method lookup order using the C3 linearization algorithm, producing a deterministic MRO accessible via ClassName.__mro__ or ClassName.mro(). Understanding MRO prevents subtle bugs when overriding methods in diamond inheritance hierarchies.

3.3 Using super() Correctly

super() returns a proxy object that delegates method calls to the next class in the MRO ‚Äî not necessarily the direct parent. This makes it cooperative: when every class in a hierarchy uses super(), method calls chain correctly through the entire MRO, which is critical for multiple inheritance to work reliably.

üí° Key Rule: Always use super().__init__(...) rather than ParentClass.__init__(self, ...) when inside a class that may participate in multiple inheritance. super() is MRO-aware; a direct call is not.

EXAMPLE 3.1 ‚Äî Inheritance, super(), and MRO 
class Animal:
    def __init__(self, name: str, sound: str) -> None:
        self.name  = name
        self.sound = sound

    def speak(self) -> str:
        return f"{self.name} says '{self.sound}!'"

    def __repr__(self) -> str:
        return f"{type(self).__name__}(name={self.name!r})"

class Pet(Animal):
    def __init__(self, name: str, sound: str, owner: str) -> None:
        super().__init__(name, sound)   # delegate to Animal.__init__
        self.owner = owner

    def speak(self) -> str:
        base = super().speak()          # extend parent behaviour
        return f"{base} (owned by {self.owner})"

class ServiceAnimal(Animal):
    def __init__(self, name: str, sound: str, role: str) -> None:
        super().__init__(name, sound)
        self.role = role

    def describe_role(self) -> str:
        return f"{self.name} is trained as: {self.role}"

# Multiple inheritance ‚Äî diamond pattern
class ServicePet(Pet, ServiceAnimal):
    def __init__(self, name: str, sound: str, owner: str, role: str) -> None:
        # super() follows MRO: ServicePet ‚Üí Pet ‚Üí ServiceAnimal ‚Üí Animal
        # Python's C3 linearization ensures Animal.__init__ runs once
        Pet.__init__(self, name, sound, owner)
        self.role = role

# Inspect the MRO
print(ServicePet.__mro__)
# (, , , , )

buddy = ServicePet("Buddy", "Woof", "Alice", "Guide Dog")
print(buddy.speak())
print(buddy.describe_role())
print(isinstance(buddy, Animal))   # True ‚Äî buddy is-an Animal
4.0 MAGIC (DUNDER) METHODS ‚Äî INTEGRATING WITH PYTHON'S DATA MODEL

4.1 What Are Dunder Methods?

Dunder methods (double-underscore methods, e.g. __str__, __add__, __len__) allow your objects to participate in Python's built-in syntax and protocols. When you write len(obj), Python calls obj.__len__(). When you write a + b, Python calls a.__add__(b). This is Python's data model ‚Äî by implementing the right dunder methods, your objects integrate seamlessly with the language.

4.2 Essential Dunder Methods

__repr__ vs __str__: __repr__ should return an unambiguous, developer-facing string ‚Äî ideally valid Python that could recreate the object. __str__ returns a human-friendly string for end users. When only __repr__ is defined, Python uses it as a fallback for str() as well.

Comparison methods: __eq__, __lt__, __le__, __gt__, __ge__, __ne__ enable ==, <, <=, >, >=, !=. Use @functools.total_ordering to auto-generate missing comparisons from just __eq__ and one other.

Arithmetic operators: __add__, __sub__, __mul__, __truediv__ and their reflected (__radd__) and in-place (__iadd__) variants.

Container protocol: __len__, __getitem__, __setitem__, __delitem__, __contains__, __iter__, __next__ ‚Äî implement these to make your object behave like a sequence or mapping.

Context manager protocol: __enter__ and __exit__ ‚Äî allow your object to be used with the with statement for resource management.

EXAMPLE 4.1 ‚Äî Implementing a Rich Vector Class with Dunder Methods 
from __future__ import annotations
import math
import functools

@functools.total_ordering
class Vector2D:
    """2D vector with full operator support via dunder methods."""

    def __init__(self, x: float, y: float) -> None:
        self.x = x
        self.y = y

    # ‚îÄ‚îÄ Representation ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
    def __repr__(self) -> str:
        return f"Vector2D({self.x}, {self.y})"   # unambiguous, recreatable

    def __str__(self) -> str:
        return f"({self.x}, {self.y})"            # friendly display

    # ‚îÄ‚îÄ Arithmetic ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
    def __add__(self, other: Vector2D) -> Vector2D:
        return Vector2D(self.x + other.x, self.y + other.y)

    def __sub__(self, other: Vector2D) -> Vector2D:
        return Vector2D(self.x - other.x, self.y - other.y)

    def __mul__(self, scalar: float) -> Vector2D:   # v * 3
        return Vector2D(self.x * scalar, self.y * scalar)

    def __rmul__(self, scalar: float) -> Vector2D:  # 3 * v
        return self.__mul__(scalar)

    def __neg__(self) -> Vector2D:
        return Vector2D(-self.x, -self.y)

    # ‚îÄ‚îÄ Comparison (total_ordering fills in the rest) ‚îÄ
    def __eq__(self, other: object) -> bool:
        if not isinstance(other, Vector2D): return NotImplemented
        return math.isclose(self.x, other.x) and math.isclose(self.y, other.y)

    def __lt__(self, other: Vector2D) -> bool:
        return abs(self) < abs(other)   # compare by magnitude

    # ‚îÄ‚îÄ Other protocols ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
    def __abs__(self) -> float:          # abs(v) ‚Üí magnitude
        return math.hypot(self.x, self.y)

    def __bool__(self) -> bool:          # bool(v) ‚Üí False only for zero vector
        return self.x != 0 or self.y != 0

    def __hash__(self) -> int:
        return hash((self.x, self.y))

    def dot(self, other: Vector2D) -> float:
        return self.x * other.x + self.y * other.y

v1 = Vector2D(3, 4)
v2 = Vector2D(1, 2)

print(v1 + v2)           # (4, 6)
print(v1 * 2)            # (6, 8)
print(3 * v2)            # (3, 6) ‚Äî uses __rmul__
print(abs(v1))            # 5.0  ‚Äî Pythagorean theorem
print(v1 > v2)            # True ‚Äî magnitude 5 > 2.24
print(repr(v1))           # Vector2D(3, 4)
print(v1.dot(v2))         # 11

5.0 PROPERTIES ‚Äî CONTROLLED ATTRIBUTE ACCESS

5.1 The Property Decorator

Properties allow you to define methods that are accessed like attributes, giving you a clean public API while maintaining internal control. A property getter is decorated with @property. The corresponding setter is decorated with @name.setter and the deleter with @name.deleter. If no setter is defined, the attribute is read-only.

This is Python's idiomatic approach to encapsulation. Unlike Java, where you define explicit getX() and setX() methods, Python starts with simple public attributes and refactors to properties only when you need validation or computed values ‚Äî without changing the caller's interface at all.

5.2 When to Use Properties

Use a property when you need to: validate data before assignment, compute a value on the fly from other attributes, log or trigger side effects on access, or make an attribute read-only. Do not add properties preemptively ‚Äî start with a plain attribute and refactor if needed.

üí° Design Principle: The Uniform Access Principle states that attributes and computed values should be accessed the same way. Python properties enforce this ‚Äî you can change from a stored attribute to a computed property without changing a single line of calling code.

EXAMPLE 5.1 ‚Äî Properties with Validation and Computed Values 
class Circle:
    """A circle with validated radius and computed geometry properties."""

    def __init__(self, radius: float) -> None:
        self.radius = radius   # triggers the setter below

    @property
    def radius(self) -> float:
        """The radius of the circle (must be positive)."""
        return self._radius

    @radius.setter
    def radius(self, value: float) -> None:
        if value <= 0:
            raise ValueError(f"Radius must be positive, got {value}")
        self._radius = value   # store in private _radius

    @radius.deleter
    def radius(self) -> None:
        raise AttributeError("Cannot delete radius.")

    @property
    def diameter(self) -> float:
        """Computed from radius ‚Äî no storage needed."""
        return self._radius * 2

    @property
    def area(self) -> float:
        import math
        return math.pi * self._radius ** 2

    @property
    def circumference(self) -> float:
        import math
        return 2 * math.pi * self._radius

    def __repr__(self) -> str:
        return f"Circle(radius={self._radius})"

c = Circle(5)
print(c.radius)           # 5 ‚Äî read via getter
print(c.diameter)         # 10
print(f"{c.area:.4f}")   # 78.5398

c.radius = 7             # triggers setter ‚Äî valid
print(c.area)             # 153.9380‚Ä¶

try:
    c.radius = -3         # triggers setter ‚Äî invalid ‚Üí ValueError
except ValueError as e:
    print(f"Caught: {e}")

6.0 CLASS METHODS AND STATIC METHODS

6.1 The Three Method Types

Python has three distinct method types, each with a different first argument and purpose. Instance methods receive self (the instance) as their first argument ‚Äî they can access and modify instance state. Class methods (decorated with @classmethod) receive cls (the class itself) as their first argument ‚Äî they can access class attributes but not instance attributes, and are commonly used as alternative constructors. Static methods (decorated with @staticmethod) receive no implicit argument ‚Äî they are simply functions that live inside a class for organisational purposes and have no access to the class or instance.

6.2 Alternative Constructors with @classmethod

One of the most useful applications of class methods is creating alternative constructors. If your class can be created from different data sources (a string, a dict, a file path), you can provide a factory method for each. Crucially, using cls(...)) instead of hardcoding the class name means subclasses will inherit and correctly return their own type.

üí° Pattern: Use @classmethod for factory methods and @staticmethod for utility functions logically related to the class but not needing access to its state. If a method doesn't need self or cls, make it a staticmethod.

EXAMPLE 6.1 ‚Äî Instance, Class, and Static Methods in Practice 
from datetime import date

class Person:
    """Demonstrates all three method types."""

    _registry: list["Person"] = []   # class attribute

    def __init__(self, name: str, birth_year: int) -> None:
        self.name       = name
        self.birth_year = birth_year
        Person._registry.append(self)

    # ‚îÄ‚îÄ Instance method: operates on self ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
    def age(self) -> int:
        return date.today().year - self.birth_year

    def greet(self) -> str:
        return f"Hello, I'm {self.name}, age {self.age()}."

    # ‚îÄ‚îÄ Class methods: alternative constructors ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
    @classmethod
    def from_string(cls, data: str) -> "Person":
        """Create from 'Name:year' string."""
        name, year = data.split(":")
        return cls(name.strip(), int(year.strip()))

    @classmethod
    def from_dict(cls, data: dict) -> "Person":
        """Create from a dictionary."""
        return cls(data["name"], data["birth_year"])

    @classmethod
    def all_people(cls) -> list["Person"]:
        return list(cls._registry)

    # ‚îÄ‚îÄ Static method: utility, no state needed ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
    @staticmethod
    def is_adult(birth_year: int) -> bool:
        return date.today().year - birth_year >= 18

    def __repr__(self) -> str:
        return f"Person({self.name!r}, {self.birth_year})"

# Instance method
alice = Person("Alice", 1990)
print(alice.greet())

# Alternative constructors via classmethods
bob   = Person.from_string("Bob : 1985")
carol = Person.from_dict({"name": "Carol", "birth_year": 2010})

print(Person.all_people())

# Static method ‚Äî call on class or instance
print(Person.is_adult(1990))   # True
print(carol.is_adult(2010))     # False

7.0 ABSTRACT BASE CLASSES ‚Äî DEFINING INTERFACES AND CONTRACTS

7.1 What Are Abstract Classes?

An abstract class is a class that cannot be instantiated directly; it exists solely as a template for subclasses. Abstract classes define a contract: any concrete subclass must implement all abstract methods, or it too becomes abstract and cannot be instantiated. Python provides this via the abc module: inherit from ABC and decorate abstract methods with @abstractmethod.

7.2 Abstract Properties and Class Methods

You can combine @abstractmethod with @property, @classmethod, or @staticmethod. The combination enforces that subclasses provide not just any implementation but one of the correct type. For example, @property @abstractmethod forces subclasses to define a property (not just a method).

üí° Design Principle: Abstract classes embody the "Program to an interface, not an implementation" principle. They let you write code that works with any object honouring the contract, making it trivially easy to swap implementations ‚Äî e.g. switching from a SQLite to a PostgreSQL data store.

EXAMPLE 7.1 ‚Äî Abstract Base Classes and the Strategy Pattern 
from abc import ABC, abstractmethod

class Shape(ABC):
    """Abstract base class ‚Äî cannot be instantiated directly."""

    @abstractmethod
    def area(self) -> float: ...

    @abstractmethod
    def perimeter(self) -> float: ...

    @property
    @abstractmethod
    def name(self) -> str: ...   # subclasses must expose .name as a property

    # Concrete method ‚Äî available to all subclasses
    def describe(self) -> str:
        return (
            f"{self.name}: area={self.area():.4f}, "
            f"perimeter={self.perimeter():.4f}"
        )

class Rectangle(Shape):
    def __init__(self, w: float, h: float) -> None:
        self.w, self.h = w, h

    @property
    def name(self) -> str: return "Rectangle"

    def area(self) -> float:      return self.w * self.h
    def perimeter(self) -> float: return 2 * (self.w + self.h)

class Circle(Shape):
    def __init__(self, r: float) -> None:
        import math; self.r = r; self._pi = math.pi

    @property
    def name(self) -> str: return "Circle"

    def area(self) -> float:      return self._pi * self.r ** 2
    def perimeter(self) -> float: return 2 * self._pi * self.r

# Polymorphism: same interface, different implementations
shapes: list[Shape] = [Rectangle(4, 6), Circle(5)]
for shape in shapes:
    print(shape.describe())

try:
    s = Shape()   # TypeError: can't instantiate abstract class
except TypeError as e:
    print(f"Caught: {e}")

8.0 DATACLASSES ‚Äî BOILERPLATE-FREE DATA CONTAINERS

8.1 The Problem Dataclasses Solve

Writing a class that primarily holds data ‚Äî with a constructor, __repr__, and __eq__ ‚Äî is repetitive. Python 3.7 introduced the @dataclass decorator to auto-generate these methods from simple field declarations. The result is a concise, readable class with no boilerplate.

8.2 Key Dataclass Features

field() is used to customise individual fields: set default_factory for mutable defaults, repr=False to exclude a field from repr, compare=False to exclude from equality checks, or init=False to exclude from the constructor.

Setting frozen=True makes the dataclass immutable ‚Äî all fields become read-only ‚Äî and auto-generates __hash__, making instances usable as dictionary keys or set members. Setting order=True generates comparison methods so instances can be sorted.

__post_init__ is called after the generated __init__, allowing you to add validation or compute derived fields.

EXAMPLE 8.1 ‚Äî Dataclasses: Basics to Advanced 
from dataclasses import dataclass, field, KW_ONLY
from typing import ClassVar
import uuid

@dataclass(order=True)
class Student:
    """Represents a student with auto-generated dunder methods."""
    # Fields used for ordering (listed first so comparisons work by priority)
    gpa:       float
    last_name: str
    first_name: str

    # Extra fields ‚Äî excluded from ordering
    _: KW_ONLY
    student_id: str = field(default_factory=lambda: str(uuid.uuid4())[:8])
    courses:    list[str] = field(default_factory=list)   # safe mutable default

    # Class variable ‚Äî not a field, not included in __init__
    _count: ClassVar[int] = 0

    def __post_init__(self) -> None:
        if not (0.0 <= self.gpa <= 4.0):
            raise ValueError(f"GPA must be 0‚Äì4, got {self.gpa}")
        Student._count += 1

    @classmethod
    def count(cls) -> int:
        return cls._count

    def full_name(self) -> str:
        return f"{self.first_name} {self.last_name}"

# Auto-generated __init__, __repr__, __eq__, __lt__, __le__, etc.
s1 = Student(gpa=3.8, last_name="Pansensoy", first_name="Glenn")
s2 = Student(gpa=3.5, last_name="Turing",    first_name="Alan",
              courses=["CS101", "MATH201"])

print(s1)                   # auto __repr__
print(s1 > s2)              # True ‚Äî ordered by gpa first
print(sorted([s1, s2]))     # ascending gpa
print(Student.count())       # 2

# Frozen dataclass ‚Äî immutable + hashable
@dataclass(frozen=True)
class Point:
    x: float
    y: float

p = Point(1.0, 2.0)
seen = {p, Point(3.0, 4.0)}   # hashable ‚Üí usable in set
print(p in seen)             # True

9.0 METACLASSES ‚Äî CLASSES THAT CREATE CLASSES

9.1 What Is a Metaclass?

In Python, everything is an object ‚Äî including classes themselves. A metaclass is the class of a class. Just as a regular class defines how its instances behave, a metaclass defines how classes behave. The default metaclass for all Python classes is type. When you write class Foo: ..., Python calls type('Foo', bases, namespace) to construct the class object.

You create a custom metaclass by inheriting from type and overriding __new__ or __init__. You apply it with the metaclass= keyword: class MyClass(metaclass=MyMeta):. Metaclasses are powerful but rarely needed ‚Äî decorators and class decorators solve most of the same problems more simply.

9.2 Practical Uses of Metaclasses

Common legitimate uses include: enforcing coding standards on all subclasses (e.g. requiring docstrings), auto-registering subclasses in a plugin system, adding methods or attributes to every class automatically, and building ORMs (like Django's models) where class User(Model): name = CharField() is transformed into a database-aware class by the metaclass.

‚ö†Ô∏è Use Sparingly: Metaclasses make code significantly harder to understand. The Python community wisdom is: "if you're not sure whether you need a metaclass, you don't." Prefer class decorators, __init_subclass__, or descriptors for most customisation needs.

EXAMPLE 9.1 ‚Äî Metaclass for Auto-Registering Plugins 
class PluginMeta(type):
    """Metaclass that auto-registers every subclass in a registry."""
    _registry: dict[str, type] = {}

    def __new__(mcs, name: str, bases: tuple, namespace: dict, **kwargs):
        cls = super().__new__(mcs, name, bases, namespace)
        if bases:   # skip registering the base class itself
            mcs._registry[name.lower()] = cls
        return cls

class Plugin(metaclass=PluginMeta):
    """Base class for all plugins."""
    def run(self) -> str:
        raise NotImplementedError

class CSVPlugin(Plugin):
    def run(self) -> str: return "Processing CSV data‚Ä¶"

class JSONPlugin(Plugin):
    def run(self) -> str: return "Processing JSON data‚Ä¶"

class XMLPlugin(Plugin):
    def run(self) -> str: return "Processing XML data‚Ä¶"

# The metaclass has automatically populated the registry
print(PluginMeta._registry)
# {'csvplugin': , 'jsonplugin': ..., 'xmlplugin': ...}

# Look up and invoke a plugin by name at runtime
def run_plugin(name: str) -> str:
    plugin_cls = PluginMeta._registry.get(name.lower())
    if not plugin_cls:
        raise ValueError(f"Unknown plugin: {name}")
    return plugin_cls().run()

print(run_plugin("JSONPlugin"))   # Processing JSON data‚Ä¶
print(run_plugin("CSVPlugin"))    # Processing CSV data‚Ä¶

10.0 OOP DESIGN PATTERNS IN PYTHON

Design patterns are proven, reusable solutions to commonly occurring software design problems. Python's first-class functions and dynamic nature often simplify classic patterns. Below are three fundamental patterns with Pythonic implementations.

10.1 Singleton ‚Äî Ensuring One Instance

The Singleton pattern ensures only one instance of a class is ever created. In Python, the cleanest implementation overrides __new__ ‚Äî the object creation step that precedes __init__.

10.2 Observer ‚Äî Event-Driven Communication

The Observer pattern defines a one-to-many dependency: when one object (the subject) changes state, all dependent objects (observers) are notified automatically. This decouples producers from consumers and is the foundation of event systems and UI frameworks.

10.3 Decorator Pattern (not the @ syntax)

The Decorator pattern wraps an object to extend its behaviour without subclassing. In Python this is especially natural because of duck typing ‚Äî the wrapper just needs to implement the same interface as the wrapped object.

EXAMPLE 10.1 ‚Äî Observer Pattern for Event-Driven Systems 
from __future__ import annotations
from typing import Callable
from dataclasses import dataclass, field

type Handler = Callable[[str, object], None]

class EventEmitter:
    """Subject in the Observer pattern ‚Äî emits named events."""

    def __init__(self) -> None:
        self._listeners: dict[str, list[Handler]] = {}

    def on(self, event: str, handler: Handler) -> None:
        self._listeners.setdefault(event, []).append(handler)

    def off(self, event: str, handler: Handler) -> None:
        self._listeners.get(event, []).remove(handler)

    def emit(self, event: str, data: object = None) -> None:
        for handler in self._listeners.get(event, []):
            handler(event, data)

class StockMarket(EventEmitter):
    def __init__(self) -> None:
        super().__init__()
        self._prices: dict[str, float] = {}

    def update_price(self, ticker: str, price: float) -> None:
        old = self._prices.get(ticker)
        self._prices[ticker] = price
        if old is not None:
            change = ((price - old) / old) * 100
            if abs(change) >= 5:
                self.emit("price_alert", {"ticker": ticker, "change": change, "price": price})
        self.emit("price_update", {"ticker": ticker, "price": price})

# Observer functions (could also be methods)
def log_all_updates(event: str, data: object) -> None:
    print(f"[LOG] {event}: {data}")

def alert_on_spike(event: str, data: object) -> None:
    print(f"üö® ALERT ‚Äî {data['ticker']} moved {data['change']:+.1f}% to ${data['price']}")

market = StockMarket()
market.on("price_update", log_all_updates)
market.on("price_alert",  alert_on_spike)

market.update_price("AAPL", 150.00)
market.update_price("AAPL", 162.50)   # +8.3% ‚Äî triggers alert
market.update_price("GOOG", 2800.00)
market.update_price("GOOG", 2812.00)   # +0.4% ‚Äî no alert

11.0 COMMON PITFALLS AND HOW TO AVOID THEM

MistakeProblem DescriptionCorrect Approach
Mutable Class Attributes Using a mutable object (list, dict) as a class attribute means all instances share the same object. Appending to it from one instance mutates it for every instance. Assign mutable defaults inside __init__: self.items = []. Use field(default_factory=list) in dataclasses.
Forgetting super() in __init__ Overriding __init__ without calling super().__init__() leaves parent-class attributes uninitialised, causing AttributeErrors or silently broken behaviour. Always call super().__init__(...) as the first line of the child's __init__, passing the required arguments.
Name Mangling Confusion Attributes with double leading underscores (e.g. __secret) are name-mangled to _ClassName__secret. Accessing them from outside (or a subclass) without the mangled name raises AttributeError. Use single underscore _attr for "private by convention". Reserve double underscores for deliberate name-mangling to avoid subclass collisions.
Implementing __eq__ Without __hash__ Defining __eq__ automatically sets __hash__ to None, making instances unhashable and unusable as dict keys or set members. Define __hash__ alongside __eq__, or use @dataclass(frozen=True) which auto-generates a consistent hash.
Using isinstance() on Old-Style Checks Checking type(obj) == MyClass excludes subclasses, breaking polymorphism. It prevents duck typing and makes code fragile. Use isinstance(obj, MyClass) ‚Äî it returns True for instances of MyClass and all its subclasses, respecting the inheritance hierarchy.
Deep Inheritance Hierarchies Chains of 4+ classes make code hard to understand, debug, and extend. The MRO becomes complex and overriding methods becomes error-prone. Prefer composition over inheritance. Use mixins for shared behaviour. Follow the rule: inherit for "is-a" relationships, compose for "has-a".

12.0 KNOWLEDGE CHECK: PYTHON CLASSES QUIZ

1. What is the difference between a class attribute and an instance attribute?
Click to reveal answer
A class attribute is defined at the class level and shared across all instances ‚Äî changing it on the class changes it everywhere. An instance attribute is defined in __init__ on self and belongs to one specific object. If you assign a class attribute on an instance, it creates a new instance attribute that shadows the class attribute for that instance only.
2. What does super().__init__() do, and why must you call it in a subclass?
Click to reveal answer
super() returns a proxy that delegates to the next class in the MRO. Calling super().__init__() runs the parent's constructor, ensuring inherited attributes are properly set up. Without it, the subclass object is missing attributes defined by the parent, causing AttributeErrors or silent bugs.
3. Why does defining __eq__ break hashing, and how do you fix it?
Click to reveal answer
Python requires that objects that compare equal have the same hash. When you define __eq__, Python cannot guarantee this is true for the default hash, so it sets __hash__ = None, making the object unhashable. Fix by also defining __hash__ based on the same fields used in __eq__, or use @dataclass(frozen=True) which handles both automatically.
4. When would you choose composition over inheritance?
Click to reveal answer
Use inheritance for genuine "is-a" relationships where the subclass really is a specialised version of the parent. Use composition (holding a reference to another object) for "has-a" relationships. Composition is preferred when: you want to reuse behaviour without the tight coupling of inheritance, you need to switch implementations at runtime, or the inheritance hierarchy is becoming deep and hard to follow.
5. What is Python's MRO and how is it determined?
Click to reveal answer
The Method Resolution Order (MRO) defines the sequence Python follows when searching for a method or attribute in an inheritance hierarchy. It is computed using the C3 linearization algorithm, which guarantees a consistent, deterministic order that respects the local precedence of each class and prevents the "diamond problem". Inspect it with ClassName.__mro__ or ClassName.mro().
