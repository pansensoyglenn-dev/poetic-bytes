♿ Accessibility Statement

Poetic Bytes is committed to making free Python education accessible to every learner — regardless of disability, device, or ability level.

Last reviewed: May 2026

Conformance Status

I aim to conform to the Web Content Accessibility Guidelines (WCAG) 2.1 Level AA published by the W3C. These guidelines explain how to make web content more accessible to people with disabilities. Conformance means content meets the guidelines' requirements.

Partially Conformant

Poetic Bytes is partially conformant with WCAG 2.1 Level AA. Some content does not fully meet all criteria due to ongoing development. We are actively working to address known gaps — see the Known Issues section below.

WCAG 2.1 Principle Coverage
👁️

Perceivable

Level AA — Met

Content is presented in ways users can perceive. All images carry descriptive alt text. Colour is never the sole means of conveying information. Text contrast ratios meet or exceed 4.5:1 on all primary content. Code blocks use additional visual cues beyond colour (syntax token labels).

⌨️

Operable

Level AA — Met

All interactive elements — navigation, accordions, code panels, forms — are reachable and operable by keyboard alone. Focus indicators are visible. No content flashes more than three times per second. A skip-to-content link is provided at the top of every page.

🧠

Understandable

Level AA — Met

The lang attribute is set on every page. Navigation is consistent across the site. Error messages in forms are descriptive and actionable. All instruction text avoids jargon. The FAQ accordion announces its expanded/collapsed state to screen readers.

🔧

Robust

Partially Met

HTML is valid and well-structured. ARIA roles and landmarks are used throughout. Some third-party components (embedded code runners, AdSense) are not fully within our control and may present robustness issues with certain assistive technologies. We document these below.

Accessibility Features

The following features are implemented across Poetic Bytes to support learners with disabilities.

  • ⌨️
    Full keyboard navigation Every interactive element — navigation links, hamburger menu, FAQ accordions, code panels, and form fields — is reachable and activatable using Tab, Shift+Tab, Enter, and Space keys alone. No mouse required.
  • 🔍
    Screen reader support Semantic HTML5 landmarks (nav, main, section, footer) allow screen reader users to jump between page regions. ARIA labels are applied to icon-only buttons. Decorative images carry aria-hidden="true". Live regions announce dynamic content changes.
  • 🎨
    High-contrast colour scheme The site's dark obsidian theme with light text achieves contrast ratios between 7:1 and 12:1 on body copy — well above the WCAG AA requirement of 4.5:1. Gold headings on dark backgrounds maintain at least 4.8:1. Interactive focus outlines are clearly visible.
  • 📐
    Responsive & zoom-friendly layout Content reflows correctly when browser text size is increased up to 200% without horizontal scrolling or loss of content. The mobile-first single-column layout scales gracefully from 320px viewport widths upward.
  • 🚦
    Skip navigation link A "Skip to main content" link appears at the top of every page when focused by keyboard. This lets keyboard and screen reader users bypass the navigation bar and jump directly to the lesson content.
  • 🏷️
    Descriptive page titles and headings Every page carries a unique, descriptive <title> tag and a clear <h1>. Heading levels are used hierarchically (h1 → h2 → h3) without skipping, allowing screen reader users to navigate the page outline meaningfully.
  • 📝
    Form accessibility All form inputs are associated with visible labels or descriptive aria-label attributes. Required fields are marked programmatically. Error messages are linked to their respective fields via aria-describedby.
  • ✋
    Reduced motion support Counter animations, typewriter effects, and hover transitions respect the prefers-reduced-motion media query. Users who have set their OS preference to reduce motion will see static equivalents of all animated elements.
  • 🔤
    Readable typography Body text is set at a base of 15px with a line height of 1.6–1.75. The Syne typeface has a clear, open character set. Code examples use JetBrains Mono, which is specifically designed for legibility in technical contexts and distinguishes easily between similar characters (0/O, 1/l/I).
  • 🌐
    Language declaration Every page declares lang="en" on the root <html> element, allowing screen readers and translation tools to apply the correct language processing rules.

Known Accessibility Issues

We document known issues transparently. Each item includes the affected component, the WCAG criterion involved, and our planned resolution timeline.

  • Third-party AdSense iframes

    Google AdSense ad slots render inside <iframe> elements whose content is not under our control. These may not be fully navigable by keyboard or announced meaningfully by all screen readers. WCAG criterion: 4.1.2 Name, Role, Value.

    Outside our direct control — monitored for Google's own accessibility improvements.
  • Inline code runner (browser-based)

    The browser-based Python code execution environment does not yet expose a fully accessible output region. Output is visually updated but the live region announcement may be delayed or absent on some screen reader/browser combinations. WCAG criterion: 4.1.3 Status Messages.

    Planned fix: Q3 2026 — adding explicit aria-live="polite" output container.
  • Syntax highlighting colour-only distinction

    Code syntax tokens (keywords, strings, comments) are currently distinguished by colour alone. While the site's dark theme provides high contrast ratios for each colour, learners with colour vision deficiencies may find some token pairs difficult to distinguish. WCAG criterion: 1.4.1 Use of Colour.

    Planned fix: Q4 2026 — adding a high-contrast / monochrome syntax theme option.
  • Auto-playing typewriter animation in hero

    The typewriter animation in the homepage hero plays automatically on page load. While it runs once and stops (not looping), it may be distracting for some users. WCAG criterion: 2.2.2 Pause, Stop, Hide.

    Resolved in progress — prefers-reduced-motion check added; animation is static for users with motion preference set.

Technical Approach

Poetic Bytes is built as a collection of self-contained static HTML files with inlined CSS and JavaScript. This architecture — deliberately chosen for reliability, speed, and zero-dependency deployment — also offers accessibility advantages: there are no client-side rendering frameworks that could introduce ARIA timing issues, no JavaScript-gated content that screen readers cannot read, and no dependency on third-party component libraries with unknown accessibility quality.

Semantic HTML first

Every page is built on a foundation of semantic HTML5 elements: <nav>, <main>, <section>, <article>, <aside>, <footer>, and heading elements in strict hierarchy. ARIA attributes are used to supplement semantics where HTML alone is insufficient — not to replace it.

Focus management

When the mobile navigation drawer opens, focus is moved inside the drawer and trapped there until the user closes it. When the drawer closes, focus returns to the element that opened it (the hamburger button). This pattern follows the WAI-ARIA Authoring Practices Guide for modal dialogs.

Colour contrast

All colour combinations on the site were verified using the WCAG contrast ratio algorithm. Primary body text (#E2E8F0 on #0B121C) achieves a ratio of approximately 12:1. The minimum contrast ratio anywhere on the site for informational text is 4.6:1, meeting WCAG AA requirements.

No reliance on JavaScript for content

All lesson content, code examples, and explanatory text is present in the HTML document and is fully readable without JavaScript. JavaScript is used only for progressive enhancement: the navigation drawer, FAQ accordions, counter animations, and search filtering. If JavaScript fails or is disabled, all content remains accessible.

Feedback & Contact

We take accessibility feedback seriously. If you encounter any barrier on Poetic Bytes — whether or not it is listed in the Known Issues above — we want to hear from you. We aim to respond to all accessibility enquiries within 5 working days.

Enforcement procedure

If you are not satisfied with our response after contacting us, you may refer your complaint to the relevant national accessibility authority in your country.
