♿ Accessibility Statement

I believe Python education should be accessible to everyone — regardless of disability, assistive technology, network speed, or device. This statement documents my current accessibility status, known issues, and how to report barriers.

Effective: June 1, 2026 Last reviewed: June 2026 Target: WCAG 2.1 AA Solo operator

My accessibility commitment

I am committed to making Poetic Bytes as accessible as possible and conforming to WCAG 2.1 Level AA. As a solo developer, I cannot always achieve perfection, but I treat every accessibility report as a priority fix. If something is preventing you from learning Python on this site, please tell me — I will act on it.

01

Scope of This Statement

This statement covers all pages on poetic-bytes.vercel.app — every Python lesson, learning track page, toolkit page, and legal page including this one.

It does not cover third-party sites linked from Poetic Bytes (the Python documentation, GitHub, YouTube, etc.). Those sites have their own accessibility commitments.

This statement was prepared in June 2026 based on a self-assessment. I am a solo developer, not an accessibility specialist, and I do not have the resources for a full third-party audit. I evaluate the site using automated tools (Lighthouse, axe), manual keyboard testing, and screen reader spot-checks with NVDA on Windows and VoiceOver on iOS.

02

The Four WCAG 2.1 Principles

The Web Content Accessibility Guidelines (WCAG) 2.1 are built on four principles. Here is what each means for Poetic Bytes specifically.

👁️

Perceivable

All information must be presentable in ways users can perceive. On Poetic Bytes: all images have alt text, colour is never the only way to convey information, code snippets have sufficient contrast, and font sizes are relative (rem-based) so they scale with browser zoom settings.

⌨️

Operable

All functionality must be operable via keyboard, not just a mouse. On Poetic Bytes: every interactive element (navigation, FAQ toggles, mobile drawer, buttons, links) is reachable and operable with Tab, Enter, and Arrow keys. Focus indicators are visible on all focusable elements.

💬

Understandable

Information and the operation of the interface must be understandable. On Poetic Bytes: the page language is declared (lang="en"), error messages are descriptive, I write at a clear reading level, and navigation is consistent across all pages.

🔧

Robust

Content must be robust enough to be interpreted by a wide variety of user agents, including assistive technologies. On Poetic Bytes: I use semantic HTML5 elements, ARIA landmarks and labels where needed, and validate markup to avoid parsing errors that break screen readers.

04

Measures I Have Taken

These are the specific accessibility decisions built into the current version of Poetic Bytes.

Structure & Semantics

  • Semantic HTML5 elements throughout — <nav>, <main>, <section>, <article>, <footer>, <h1>–<h6> in correct logical order.
  • ARIA landmark roles (role="navigation", aria-label) on nav regions and the mobile drawer.
  • Breadcrumb navigation marked with aria-label="Breadcrumb" and aria-current="page" on the current page.
  • All decorative Font Awesome icons have aria-hidden="true" so screen readers skip them.
  • Functional icon-only links and buttons have aria-label attributes with descriptive text.

Keyboard & Focus

  • A visible "Skip to main content" link is the first focusable element on every page — it appears on keyboard focus.
  • The mobile navigation drawer closes with the Escape key and returns focus to the toggle button that opened it.
  • FAQ accordion items are operable with Enter and Space keys.
  • All interactive elements have a visible focus outline using outline: 2px solid var(--gold).
  • Tab order follows the visual reading sequence on all pages.

Visual Design

  • All fonts are set in rem units so they respect the user's browser default font size setting.
  • The single-column layout reflows correctly at 320px viewport width and at 200% browser zoom without horizontal scrolling.
  • Status badges use both colour and text — they do not rely on colour alone to convey meaning.
  • The prefers-reduced-motion media query is respected — the typewriter animation and counter animations are disabled when motion reduction is preferred.
  • The site works correctly in Windows High Contrast mode (forced colours).

Code Content

  • All code blocks use <pre><code> semantic markup and are contained within elements with appropriate ARIA roles.
  • Inline code elements are used for all code references in prose, giving screen readers a signal that the content is code.
  • Pyodide-powered interactive code runners include visible labels and keyboard-accessible run buttons.
05

Known Accessibility Issues

I document known issues transparently. Each has a planned fix date or workaround.

Muted text contrast — some body text below 4.5:1

In Progress

Affected: Secondary text using --muted: #8BA3BF on --bg: #0B121C achieves approximately 4.2:1 contrast ratio — slightly below the WCAG AA requirement of 4.5:1 for normal text.

Impact: Mildly harder to read for users with low vision. Primary content and headings are unaffected.

Fix planned: Adjusting --muted to #96B4CE (5.1:1) in the next CSS update. Workaround: use browser zoom or high-contrast mode.

Card border contrast — some decorative borders below 3:1

In Progress

Affected: Some card borders using --border: #1F4060 do not meet the 3:1 non-text contrast requirement against the background.

Impact: Card boundaries may be harder to distinguish for users with low contrast sensitivity. Content inside cards is unaffected.

Fix planned: Incrementally strengthening border values on card components. No workaround needed — card content remains fully accessible.

Newsletter form — error messages not announced to screen readers

Planned

Affected: The newsletter subscription form's validation error messages are visually displayed but not announced via aria-live regions, so screen reader users may not hear the error.

Impact: Screen reader users who enter an invalid email may not receive immediate feedback about the error.

Fix planned: Adding role="alert" or aria-live="polite" to the error message container. ETA: next site update.

Pyodide code runner — complex keyboard interaction

Ongoing

Affected: The in-browser Python code editor powered by Pyodide uses CodeMirror, which has its own keyboard interaction model that may conflict with screen reader virtual cursor modes.

Impact: Screen reader users may find the interactive code editor difficult to use. All lesson content — explanations, code examples as static text — remains fully accessible without the interactive runner.

Ongoing: Monitoring CodeMirror's accessibility improvements. The interactive runner is an enhancement — all lesson content is readable as static text without it.

Third-party ads — Google AdSense accessibility

Third Party

Affected: Google AdSense advertisements are rendered in iframes by Google. I have no control over the accessibility of ad content itself — alt text on ad images, keyboard operability of ad interactions, and screen reader announcements are Google's responsibility.

Impact: Variable — depends on the specific ad being served. If an ad is causing a serious accessibility barrier, please report it to me and I will raise it with Google AdSense.

No fix possible on my end. Google's ad accessibility policies: AdSense accessibility support.

07

Keyboard Navigation Guide

Every feature on Poetic Bytes is operable without a mouse. Here is a complete reference of keyboard shortcuts and interactions.

General Navigation

  • Tab — move focus forward through interactive elements (links, buttons, inputs).
  • Shift + Tab — move focus backward through interactive elements.
  • Enter — activate a focused link or button.
  • Space — activate a focused button; scroll page down.
  • Home / End — jump to top or bottom of page.

Skip Link

  • Press Tab once immediately after loading any page — a "Skip to main content" link appears. Press Enter to jump past the navigation to the lesson content.

Mobile Navigation Drawer

  • Enter / Space on the hamburger menu button — opens the navigation drawer.
  • Escape — closes the drawer and returns focus to the hamburger button.
  • Tab inside the open drawer — cycles through navigation links within the drawer.

FAQ Accordions

  • Tab to focus a FAQ question.
  • Enter — expand or collapse the answer.
  • Only one answer is expanded at a time; expanding a new one collapses the previous.

Pyodide Code Runner (lesson pages)

  • Tab to focus the Run button, then Enter to execute the code.
  • Inside the CodeMirror editor: Escape exits the editor and returns focus to the surrounding page.
08

Respecting Your Preferences

Poetic Bytes reads and respects the following browser and OS-level accessibility preferences via CSS media queries.

prefers-reduced-motion

  • When this preference is enabled in your OS (Settings → Accessibility → Reduce Motion on most systems), all animations are disabled: the scroll progress bar stops animating, the counter number animations skip directly to their final values, the typewriter animation in the hero is removed, and hover transform effects are suppressed.
  • The site remains fully functional and visually complete with all motion disabled.

prefers-color-scheme

  • Poetic Bytes is a dark-themed site by design. I do not currently offer a light mode toggle. I am aware this may be a barrier for some users — a light mode preference is on my roadmap.
  • Workaround: most browsers and operating systems offer a "force light mode" or page inversion option in their accessibility settings. Windows Magnifier, macOS Invert Colours, and Chrome's forced colour options can all help.

Font Size Scaling

  • All font sizes are set in rem units, which scale with your browser's base font size. To increase all text size: in Chrome, go to Settings → Appearance → Font size and set it to Large or Very Large. All Poetic Bytes text will scale proportionally.

forced-colors (Windows High Contrast)

  • When Windows High Contrast Mode is active, the browser applies forced-colors: active. I tested Poetic Bytes in this mode — all text remains readable, interactive elements remain identifiable, and no information is conveyed only through background colours that get overridden.
09

Reporting an Accessibility Barrier

If you encounter an accessibility barrier on Poetic Bytes that prevents you from accessing content or using a feature, please report it. I take every accessibility report seriously and treat them as priority fixes.

What to include in your report

  • The URL of the page where you found the issue.
  • A description of the barrier — what you were trying to do and what happened instead.
  • The assistive technology or browser you were using (e.g., NVDA with Chrome on Windows 11).
  • If applicable, a screenshot or screen recording — though this is not required.

What happens after you report

  • I will acknowledge your report within 7 days.
  • For issues that block access to lesson content, I aim to fix or provide a workaround within 14 days.
  • For non-blocking issues, I will add them to my accessibility backlog and address them in order of severity.
  • I will update the Known Issues section of this page when issues are resolved.

Reach out directly — I respond personally to every accessibility report.
