My Commitment

Poetic Bytes is dedicated to ensuring that my platform is accessible to people with all types of disabilities. I continuously work to improve the user experience for everyone and apply the relevant accessibility standards.

I recognise that accessibility is not a one-time effort but an ongoing commitment. As I develop new features and content, I embed accessibility into every step of my process — from design to development to testing. ✅ My Accessibility Commitments

Concrete actions I take to ensure everyone can learn Python

  • Visual Accessibility — All text meets WCAG contrast ratios (≥ 4.5:1 for normal text, ≥ 3:1 for large text). I never rely on colour alone to convey information.
  • Keyboard Navigation — Every interactive element is fully operable via keyboard. Focus indicators are clearly visible. Tab order follows a logical reading sequence.
  • Colour Blindness Consideration — I use patterns, labels, and icons in addition to colour to distinguish information. Colour combinations are tested for common colour vision deficiencies.
  • Readable Typography — I use a clear, sans-serif font family (Syne) with adequate line height (1.6) and spacing. Text can be resized up to 200% without breaking the layout.
  • Screen Reader Compatibility — Semantic HTML elements (headings, landmarks, lists) are used throughout. All images have descriptive alt text. ARIA labels provide additional context where needed.
  • Code Accessibility — Code blocks use semantic markup with proper syntax highlighting. The code editor and output panels are keyboard-accessible.
  • Focus Management — Focus is managed predictably across interactions. Skip links allow bypassing repeated navigation. Focus indicators are high-contrast and visible.
  • Alternative Text — Every meaningful image, icon, and graphic has descriptive alternative text. Decorative images are hidden from screen readers.
  • Mobile & Touch Accessibility — Touch targets are at least 44px × 44px. The interface is fully responsive and usable on all screen sizes.
  • Regular Testing — I conduct regular accessibility audits using automated tools and manual testing with screen readers (NVDA, VoiceOver, TalkBack).
  • User Feedback — I welcome and act on accessibility feedback. Report issues using the form below or contact me directly.
  • Continuous Improvement — Accessibility is integrated into my development lifecycle. Every new feature is reviewed for accessibility before release.

About WCAG 2.1

WCAG 2.1 (Web Content Accessibility Guidelines) is the internationally recognised standard for web accessibility. Level AA is the target for most organisations and is considered the minimum for inclusive design. I aim to meet or exceed Level AA across all content and features.
